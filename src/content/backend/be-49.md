**Compare MQTT, WebSocket, and Server-Sent Events (SSE) for real-time communication — when do you use each?**

Real-time communication is a core backend challenge. Understanding the tradeoffs helps you pick the right protocol for each use case.

```python
# ═══════════════════════════════════

# Is the client a browser?
#   Yes, need to send AND receive → WebSocket
#   Yes, server push only → SSE (simpler, better HTTP/2 compat)
#   Yes, IoT-style pub/sub → MQTT over WebSocket (MQTT.js)
#
# Is the client an embedded device / IoT sensor?
#   Need reliable delivery (QoS) → MQTT
#   Have reliable network → MQTT or WebSocket
#
# Is the client a mobile app?
#   Background push notifications → FCM/APNs (not these)
#   In-app real-time → WebSocket or MQTT
#
# Do you need to fan-out to thousands of subscribers?
#   Use MQTT broker or dedicated pub/sub (Pub/Sub, SNS)
#   WebSocket requires broadcasting yourself + sticky sessions

# ═══════════════════════════════════
# SCALING WEBSOCKET vs MQTT
# ═══════════════════════════════════

# WebSocket scaling challenge:
#   WS connections are stateful — client A on server 1,
#   client B on server 2. Server 1 can't send to client B directly.
#   Solutions:
#     - Sticky sessions (LB routes same client to same server)
#     - Pub/Sub backbone: servers publish to Redis/Kafka,
#       all servers broadcast to their local WS clients

# Server 1 ─── WebSocket ─── [Client A]
#     │
#   Redis Pub/Sub (or Kafka)
#     │
# Server 2 ─── WebSocket ─── [Client B]

# MQTT scaling:
#   Broker clusters handle fan-out natively (HiveMQ, EMQX).
#   Devices connect to any broker node;
#   cluster routes messages to all matching subscribers.
```

## Explanation

**SSE's underrated simplicity**: SSE runs over plain HTTP — no protocol upgrade, works through HTTP/2 multiplexing without holding a dedicated TCP connection, and browsers auto-reconnect with the `Last-Event-ID` header for resumable streams. For a dashboard showing live metrics or a notification feed, SSE is often the right answer — less infrastructure than WebSocket, native browser support, and HTTP caching headers still work.

**WebSocket's hidden cost**: Unlike SSE (HTTP), WebSockets are long-lived TCP connections. Each connected client holds resources on your server. At 100K concurrent users, you need a strategy: sticky sessions at the load balancer, a pub/sub backbone (Redis, Kafka) to bridge messages across server instances, and careful resource management.

**MQTT's niche advantage**: QoS + persistent sessions. A temperature sensor in a factory might lose connectivity for 30 seconds. With MQTT's `clean_session=false` and QoS 1, the broker queues all missed commands and delivers them when the device reconnects. Neither WebSocket nor SSE offers this out of the box.

**Practical rule of thumb:**
- Web dashboards, chat, collaborative features → **WebSocket**
- Live feeds, notifications, log streaming → **SSE**
- IoT sensors, industrial equipment, mobile IoT → **MQTT**

## ELI5

A **WebSocket** is like a phone call — once connected, you can both talk at the same time freely. **SSE** is like a live radio broadcast — the station (server) talks, you (browser) only listen, but you can tune in and out. **MQTT** is like a walkie-talkie channel with a relay station (broker) — devices post messages to channels, everyone tuned in receives them, and if you were out of range for a bit, the relay station saved your messages.
