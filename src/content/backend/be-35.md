**Explain the CAP theorem. What is eventual consistency and how does it relate to ACID vs BASE?**

CAP theorem is the foundational constraint of distributed systems. Understanding it explains why distributed databases make different tradeoffs and how to choose the right one for your use case.

## Solution

```text
CAP THEOREM
═══════════

A distributed system can guarantee at most 2 of 3:

  C - Consistency
      Every read returns the most recent write
      (all nodes see the same data at the same time)

  A - Availability
      Every request receives a response (not an error)
      (system is always operational)

  P - Partition Tolerance
      System continues operating despite network
      partitions (messages lost/delayed between nodes)


      C
     / \
    /   \
   CP   CA ← Only possible if network never fails
    \   /     (single datacenter, not truly distributed)
     \ /
      P ──── AP


WHY P IS MANDATORY
══════════════════

In any distributed system, network partitions WILL happen:
  - Cables get cut
  - Switches fail
  - Datacenters lose connectivity

Since P is non-negotiable, the real choice is:

  CP (Consistency + Partition Tolerance):
    During partition → reject requests that can't be consistent
    "I'd rather give you an error than wrong data"

  AP (Availability + Partition Tolerance):
    During partition → serve responses that may be stale
    "I'd rather give you old data than no data"


REAL-WORLD EXAMPLES
═══════════════════

┌─────────────────┬──────┬──────────────────────────────┐
│ System          │ Type │ Behavior During Partition    │
├─────────────────┼──────┼──────────────────────────────┤
│ PostgreSQL      │ CP   │ Single master rejects writes │
│ (single master) │      │ if can't replicate           │
├─────────────────┼──────┼──────────────────────────────┤
│ MongoDB         │ CP   │ Elections pick new primary,  │
│                 │      │ brief unavailability         │
├─────────────────┼──────┼──────────────────────────────┤
│ Redis Cluster   │ CP   │ Rejects writes if no quorum  │
├─────────────────┼──────┼──────────────────────────────┤
│ DynamoDB        │ AP   │ All nodes accept writes,     │
│                 │      │ resolve conflicts later      │
├─────────────────┼──────┼──────────────────────────────┤
│ Cassandra       │ AP   │ Tunable: can be CP per query │
│                 │      │ (configurable consistency)   │
├─────────────────┼──────┼──────────────────────────────┤
│ CockroachDB     │ CP   │ Strong consistency, some     │
│                 │      │ requests rejected            │
└─────────────────┴──────┴──────────────────────────────┘


ACID vs BASE
════════════

ACID (Traditional SQL):
  Atomicity    → All or nothing
  Consistency  → Valid state transitions
  Isolation    → Transactions don't interfere
  Durability   → Committed data survives crashes

  Strong guarantees. Single database. Vertical scaling.

BASE (Distributed/NoSQL):
  Basically Available → System usually responds
  Soft state          → State may change without input
                        (due to async replication)
  Eventually consistent → Given time, all replicas converge

  Weaker guarantees. Multiple nodes. Horizontal scaling.

  ACID ←─────────────────────────────────→ BASE
  Strong consistency                    Eventual consistency
  Single machine                        Distributed
  Lower throughput                      Higher throughput


CONSISTENCY LEVELS
══════════════════

Strong Consistency:
  After a write completes, all subsequent reads
  return the new value. No stale reads ever.
  Cost: higher latency (must synchronize)

Eventual Consistency:
  After a write, replicas converge "eventually"
  Reads may return stale data temporarily
  Guarantee: if no new writes, all replicas converge
  Cost: lower latency, but stale reads possible

Causal Consistency:
  If operation B depends on A, everyone sees A before B
  Independent operations may be seen in different orders
  Middle ground between strong and eventual

Read-your-writes:
  You always see your own writes immediately
  Others may see them with delay
  Common practical requirement


CONFLICT RESOLUTION
═══════════════════

When two nodes modify the same data during a partition:

1. Last-Write-Wins (LWW):
   Use timestamp to pick winner
   Simple but can lose data
   Used by: DynamoDB, Cassandra

2. Vector Clocks:
   Track causal history of updates
   Detect conflicts → let application resolve
   Used by: Riak, Amazon's original Dynamo

3. CRDTs (Conflict-free Replicated Data Types):
   Data structures that automatically merge without conflicts
   Types: counters, sets, registers, maps
   Used by: Redis (CRDT counters), Figma (collaborative editing)

   Example: G-Counter (grow-only counter)
   Node A: {A:3, B:0} → increment → {A:4, B:0}
   Node B: {A:0, B:2} → increment → {A:0, B:3}
   Merge:  {A:4, B:3} → total = 7 (correct!)


PRACTICAL GUIDANCE
══════════════════

Choose CP (strong consistency) for:
  ✓ Financial transactions (bank transfers)
  ✓ Inventory management (prevent overselling)
  ✓ User authentication/authorization
  ✓ Anything where wrong data is worse than no data

Choose AP (eventual consistency) for:
  ✓ Social media feeds (slightly stale is OK)
  ✓ Product catalogs (eventually updated)
  ✓ Analytics/metrics (approximate is fine)
  ✓ Shopping carts (merge conflicts acceptable)
  ✓ DNS (cached records eventually update)

Many systems mix both:
  ✓ Stripe: CP for payment processing, AP for dashboards
  ✓ Amazon: CP for checkout, AP for product recommendations
```

## Explanation

CAP theorem is often oversimplified as "pick 2 of 3," but the real insight is that **partition tolerance is mandatory** in distributed systems. Network failures are inevitable — you can't wish them away. So the real decision is what happens during a partition: do you prioritize consistency (reject potentially inconsistent requests) or availability (serve possibly stale data)?

Most modern distributed databases offer **tunable consistency** rather than a fixed CP/AP choice. Cassandra lets you specify consistency per query — `QUORUM` for strong consistency on critical reads, `ONE` for fast eventual consistency on non-critical reads. This lets you make the right tradeoff at the operation level, not the system level.

The shift from ACID to BASE isn't about giving up correctness — it's about **being explicit about which operations need strong consistency** and which can tolerate temporary staleness. A bank transfer absolutely needs ACID guarantees. A social media feed showing a post 2 seconds late is perfectly fine. Designing systems that use strong consistency where it matters and eventual consistency everywhere else gives you both correctness and scalability.

## ELI5

Imagine three friends (nodes) who need to stay in sync about their club's member list. CAP says if their phones stop working (partition), they can either: (CP) stop accepting new members until phones work again — safe but club activities pause. Or (AP) each keep accepting members independently — club stays active but their lists won't match until phones work. Eventually consistent means "we might disagree for a minute, but we'll sort it out." Strong consistency means "we all agree before anything happens."
