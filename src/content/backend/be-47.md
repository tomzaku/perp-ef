**What is MQTT, how does its pub/sub model and QoS system work, and when should you use it?**

MQTT is the dominant messaging protocol for IoT. Understanding its design tradeoffs is essential for connected device backends.

## Solution

```text
// ═══════════════════════════════════
// 1. MQTT ARCHITECTURE
// ═══════════════════════════════════

                    ┌──────────────┐
  [Publisher]  ───▶ │    BROKER    │ ───▶  [Subscriber A]
  (sensor,          │  (Mosquitto, │
   mobile app)      │   HiveMQ,    │ ───▶  [Subscriber B]
                    │   AWS IoT)   │
  [Publisher2] ───▶ │              │ ───▶  [Subscriber C]
                    └──────────────┘

Key components:
  Broker  — central server that routes messages
  Publisher — sends messages to a topic
  Subscriber — receives messages from a topic
  Topic  — hierarchical path like filesystem paths

Topics use "/" as separator and support wildcards:
  sensors/home/livingroom/temperature   (specific)
  sensors/home/+/temperature            (+ = single level wildcard)
  sensors/#                             (# = multi-level wildcard)
  $SYS/broker/clients/connected        ($ = broker system topics)
```

```python
# ═══════════════════════════════════
# 2. QoS LEVELS — The Key Differentiator
# ═══════════════════════════════════

# QoS 0 — At most once (fire and forget)
#   Publisher sends once. No ACK. Message may be lost.
#   Use: High-frequency sensor data (missing a reading is OK)
#   Overhead: Minimal

# QoS 1 — At least once
#   Publisher retries until PUBACK received.
#   Message may arrive MULTIPLE TIMES (duplicates possible).
#   Subscriber must handle duplicates (idempotent processing).
#   Use: Command delivery where duplicate is acceptable

# QoS 2 — Exactly once (4-way handshake)
#   PUBLISH → PUBREC → PUBREL → PUBCOMP
#   Message arrives exactly once. Highest overhead.
#   Use: Financial transactions, billing events, critical commands

# QoS Flow:
# QoS 0: Publisher ──[PUBLISH]──▶ Broker ──[PUBLISH]──▶ Subscriber
#
# QoS 1: Publisher ──[PUBLISH]──▶ Broker
#         Publisher ◀──[PUBACK]── Broker  (retry if no PUBACK)
#         Broker ──[PUBLISH]──▶ Subscriber (may duplicate)
#
# QoS 2: Publisher ──[PUBLISH]──▶ Broker
#         Publisher ◀──[PUBREC]── Broker
#         Publisher ──[PUBREL]──▶ Broker
#         Publisher ◀──[PUBCOMP]─ Broker  (exactly once)

import paho.mqtt.client as mqtt
import json

# ═══════════════════════════════════
# 3. PUBLISHER (Python — paho-mqtt)
# ═══════════════════════════════════

client = mqtt.Client(client_id="sensor-001", clean_session=False)
client.username_pw_set("device", "secret")
client.tls_set()  # TLS over TCP (port 8883)

# Last Will Testament — broker publishes this if device disconnects ungracefully
client.will_set(
    "devices/sensor-001/status",
    payload=json.dumps({"online": False}),
    qos=1,
    retain=True  # broker stores last message for new subscribers
)

client.connect("mqtt.broker.io", 8883, keepalive=60)

# Publish a temperature reading
payload = json.dumps({"value": 23.5, "unit": "C", "ts": 1712345678})
client.publish(
    topic="sensors/home/livingroom/temperature",
    payload=payload,
    qos=1,       # at least once
    retain=False # don't store for future subscribers
)

# ═══════════════════════════════════
# 4. SUBSCRIBER (Python — paho-mqtt)
# ═══════════════════════════════════

def on_connect(client, userdata, flags, rc):
    print(f"Connected with code {rc}")
    # Subscribe after connect (handles auto-reconnect)
    client.subscribe("sensors/home/+/temperature", qos=1)
    client.subscribe("devices/+/status", qos=1)

def on_message(client, userdata, msg):
    topic = msg.topic
    payload = json.loads(msg.payload)
    print(f"[{topic}] {payload}")
    # Process: store to time-series DB, trigger alerts, etc.

client = mqtt.Client(client_id="dashboard-001", clean_session=False)
client.on_connect = on_connect
client.on_message = on_message
client.connect("mqtt.broker.io", 8883)
client.loop_forever()  # blocking event loop
```

```text
// ═══════════════════════════════════
// 5. KEY FEATURES
// ═══════════════════════════════════

RETAIN flag:
  Broker stores the LAST retained message per topic.
  New subscribers immediately receive it on subscribe.
  Use: Device status, last-known sensor value.

Persistent Sessions (clean_session=False):
  Broker stores subscriptions + missed QoS 1/2 messages while device is offline.
  Device reconnects and receives queued messages.
  Use: Mobile clients, intermittently connected sensors.

Keep Alive:
  Client sends PINGREQ every N seconds; broker expects it.
  If broker doesn't hear from client in 1.5x keepalive → disconnect + LWT.
  Detects "half-open" TCP connections on unreliable networks.

Shared Subscriptions (MQTT 5):
  $share/group/topic
  Multiple consumers in a group, each message goes to ONE member.
  Like consumer groups in Kafka — enables horizontal scaling.

// ═══════════════════════════════════
// 6. MQTT vs KAFKA vs WebSocket
// ═══════════════════════════════════

                  MQTT           Kafka          WebSocket
Transport:        TCP            TCP            TCP (HTTP upgrade)
Model:            Pub/Sub        Pub/Sub        Peer-to-peer
Broker:           Yes            Yes            No (direct)
Offline msgs:     Yes (sessions) Yes (offsets)  No
Ordering:         Per-topic      Per-partition  Per-connection
Throughput:       Medium         Very high      Low-medium
Device-friendly:  Yes (tiny)     No (heavy)     Partial
Replay:           No             Yes            No
Best for:         IoT, mobile    Event streams  Browser real-time
```

## Explanation

MQTT was designed in 1999 for satellite-connected oil pipelines — unreliable networks, battery-powered sensors, and constrained memory. These constraints shaped every design decision:

**Tiny packet overhead**: The fixed header is just 2 bytes. A minimal PUBLISH packet can be under 10 bytes. Compare to HTTP where headers alone are hundreds of bytes.

**QoS selection guide**: Most IoT sensor data uses QoS 0 — if you're reading temperature every second, losing one reading doesn't matter. Use QoS 1 for commands (open valve, unlock door) where you need confirmation. Use QoS 2 only when business logic demands exactly-once (billing meters, financial events) — the 4-way handshake costs 4 round trips.

**The broker is the heart**: Unlike WebSockets (direct connection), MQTT decouples publishers and subscribers completely. A device only knows about the broker, not who's consuming its data. This lets you add a new analytics consumer or alerting system without touching the device firmware.

**MQTT 5 (2019)** added: reason codes on disconnect, message expiry, topic aliases, user properties, and shared subscriptions for load balancing consumers.

## ELI5

Imagine a school PA system. Students (subscribers) tune in to the morning announcements channel (topic). The principal (publisher) broadcasts once. The PA system (broker) delivers the message to everyone listening. If a student is late to class and missed the announcement, the bulletin board (retain) has the last message posted. MQTT works the same way — perfect for when you have thousands of "students" (devices) who may go offline briefly but still need important "announcements" (commands/updates) when they return.
