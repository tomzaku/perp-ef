**How do you architect a production AI application?**

Building production AI applications requires combining traditional software engineering with LLM-specific patterns for reliability, observability, and cost management.

## Solution

```text
CORE ARCHITECTURE
═════════════════

┌─────────────────────────────────────────────────┐
│                   Client (UI)                    │
│         Streaming SSE / WebSocket                │
└──────────────────────┬──────────────────────────┘
                       ↓
┌──────────────────────────────────────────────────┐
│              API Gateway / Router                 │
│   Rate limiting, auth, model routing, caching    │
└──────────────────────┬───────────────────────────┘
                       ↓
┌──────────────────────────────────────────────────┐
│           Orchestration Layer                     │
│   Prompt templates, tool execution, agent loops  │
└────┬─────────┬──────────┬───────────┬────────────┘
     ↓         ↓          ↓           ↓
  [LLM API] [Vector DB] [Tools]  [Cache Layer]


KEY PATTERNS
════════════

1. GATEWAY / ROUTER PATTERN
   Route requests to different models by task complexity/cost.
   Simple query    → Fast/cheap model (Haiku)
   Complex reason  → Powerful model (Opus)
   Code generation → Code-specialized model

2. FALLBACK CHAIN
   If primary model fails, automatically fall back.
   Claude Opus → Claude Sonnet → GPT-4o → Cached response → Error

3. STREAMING RESPONSES
   Stream tokens via SSE/WebSocket for responsive UX.
   Time-to-first-token matters more than total latency.

4. SEMANTIC CACHING
   Cache responses for semantically similar queries.
   "What is React?" → Cache hit for "Explain React"

5. OBSERVABILITY STACK
   Every LLM call logs:
   - Input/output tokens + cost
   - Latency (time to first token, total)
   - Model + parameters used
   - Trace ID for multi-step workflows
   - User feedback (thumbs up/down)


PRODUCTION CONCERNS
═══════════════════

Cost Management:
  - Token budgets per user/request
  - Model routing (cheap model first, escalate if needed)
  - Prompt caching (reuse system prompt across calls)
  - Batch processing for non-real-time tasks

Reliability:
  - Treat LLM as unreliable external service
  - Retries with exponential backoff
  - Circuit breaker for sustained failures
  - Graceful degradation (cached/templated responses)

Latency:
  - Streaming for perceived speed
  - Parallel tool calls where possible
  - Pre-compute embeddings, cache retrieval results
  - Edge deployment for embedding models

Testing:
  - Golden dataset with expected outputs
  - LLM-as-judge for automated eval
  - A/B testing prompt/model changes
  - Regression tests in CI/CD pipeline
```

## Explanation

The key mental model is to treat the LLM as a **powerful but unreliable external service** — similar to how you'd architect around a third-party API. You need caching, retries, fallbacks, monitoring, and cost controls.

The LLM-specific additions are:
- **Streaming** for responsive UX
- **Semantic caching** (not just exact-match)
- **Token-based cost management** (not just request-based)
- **Eval pipelines** for continuous quality assurance

**Rule of thumb:** If your AI feature can't gracefully handle the LLM being down or returning garbage, it's not production-ready.

## ELI5

Building an AI app is like running a restaurant where your chef is brilliant but unpredictable. You need a manager (orchestration) to give clear orders, a backup plan if the chef is unavailable (fallbacks), a way to serve food quickly (streaming), and a system to track how much ingredients cost (token management). You also need food critics to regularly test the quality (evaluation).
