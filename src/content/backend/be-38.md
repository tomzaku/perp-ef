**How does SSL/TLS work? Explain the handshake process and certificate verification.**

TLS secures virtually all modern internet communication. Understanding how it works — not just that "HTTPS is secure" — is essential for debugging certificate issues, designing secure systems, and answering interview questions.

## Solution

```text
WHY TLS?
════════

TLS provides three guarantees:
  1. Confidentiality: data is encrypted (can't be read)
  2. Integrity: data can't be modified without detection
  3. Authentication: you're talking to who you think you are


TLS HANDSHAKE (TLS 1.2)
════════════════════════

Client                              Server
  |                                   |
  |--- ClientHello -----------------→|
  |    - TLS version                  |
  |    - Supported cipher suites      |
  |    - Client random number         |
  |                                   |
  |←-- ServerHello -------------------|
  |    - Chosen cipher suite          |
  |    - Server random number         |
  |                                   |
  |←-- Certificate -------------------|
  |    - Server's public key          |
  |    - Signed by Certificate Auth.  |
  |                                   |
  |←-- ServerHelloDone ---------------|
  |                                   |
  |  [Client verifies certificate]    |
  |                                   |
  |--- ClientKeyExchange -----------→|
  |    - Pre-master secret            |
  |    (encrypted with server's       |
  |     public key)                   |
  |                                   |
  |--- ChangeCipherSpec ------------→|
  |--- Finished (encrypted) --------→|
  |                                   |
  |←-- ChangeCipherSpec --------------|
  |←-- Finished (encrypted) ----------|
  |                                   |
  |====== Encrypted Communication ====|

Both sides derive the same symmetric key from:
  client random + server random + pre-master secret


TLS 1.3 (FASTER)
═════════════════

Client                              Server
  |--- ClientHello + KeyShare -----→|  (1 RTT total!)
  |←-- ServerHello + KeyShare ------|
  |←-- Certificate + Finished ------|
  |--- Finished ------------------→|
  |====== Encrypted ================|

Improvements over TLS 1.2:
  - 1 RTT handshake (vs 2 RTT)
  - 0-RTT resumption (for repeat connections)
  - Removed insecure ciphers (RC4, 3DES, SHA-1)
  - Forward secrecy mandatory (Diffie-Hellman)


CERTIFICATE CHAIN OF TRUST
═══════════════════════════

  ┌──────────────────────┐
  │   Root CA            │ ← Pre-installed in OS/browser
  │   (DigiCert, etc.)   │    (~150 trusted root CAs)
  │   Self-signed         │
  └──────────┬───────────┘
             │ Signs
  ┌──────────↓───────────┐
  │  Intermediate CA     │ ← Signed by Root CA
  │  (DigiCert SHA2)     │    Root kept offline for security
  └──────────┬───────────┘
             │ Signs
  ┌──────────↓───────────┐
  │  Server Certificate  │ ← Your domain's certificate
  │  (example.com)       │    Contains: domain, public key,
  └──────────────────────┘    expiry, issuer signature


HOW CERTIFICATE VERIFICATION WORKS
═══════════════════════════════════

Client receives server certificate and:

  1. Check signature chain:
     Server cert signed by Intermediate CA? ✓
     Intermediate CA signed by Root CA? ✓
     Root CA in trusted store? ✓

  2. Check domain:
     Certificate says "example.com"
     We connected to "example.com" → Match ✓
     (also checks Subject Alternative Names: SAN)

  3. Check expiry:
     Not Before: 2024-01-01
     Not After:  2025-01-01
     Current date within range? ✓

  4. Check revocation:
     OCSP (Online Certificate Status Protocol) or CRL
     Certificate not revoked? ✓

  If ANY check fails → browser shows warning/error


CERTIFICATE TYPES
═════════════════

DV (Domain Validation):
  Proves: you control the domain
  Cost: free (Let's Encrypt)
  Shows: padlock icon only
  Good for: most websites

OV (Organization Validation):
  Proves: organization identity verified
  Cost: $50-$200/year
  Shows: organization name in cert details

EV (Extended Validation):
  Proves: thorough business verification
  Cost: $100-$500/year
  Shows: was green bar, now just in cert details
  Declining in popularity (browsers removed green bar)


SYMMETRIC vs ASYMMETRIC IN TLS
══════════════════════════════

Problem: Asymmetric encryption is ~1000x slower than symmetric

Solution: TLS uses BOTH:
  1. Asymmetric (RSA/ECDH) during handshake only
     → Securely exchange a symmetric key
  2. Symmetric (AES-256-GCM) for all data after handshake
     → Fast encryption for bulk data transfer

This hybrid approach gives you:
  - Security of asymmetric (no pre-shared key needed)
  - Speed of symmetric (fast data transfer)


FORWARD SECRECY
═══════════════

Without forward secrecy:
  If private key is compromised later,
  attacker can decrypt ALL past recorded traffic

With forward secrecy (Ephemeral Diffie-Hellman):
  Each connection generates a unique session key
  Even if private key is stolen → past traffic is safe
  → TLS 1.3 requires forward secrecy


MUTUAL TLS (mTLS)
═════════════════

Normal TLS: only server presents certificate
mTLS: BOTH client and server present certificates

  Used for:
  - Service-to-service communication (microservices)
  - API authentication (instead of API keys)
  - Zero-trust networks

  Client needs its own certificate signed by internal CA


LET'S ENCRYPT (FREE CERTIFICATES)
══════════════════════════════════

  - Free DV certificates
  - Automated via ACME protocol
  - 90-day validity (auto-renewal)
  - Certbot tool handles everything

  # Certbot auto-setup with Nginx
  certbot --nginx -d example.com -d www.example.com
  # Auto-renews via cron/systemd timer
```

## Explanation

TLS is a masterpiece of applied cryptography that solves the **key distribution problem**. You want fast symmetric encryption for data transfer, but how do two strangers agree on a secret key over a public network? TLS uses asymmetric cryptography during the handshake — the client encrypts a pre-master secret with the server's public key, which only the server can decrypt with its private key. Both sides then derive the same symmetric key from this shared secret.

**Certificate verification** is how TLS provides authentication. Without it, encryption alone wouldn't prevent man-in-the-middle attacks — an attacker could present their own public key. Certificates tie a public key to a domain name, signed by a trusted Certificate Authority. Your browser/OS ships with ~150 trusted root CA certificates, forming the root of the chain of trust.

**TLS 1.3** is a significant improvement over 1.2: fewer round trips (1 RTT vs 2), mandatory forward secrecy, and removal of all weak cipher suites. The 0-RTT resumption feature for repeat connections makes HTTPS nearly as fast as HTTP, eliminating the historical performance argument against encryption.

## ELI5

TLS is like sending a secret message in class. First, your friend holds up a special open padlock (public key) for anyone to see. You put your message in a box, lock it with their padlock, and pass it over. Only your friend has the key (private key) to open it. But how do you know the padlock is really from your friend and not a spy? The teacher (Certificate Authority) stamps a seal on it saying "this padlock belongs to your friend." You trust the teacher, so you trust the padlock.
