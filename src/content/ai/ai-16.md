**What is the Model Context Protocol (MCP), and how does it change AI application architecture?**

MCP is an open protocol (created by Anthropic) that provides a standardized way for AI models to connect to external data sources and tools. Think of it as "USB-C for AI" — one universal connector instead of custom integrations for every tool.

## Solution

```text
THE M×N PROBLEM MCP SOLVES
═══════════════════════════

Before MCP:
  App1 ──custom──→ Tool A
  App1 ──custom──→ Tool B
  App2 ──custom──→ Tool A     M apps × N tools = M×N integrations
  App2 ──custom──→ Tool B
  App3 ──custom──→ Tool A
  App3 ──custom──→ Tool B

With MCP:
  App1 ─┐                ┌─→ Tool A
  App2 ─┼── MCP Protocol ─┼─→ Tool B     M + N integrations
  App3 ─┘                └─→ Tool C


MCP ARCHITECTURE
════════════════

┌─────────────────────┐
│   MCP Host          │  (e.g., Claude Desktop, IDE, AI App)
│  ┌────────────────┐ │
│  │  MCP Client    │ │  ← Maintains 1:1 connection with server
│  └───────┬────────┘ │
└──────────┼──────────┘
           │  JSON-RPC 2.0 (stdio or HTTP+SSE)
           ↓
┌─────────────────────┐
│   MCP Server        │  (e.g., GitHub MCP, Postgres MCP, Slack MCP)
│                     │
│  ┌───────────────┐  │
│  │ Tools         │  │  ← Actions the model can invoke
│  │ Resources     │  │  ← Data the model can read
│  │ Prompts       │  │  ← Reusable prompt templates
│  └───────────────┘  │
└─────────────────────┘


THREE CORE PRIMITIVES
═════════════════════

1. TOOLS (Model-controlled)
   Functions that the LLM can invoke to perform actions.
   Examples: create_issue, query_database, send_email

   {
     "name": "create_github_issue",
     "description": "Create a new GitHub issue",
     "inputSchema": {
       "type": "object",
       "properties": {
         "repo": { "type": "string" },
         "title": { "type": "string" },
         "body": { "type": "string" }
       },
       "required": ["repo", "title"]
     }
   }

2. RESOURCES (Application-controlled)
   Data that the application can read and provide as context.
   Identified by URIs: file:///path, postgres://db/table, github://repo
   Examples: file contents, database schemas, API docs

3. PROMPTS (User-controlled)
   Reusable prompt templates exposed by servers.
   Examples: "code-review" prompt, "explain-error" prompt
   Users select these explicitly (like slash commands).


TRANSPORT MECHANISMS
════════════════════

1. stdio (Local servers)
   - Server runs as a subprocess
   - Communication via stdin/stdout
   - Best for: local tools, CLI integrations

2. Streamable HTTP (Remote servers)
   - Server exposes HTTP endpoints
   - Supports Server-Sent Events for streaming
   - Best for: cloud services, shared infrastructure

CONNECTION LIFECYCLE
════════════════════

1. Initialization
   Client → Server: initialize (protocol version, capabilities)
   Server → Client: response (supported features)
   Client → Server: initialized (notification)

2. Discovery
   Client calls tools/list, resources/list, prompts/list
   Server responds with available capabilities

3. Operation
   Client invokes tools, reads resources as needed
   Server executes and returns results

4. Shutdown
   Graceful disconnect


BUILDING AN MCP SERVER (TypeScript SDK)
═══════════════════════════════════════

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "my-mcp-server",
  version: "1.0.0"
});

// Register a tool
server.tool(
  "get_todos",
  "Fetch todos for a user",
  { userId: z.string() },
  async ({ userId }) => {
    const todos = await fetchTodos(userId);
    return {
      content: [{ type: "text", text: JSON.stringify(todos) }]
    };
  }
);

// Register a resource
server.resource(
  "config",
  "config://app",
  async (uri) => ({
    contents: [{ uri: uri.href, text: JSON.stringify(appConfig) }]
  })
);

// Connect via stdio
const transport = new StdioServerTransport();
await server.connect(transport);


SECURITY CONSIDERATIONS
═══════════════════════
- Servers run with user permissions — principle of least privilege
- Clients should get user consent before tool invocations
- Transport security: TLS for HTTP, process isolation for stdio
- Input validation on server side — never trust LLM-generated params
- Rate limiting for expensive operations
```

## Explanation

MCP standardizes the way AI applications talk to the outside world. Before MCP, every AI app had to build custom integrations for each tool. With MCP, tool authors build one server and it works with any MCP-compatible AI application.

**Why this matters for engineers:**

1. **Build once, use everywhere** — An MCP server for your internal API works with Claude Desktop, VS Code, Cursor, and any MCP client
2. **Separation of concerns** — AI app developers focus on UX; tool developers focus on capabilities
3. **Composability** — Users can mix and match MCP servers (GitHub + Slack + Database) in any AI host
4. **Security model** — Clear boundaries between what the model controls (tools) vs. what the user controls (prompts) vs. what the app controls (resources)

**Key distinction from raw function calling:** Function calling is a model-level feature (the LLM generates tool calls). MCP is an application-level protocol that standardizes how tools are discovered, described, and invoked across different AI applications. MCP uses function calling under the hood.

## ELI5

Imagine every phone needed a different charger — one for each brand, each app, each accessory. That would be a mess! MCP is like creating one universal charger (USB-C) for AI. Any AI assistant can plug into any tool — whether it's GitHub, Slack, or your database — using the same standard connector. Tool makers build one plug, and it works everywhere.
