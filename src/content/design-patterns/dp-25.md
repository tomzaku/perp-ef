The Repository pattern **abstracts the data access layer** behind a clean collection-like interface. Business logic (services, use cases) never talks directly to a database driver, ORM, or API — it talks to a repository.

**Why it matters:**
- Business logic stays pure and testable — inject a mock repository, no database needed
- You can swap the data source (Postgres → MongoDB → in-memory) by swapping one class
- Centralises all query logic — no scattered `db.query()` calls spread across the codebase

**Frontend use cases:**
- `UserRepository` abstracts whether users come from REST, GraphQL, or localStorage
- `CartRepository` centralises all cart read/write logic regardless of storage backend
- Any hook like `useUsers()` is effectively a thin repository over an API

## Solution

```js
// Repository interface (the contract)
class UserRepository {
  findById(id)           { throw new Error('Not implemented'); }
  findAll()              { throw new Error('Not implemented'); }
  findByEmail(email)     { throw new Error('Not implemented'); }
  save(user)             { throw new Error('Not implemented'); }
  delete(id)             { throw new Error('Not implemented'); }
}

// Concrete implementation — Postgres
class PostgresUserRepository extends UserRepository {
  constructor(db) { super(); this.db = db; }

  async findById(id) {
    const result = await this.db.query('SELECT * FROM users WHERE id = $1', [id]);
    return result.rows[0] ?? null;
  }

  async findByEmail(email) {
    const result = await this.db.query('SELECT * FROM users WHERE email = $1', [email]);
    return result.rows[0] ?? null;
  }

  async save(user) {
    if (user.id) {
      await this.db.query('UPDATE users SET name=$1, email=$2 WHERE id=$3', [user.name, user.email, user.id]);
    } else {
      const result = await this.db.query(
        'INSERT INTO users (name, email) VALUES ($1, $2) RETURNING *',
        [user.name, user.email]
      );
      return result.rows[0];
    }
  }

  async delete(id) {
    await this.db.query('DELETE FROM users WHERE id = $1', [id]);
  }
}

// In-memory implementation — for tests
class InMemoryUserRepository extends UserRepository {
  constructor() {
    super();
    this.store = new Map();
    this.nextId = 1;
  }

  async findById(id)       { return this.store.get(id) ?? null; }
  async findAll()          { return [...this.store.values()]; }
  async findByEmail(email) { return [...this.store.values()].find(u => u.email === email) ?? null; }
  async save(user) {
    const id = user.id ?? this.nextId++;
    const saved = { ...user, id };
    this.store.set(id, saved);
    return saved;
  }
  async delete(id) { this.store.delete(id); }
}

// Business logic — depends only on the interface, not the implementation
class UserService {
  constructor(userRepository) {
    this.repo = userRepository; // injected
  }

  async register(name, email) {
    const existing = await this.repo.findByEmail(email);
    if (existing) throw new Error('Email already registered');
    return this.repo.save({ name, email });
  }

  async getUser(id) {
    const user = await this.repo.findById(id);
    if (!user) throw new Error('User not found');
    return user;
  }
}

// Production: new UserService(new PostgresUserRepository(db))
// Tests:      new UserService(new InMemoryUserRepository())
```

## Explanation

```
Business Logic (UserService)
         │
         │ depends on interface only
         ▼
   UserRepository (interface)
    findById / save / delete
         │
    ┌────┴────────────────────┐
    ▼                         ▼
PostgresUserRepository   InMemoryUserRepository
(production)             (tests / local dev)
```

**The critical rule:** `UserService` imports `UserRepository` (the interface/abstract class), never `PostgresUserRepository` (the concrete class). This is DIP applied to data access.

**Repository vs DAO (Data Access Object):**
- Both abstract data access
- **DAO** maps closely to a single database table/entity
- **Repository** maps to a *domain concept* and may coordinate multiple tables/sources

## ELI5

A library has books. You don't go into the stacks yourself — you give a request to the librarian (repository). You say "give me book #42." Whether the book is in the main stacks, the basement archive, or a digital copy doesn't matter to you. The librarian handles the details. If the library moves buildings, you still talk to the same librarian with the same interface.
