**What are the differences between hashing, encryption, and encoding? When do you use each?**

Confusing hashing, encryption, and encoding is one of the most common security mistakes. Each serves a completely different purpose.

## Solution

```text
HASHING vs ENCRYPTION vs ENCODING
══════════════════════════════════

┌────────────┬───────────┬────────────┬─────────────┐
│            │ Hashing   │ Encryption │ Encoding    │
├────────────┼───────────┼────────────┼─────────────┤
│ Direction  │ One-way   │ Two-way    │ Two-way     │
│ Key needed │ No        │ Yes        │ No          │
│ Purpose    │ Integrity │ Confiden-  │ Format      │
│            │ Verify    │ tiality    │ conversion  │
│ Reversible │ No        │ Yes (key)  │ Yes (always)│

│ Output     │ Fixed size│ Variable   │ Variable    │
└────────────┴───────────┴────────────┴─────────────┘


HASHING (One-way)
═════════════════

Input → [Hash Function] → Fixed-size digest

"hello"  → SHA-256 → "2cf24dba5fb0a30e26e83b2ac5b9e29e..."
"hello!" → SHA-256 → "ce06092fb948d9ffac7d1a376e404b26..."
  (tiny change → completely different hash: avalanche effect)

Properties:
  - Deterministic: same input always gives same output
  - One-way: cannot reverse hash to get input
  - Fixed output: SHA-256 always outputs 256 bits
  - Collision resistant: hard to find two inputs with same hash

Fast hashes (for integrity, NOT passwords):
  MD5:     128 bits (broken — collisions found, avoid)
  SHA-1:   160 bits (broken — avoid)
  SHA-256: 256 bits (current standard)
  SHA-3:   variable (newest standard)

Slow hashes (for passwords):
  bcrypt:  intentionally slow, includes salt
  scrypt:  memory-hard (resists GPU attacks)
  argon2:  winner of Password Hashing Competition (best)

Use cases:
  - Password storage (bcrypt/argon2)
  - Data integrity verification (checksums)
  - Digital signatures (hash then sign)
  - Deduplication (same hash = same content)


PASSWORD HASHING (CRITICAL)
═══════════════════════════

NEVER: store passwords as plaintext
NEVER: use MD5/SHA for passwords (too fast — billions/sec on GPU)
ALWAYS: use bcrypt/argon2 with unique salt

Why slow hashing?
  SHA-256: 10 billion hashes/second on GPU → cracked instantly
  bcrypt:  ~1,000 hashes/second → years to brute force

What is a salt?
  Random value added to password before hashing
  Prevents: rainbow table attacks (precomputed hash lookups)
  
  Without salt: hash("password") → same hash for ALL users
  With salt:    hash("password" + "x7f2k") → unique per user

ENCRYPTION (Two-way)
════════════════════

Plaintext + Key → [Encrypt] → Ciphertext
Ciphertext + Key → [Decrypt] → Plaintext

SYMMETRIC (same key for encrypt/decrypt):
  AES-256-GCM (standard, fast)
  
  Key:  "mysecretkey123"
  Encrypt("hello", key) → "x9f2a7b..."
  Decrypt("x9f2a7b...", key) → "hello"
  
  ✓ Fast (hardware accelerated)
  ✗ Key distribution problem (how to share key securely?)

ASYMMETRIC (public key + private key):
  RSA, ECDSA, Ed25519
  
  Public key:  anyone can have (encrypt/verify)
  Private key: only you have (decrypt/sign)
  
  Encrypt with public  → only private can decrypt
  Sign with private    → anyone with public can verify
  
  ✓ Solves key distribution (share public key openly)
  ✗ Much slower than symmetric (~1000x)

In practice: use BOTH together (TLS does this)
  1. Asymmetric to exchange a symmetric key
  2. Symmetric for the actual data (fast)


DIGITAL SIGNATURES
══════════════════

Proves: message came from sender AND wasn't modified

  Sender:
    1. Hash the message → digest
    2. Encrypt digest with PRIVATE key → signature
    3. Send message + signature

  Receiver:
    1. Hash the message → digest
    2. Decrypt signature with PUBLIC key → original digest
    3. Compare: if equal → authentic and unmodified

HMAC (Hash-based Message Authentication Code):
  HMAC = Hash(key + message)
  Both parties share the key
  Simpler than digital signatures (symmetric)
  Used for: API authentication, JWT signing, webhook verification


ENCODING (NOT security!)
═════════════════════════

Format conversion — anyone can decode, no key needed

  Base64:      binary → ASCII text
               Used for: embedding images in HTML, email attachments
               btoa("hello") → "aGVsbG8="
               atob("aGVsbG8=") → "hello"

  URL encoding: special chars → percent-encoded
               "hello world" → "hello%20world"

  Hex:         binary → hexadecimal string
               0xFF → "ff"

  NEVER use encoding for security!
  Base64 is NOT encryption — anyone can decode it
```

## Explanation

The critical distinction is **purpose**: hashing verifies integrity (is this data unchanged?), encryption provides confidentiality (can only authorized parties read this?), and encoding converts format (make this binary data safe for text transmission).

The most common real-world mistake is **using fast hashes for passwords**. SHA-256 is great for file integrity checks, but it's designed to be fast — modern GPUs can compute billions of SHA-256 hashes per second, making brute-force attacks trivial. Password hashing algorithms like bcrypt deliberately slow down computation (via configurable "cost factors") to make brute force infeasible. Always use bcrypt or argon2 for passwords.

Another common mistake is **treating Base64 as encryption**. Developers sometimes Base64-encode sensitive data (API keys, tokens) thinking it's "encrypted." Base64 is just a format conversion — anyone can decode it instantly. If you need to protect data, use actual encryption (AES-256-GCM).

## ELI5

**Hashing** is like a meat grinder — you put in a steak and get ground meat. You can verify "this came from a steak" but you can't turn ground meat back into a steak. **Encryption** is like a locked box — you put something in, lock it with a key, and only someone with the key can open it and get the original thing back. **Encoding** is like translating English to Morse code — it looks different but anyone who knows Morse code can read it. It's not a secret, just a different format.
