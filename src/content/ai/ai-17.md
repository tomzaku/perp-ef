**What are AI skills, and how do you design reusable, composable AI capabilities?**

AI skills are self-contained units of AI behavior that combine prompts, tools, and logic into reusable building blocks. They are the "functions" of AI application design — encapsulated, testable, and composable.

## Solution

```text
WHAT IS AN AI SKILL?
════════════════════

A skill is an encapsulated AI capability that combines:
  1. A focused system prompt (role + instructions)
  2. A set of tools the skill needs
  3. Input/output contracts
  4. Optional workflows or multi-step logic

Think of it as: Skill = Prompt + Tools + Schema


SKILL vs MONOLITHIC PROMPT
══════════════════════════

Monolithic approach:
┌──────────────────────────────────────────────┐
│ System Prompt (2000 lines)                   │
│ "You are an assistant that can:              │
│   - Review code                              │
│   - Write tests                              │
│   - Generate docs                            │
│   - Debug errors                             │
│   - ..."                                     │
│ 50 tools attached                            │
└──────────────────────────────────────────────┘
Problems: Context dilution, unpredictable behavior,
          hard to test, expensive token usage

Skill-based approach:
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ Code Review  │ │ Test Writer  │ │ Doc Generator│
│ Skill        │ │ Skill        │ │ Skill        │
│ - 1 prompt   │ │ - 1 prompt   │ │ - 1 prompt   │
│ - 3 tools    │ │ - 2 tools    │ │ - 2 tools    │
│ - 1 schema   │ │ - 1 schema   │ │ - 1 schema   │
└──────────────┘ └──────────────┘ └──────────────┘
Benefits: Focused, testable, composable, cost-effective


ANATOMY OF A SKILL
═══════════════════

// Example: Code Review Skill

Skill Definition:
{
  name: "code-review",
  description: "Reviews code for bugs, security, and best practices",

  // Focused system prompt
  systemPrompt: `You are a senior code reviewer. Analyze the given
    code diff for: security vulnerabilities, performance issues,
    bugs, and style violations. Be specific and actionable.`,

  // Only the tools this skill needs
  tools: ["read_file", "search_codebase", "get_git_diff"],

  // Structured input
  inputSchema: {
    diff: "string",        // The code diff to review
    context?: "string",    // Optional repo context
    severity?: "string"    // Focus level: "critical" | "all"
  },

  // Structured output
  outputSchema: {
    issues: [{
      file: "string",
      line: "number",
      severity: "critical | warning | info",
      description: "string",
      suggestion: "string"
    }],
    summary: "string",
    approvalStatus: "approve | request-changes | comment"
  }
}


SKILL DESIGN PATTERNS
═════════════════════

1. SINGLE-RESPONSIBILITY SKILLS
   Each skill does one thing well.
   ✗ "code-assistant" (too broad)
   ✓ "code-review", "test-generator", "bug-fixer" (focused)

2. COMPOSABLE PIPELINES
   Chain skills together for complex workflows.

   User Request
     ↓
   [Code Review Skill] → issues found
     ↓
   [Bug Fix Skill] → proposed fixes
     ↓
   [Test Writer Skill] → tests for fixes
     ↓
   Final Output

3. SKILL ROUTING
   A meta-skill or router selects the right skill for each request.

   User: "This function is slow"
     ↓
   [Router] → detects performance concern
     ↓
   [Performance Analysis Skill]

4. SKILL INHERITANCE / COMPOSITION
   Base skills provide common behavior; specialized skills extend them.

   BaseCodeSkill (read files, understand repo structure)
     ├── ReviewSkill (+ review-specific tools and prompts)
     ├── RefactorSkill (+ refactoring patterns)
     └── MigrationSkill (+ migration strategies)


REAL-WORLD SKILL EXAMPLES
═════════════════════════

Claude Code Skills:
  /commit     → Analyzes changes, writes commit message
  /review-pr  → Reviews PR with structured feedback
  /simplify   → Finds opportunities to simplify code

  "Explain"    → Explains selected code in context

═════════════════

SKILL CONFIGURATION PATTERNS
═════════════════════════════

1. File-based skills (YAML/Markdown):
   ---
   name: commit
   description: Generate commit messages
   tools: [git_diff, git_log]
   ---
   Analyze the staged changes and write a commit message...

2. Code-defined skills:
   const skill = defineSkill({
     name: "translate",
     prompt: "Translate to {targetLanguage}...",
     parameters: { targetLanguage: "string" }
   });

3. MCP-exposed skills (Prompts primitive):
   Servers expose skills that any MCP client can discover and use
```

## Explanation

Skills bring **software engineering principles** to AI development. Just as we don't write 2000-line functions, we shouldn't write 2000-line system prompts that try to do everything.

**Why skills matter:**

1. **Reliability** — A focused skill with 3 tools outperforms a general assistant with 50 tools. Less context dilution = better results
2. **Testability** — Defined inputs/outputs make skills testable with standard testing frameworks
3. **Cost efficiency** — Only include the prompt and tools needed for the task. A code review doesn't need email tools
4. **Maintainability** — Update one skill without breaking others. Version skills independently
5. **Composability** — Build complex workflows from simple, proven building blocks

**The key insight:** Treat AI capabilities like microservices — small, focused, independently deployable, and composable through well-defined interfaces.

## ELI5

Imagine you have a Swiss Army knife with 50 tools on it. It's hard to find the right one and awkward to use. Now imagine instead you have a toolbox with separate, well-organized tools — a screwdriver, a hammer, a wrench. Each one is easy to grab and does its job perfectly. AI skills are like that toolbox — instead of one giant AI that tries to do everything, you have specialized mini-AIs that each do one thing really well, and you can combine them when needed.
