**What are background jobs and task queues, and when should you use them?**

Background jobs are the backbone of reliable backend systems. Any work that doesn't need to happen synchronously in the request should be a background job.

## Solution

```typescript
// ═══════════════════════════════════
// WHY BACKGROUND JOBS?
// ═══════════════════════════════════
//
// Synchronous (BAD for slow tasks):
// POST /api/orders
//   → Create order (5ms)
//   → Charge payment (500ms)
//   → Send email (200ms)       ← User is waiting...
//   → Generate PDF (1000ms)    ← Still waiting...
//   → Update analytics (100ms)
//   → Response (1805ms total)
//
// With background jobs (GOOD):
// POST /api/orders
//   → Create order (5ms)
//   → Charge payment (500ms)
//   → Enqueue: send_email, generate_pdf, update_analytics
//   → Response (510ms)         ← User gets fast response
//
// Background workers process the rest asynchronously


// ═══════════════════════════════════
// BULLMQ — REDIS-BACKED QUEUE
// ═══════════════════════════════════

import { Queue, Worker } from 'bullmq';
import IORedis from 'ioredis';

const connection = new IORedis({ maxRetriesPerRequest: null });

// --- PRODUCER (API server) ---

const emailQueue = new Queue('emails', { connection });

// Add job to queue
await emailQueue.add('welcome-email', {
  userId: 'user-123',
  email: 'alice@example.com',
  template: 'welcome'
}, {
  attempts: 3,                    // Retry 3 times on failure
  backoff: {
    type: 'exponential',          // 1s, 2s, 4s
    delay: 1000
  },
  removeOnComplete: 1000,         // Keep last 1000 completed jobs
  removeOnFail: 5000              // Keep last 5000 failed jobs
});

// Delayed job (send in 24 hours)
await emailQueue.add('follow-up', { userId: 'user-123' }, {
  delay: 24 * 60 * 60 * 1000     // 24 hours
});

// Scheduled/recurring job
await emailQueue.upsertJobScheduler('daily-report', {
  pattern: '0 9 * * *'            // Every day at 9 AM (cron)
}, {
  name: 'generate-daily-report'
});


// --- CONSUMER (Worker process) ---

const worker = new Worker('emails', async (job) => {
  switch (job.name) {
    case 'welcome-email':
      await sendWelcomeEmail(job.data.email, job.data.template);
      break;
    case 'follow-up':
      await sendFollowUp(job.data.userId);
      break;
    case 'generate-daily-report':
      await generateAndSendReport();
      break;
  }
}, {
  connection,
  concurrency: 5,                 // Process 5 jobs simultaneously
  limiter: {
    max: 10,                      // Max 10 jobs
    duration: 1000                // Per second (rate limiting)
  }
});

// Event listeners
worker.on('completed', (job) => {
  console.log(`Job ${job.id} completed`);
});

worker.on('failed', (job, err) => {
  console.error(`Job ${job.id} failed: ${err.message}`);
  if (job.attemptsMade >= job.opts.attempts) {
    // All retries exhausted → dead letter queue / alert
    alertOps(`Job ${job.id} permanently failed`, err);
  }
});


// ═══════════════════════════════════
// COMMON JOB PATTERNS
// ═══════════════════════════════════

// 1. IDEMPOTENT JOBS (safe to retry)
async function processPayment(job) {
  const { orderId, idempotencyKey } = job.data;

  // Check if already processed
  const existing = await db.payment.findUnique({
    where: { idempotencyKey }
  });
  if (existing) return existing; // Already done, skip

  // Process payment
  const result = await stripe.charges.create({
    amount: job.data.amount,
    idempotency_key: idempotencyKey
  });

  await db.payment.create({
    data: { orderId, idempotencyKey, status: 'completed' }
  });
  return result;
}

// 2. JOB CHAINING (one job triggers next)
const imageQueue = new Queue('images', { connection });

// Upload flow: resize → optimize → upload to CDN
await imageQueue.add('resize', { imageId: 'img-1', width: 800 });

const imageWorker = new Worker('images', async (job) => {
  switch (job.name) {
    case 'resize':
      const resized = await resizeImage(job.data);
      await imageQueue.add('optimize', { ...job.data, path: resized });
      break;
    case 'optimize':
      const optimized = await optimizeImage(job.data);
      await imageQueue.add('upload-cdn', { ...job.data, path: optimized });
      break;
    case 'upload-cdn':
      await uploadToCDN(job.data);
      break;
  }
}, { connection });

// 3. PROGRESS TRACKING
const exportWorker = new Worker('exports', async (job) => {
  const records = await getRecords();
  for (let i = 0; i < records.length; i++) {
    await processRecord(records[i]);
    await job.updateProgress(Math.round((i / records.length) * 100));
  }
}, { connection });


// ═══════════════════════════════════
// WHEN TO USE BACKGROUND JOBS
// ═══════════════════════════════════
//
// ✓ Sending emails / notifications
// ✓ Image/video processing
// ✓ PDF generation
// ✓ Data import/export
// ✓ Webhook delivery (with retries)
// ✓ Scheduled tasks (reports, cleanup)
// ✓ Third-party API calls (unreliable, slow)
// ✓ Analytics / event processing
//
// ✗ Authentication checks (must be synchronous)
// ✗ Data validation (user needs immediate feedback)
// ✗ Simple database writes (no need for queue overhead)
```

## Explanation

Background jobs solve two fundamental problems:
1. **User experience** — Don't make users wait for slow operations
2. **Reliability** — Retry failed operations automatically instead of losing them

**Design principles:**
- **Idempotency**: Jobs should be safe to run multiple times (retries happen)
- **Small payloads**: Store data in DB, pass only IDs in the job
- **Timeouts**: Set job timeouts to prevent stuck workers
- **Dead letter queue**: Jobs that fail all retries go to a separate queue for investigation
- **Monitoring**: Track queue depth, processing time, failure rate

**Choosing a queue:**
- **BullMQ** — Great for Node.js, Redis-backed, feature-rich
- **AWS SQS** — Managed, scales infinitely, good for serverless
- **RabbitMQ** — Versatile, supports complex routing patterns
- **Kafka** — For event streaming at massive scale (different use case)

## ELI5

Background jobs are like placing an order at a restaurant. You don't stand in the kitchen watching the chef cook — you go sit down and they bring the food when it's ready. If the chef drops your order, they make it again (retry). If the kitchen is backed up, orders wait in line (queue). And you can check on your order status anytime (progress tracking).
