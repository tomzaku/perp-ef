**What is SQL injection and how do parameterized queries prevent it?**

SQL injection has been the #1 web vulnerability for decades. Understanding how prepared statements work internally — not just that you should use them — is a critical backend skill.

## Solution

```text
SQL INJECTION EXPLAINED
═══════════════════════

Vulnerable code (string concatenation):
  query = "SELECT * FROM users WHERE name = '" + userInput + "'"

Normal input: "Alice"
  → SELECT * FROM users WHERE name = 'Alice'  ✓

Malicious input: "'; DROP TABLE users; --"
  → SELECT * FROM users WHERE name = '';
    DROP TABLE users;
    --'
  → Database deletes the entire users table!

Other attacks:
  Input: "' OR '1'='1"
  → SELECT * FROM users WHERE name = '' OR '1'='1'
  → Returns ALL users (authentication bypass)

  Input: "' UNION SELECT password FROM admin --"
  → Leaks admin passwords from another table


HOW PREPARED STATEMENTS WORK
═════════════════════════════

Step 1: PREPARE (parse query structure)
  Database receives: SELECT * FROM users WHERE name = ?
  → Parses SQL syntax
  → Creates query execution plan
  → The ? is a parameter placeholder

Step 2: BIND (attach parameters as data)
  Database receives: parameter 1 = "'; DROP TABLE users; --"
  → Treats ENTIRE string as a data value
  → Never parsed as SQL

Step 3: EXECUTE
  → Runs query with bound parameter
  → Equivalent to: WHERE name = "'; DROP TABLE users; --"
  → Searches for a user literally named "'; DROP TABLE users; --"
  → Returns 0 results. No injection. Table is safe.

Key insight: the SQL structure is FIXED at parse time.
Parameters can NEVER change the query structure.


CODE EXAMPLES
═════════════

VULNERABLE (never do this):
  // JavaScript - string concatenation
  const query = `SELECT * FROM users WHERE id = ${req.params.id}`;
  db.query(query);

  // Python - f-string
  cursor.execute(f"SELECT * FROM users WHERE name = '{name}'")

SAFE (always do this):
  // JavaScript - parameterized query
  db.query("SELECT * FROM users WHERE id = $1", [req.params.id]);

  // Python - parameterized
  cursor.execute("SELECT * FROM users WHERE name = %s", (name,))

  // Java - PreparedStatement
  PreparedStatement ps = conn.prepareStatement(
    "SELECT * FROM users WHERE id = ?"
  );
  ps.setInt(1, userId);

  // ORM (Prisma) - automatically parameterized
  const user = await prisma.user.findUnique({
    where: { id: userId }
  });


PERFORMANCE BENEFIT
═══════════════════

Prepared statements are also FASTER for repeated queries:

  Without prepared statement:
    Query 1: Parse → Plan → Execute  (3 steps)
    Query 2: Parse → Plan → Execute  (3 steps again)

  With prepared statement:
    Prepare:  Parse → Plan            (once)
    Execute:  Execute with params     (just 1 step)
    Execute:  Execute with params     (reuses plan)

  The query plan is cached and reused across executions.
  Significant speedup for frequently executed queries.


TRICKY INJECTION VECTORS
═════════════════════════

These are often missed:

1. LIKE clauses:
   SAFE:   WHERE name LIKE $1         param = "%alice%"
   UNSAFE: WHERE name LIKE '%${input}%'

2. ORDER BY (can't parameterize column names!):
   UNSAFE: ORDER BY ${userInput}
   SAFE:   Whitelist allowed columns:
     const allowed = ['name', 'created_at', 'email'];
     const col = allowed.includes(input) ? input : 'id';
     query = `SELECT * FROM users ORDER BY ${col}`;

3. Table/column names:
   Parameters only work for VALUES, not identifiers
   → Always whitelist table/column names

4. IN clauses:
   SAFE:   WHERE id = ANY($1)        param = [1, 2, 3]
   UNSAFE: WHERE id IN (${ids.join(',')})


DEFENSE IN DEPTH
════════════════

Layer 1: Parameterized queries (primary defense)
  → Prevents injection at the database level

Layer 2: Input validation
  → Reject obviously malicious input early
  → Validate type, length, format, range

Layer 3: Least privilege
  → App's DB user should NOT have DROP, ALTER permissions
  → Read-only replicas for read-only endpoints

Layer 4: ORM usage
  → ORMs parameterize by default
  → But raw SQL in ORMs can still be vulnerable!

Layer 5: WAF (Web Application Firewall)
  → Detect/block common injection patterns
  → Last line of defense, not primary

Layer 6: Monitoring
  → Log and alert on SQL errors
  → Unusual query patterns may indicate attack attempts
```

## Explanation

SQL injection works because of a **fundamental design flaw**: mixing code (SQL) and data (user input) in the same string. When the database receives a concatenated query, it can't distinguish between the SQL commands the developer wrote and the "SQL commands" injected by the attacker — it parses and executes everything.

Parameterized queries fix this by **separating the parsing phase from the data phase**. The database first parses the query structure with placeholders, creating an execution plan. Then parameters are bound as pure data values — they're never run through the SQL parser, so they can never be interpreted as commands, regardless of what characters they contain.

This is why parameterized queries are the **only reliable defense** against SQL injection. Input validation and escaping are helpful additional layers, but they're error-prone — there's always a new encoding trick or edge case. Parameterized queries are fundamentally different: they make injection structurally impossible, not just difficult.

## ELI5

Imagine filling out a form at the post office. SQL injection is like someone writing in the "Name" field: "Alice, and also please empty safe #5." If the clerk reads the whole thing aloud, the instruction gets executed. A parameterized query is like a form where the clerk reads the template first ("deliver package to [NAME]"), then pastes whatever you wrote into the NAME slot — even if you wrote "empty safe #5," it just becomes the recipient's name, not an instruction.
