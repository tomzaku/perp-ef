**What is event-driven architecture, and how do message queues enable decoupled systems?**

Event-driven architecture is a design pattern where services communicate through events rather than direct calls. It's the foundation for scalable, resilient microservice systems.

## Solution

```text
REQUEST-DRIVEN vs EVENT-DRIVEN
══════════════════════════════

Request-driven (synchronous, coupled):
  [Order Service] → HTTP → [Payment Service]
                  → HTTP → [Email Service]
                  → HTTP → [Inventory Service]
  If Email Service is down, order creation fails!

Event-driven (asynchronous, decoupled):
  [Order Service] → publishes "OrderCreated" event
       ↓
  [Message Queue / Broker]
       ↓               ↓               ↓
  [Payment Service] [Email Service] [Inventory Service]
  Each consumes independently. If Email is down, it processes later.


KEY PATTERNS
════════════

1. PUBLISH/SUBSCRIBE (fan-out)
   One event → multiple consumers

   Producer: "OrderCreated" event
     → Payment consumer: charges card
     → Email consumer: sends confirmation
     → Analytics consumer: tracks conversion
     → Inventory consumer: decrements stock

2. POINT-TO-POINT (competing consumers)
   One message → exactly one consumer
   Multiple workers share the load

   [Image Upload Queue]
     → Worker 1 picks job (image A)
     → Worker 2 picks job (image B)
     → Worker 3 picks job (image C)

3. EVENT SOURCING
   Events ARE the source of truth (not current state)

   Account events:
     AccountCreated { id: 1, balance: 0 }
     MoneyDeposited { id: 1, amount: 100 }
     MoneyWithdrawn { id: 1, amount: 30 }
   Current state: balance = 70 (replay events)


MESSAGE BROKER COMPARISON
═════════════════════════

              │ Redis Pub/Sub │ RabbitMQ    │ Kafka          │
──────────────┼───────────────┼─────────────┼────────────────┤
Model         │ Pub/Sub       │ Both        │ Log-based      │
Persistence   │ No*           │ Yes         │ Yes (durable)  │
Throughput    │ Very High     │ High        │ Very High      │
Ordering      │ Per channel   │ Per queue   │ Per partition   │
Replay        │ No            │ No          │ Yes (key!)     │
Consumer grps │ No            │ Yes         │ Yes            │
Complexity    │ Low           │ Medium      │ High           │
Best for      │ Real-time,    │ Task routing│ Event streaming│
              │ notifications │ complex     │ high volume    │

* Redis Streams (not Pub/Sub) does support persistence
```

```typescript
// ═══════════════════════════════════
// REDIS PUB/SUB — Simple real-time
// ═══════════════════════════════════

import Redis from 'ioredis';

// Publisher
const pub = new Redis();
await pub.publish('orders', JSON.stringify({
  type: 'OrderCreated',
  data: { orderId: 'ord-123', userId: 'user-1', total: 99.99 }
}));

// Subscriber
const sub = new Redis();
sub.subscribe('orders');
sub.on('message', (channel, message) => {
  const event = JSON.parse(message);
  if (event.type === 'OrderCreated') {
    sendConfirmationEmail(event.data);
  }
});


// ═══════════════════════════════════
// EVENT-DRIVEN WITH TYPED EVENTS
// ═══════════════════════════════════

// Define event types
interface OrderCreated {
  type: 'OrderCreated';
  data: { orderId: string; userId: string; total: number };
  timestamp: string;
  correlationId: string;
}

interface PaymentProcessed {
  type: 'PaymentProcessed';
  data: { orderId: string; paymentId: string; status: 'success' | 'failed' };
  timestamp: string;
  correlationId: string;
}

type DomainEvent = OrderCreated | PaymentProcessed;

// Event bus abstraction
class EventBus {
  private handlers = new Map<string, ((event: DomainEvent) => Promise<void>)[]>();

  on(eventType: string, handler: (event: DomainEvent) => Promise<void>) {
    const handlers = this.handlers.get(eventType) || [];
    handlers.push(handler);
    this.handlers.set(eventType, handlers);
  }

  async emit(event: DomainEvent) {
    // Persist event first (outbox pattern)
    await db.events.create({ data: event });

    // Then notify handlers
    const handlers = this.handlers.get(event.type) || [];
    await Promise.allSettled(handlers.map(h => h(event)));
  }
}

// Usage
const bus = new EventBus();

bus.on('OrderCreated', async (event) => {
  await processPayment(event.data);
});

bus.on('OrderCreated', async (event) => {
  await sendEmail(event.data);
});

bus.on('PaymentProcessed', async (event) => {
  if (event.data.status === 'success') {
    await updateOrderStatus(event.data.orderId, 'paid');
  }
});


// ═══════════════════════════════════
// OUTBOX PATTERN (reliable events)
// ═══════════════════════════════════
//
// Problem: What if the DB write succeeds but publishing fails?
//
// Solution: Write event to an "outbox" table in the same transaction
//
// 1. BEGIN TRANSACTION
//    INSERT INTO orders (...) VALUES (...);
//    INSERT INTO outbox (event_type, payload) VALUES ('OrderCreated', ...);
//    COMMIT;
//
// 2. Separate process polls outbox table and publishes events
//    SELECT * FROM outbox WHERE published = false;
//    → Publish to message broker
//    → Mark as published
//
// This guarantees: if the order exists, the event will be published
```

## Explanation

Event-driven architecture excels when:
- **Multiple services** need to react to the same event
- **Services should be independent** (failure isolation)
- **High throughput** is needed (consumers scale independently)
- **Temporal decoupling** is valuable (producers and consumers don't need to be online simultaneously)

**Key trade-offs:**
- **Eventual consistency** — consumers process events asynchronously, so there's a delay
- **Complexity** — debugging async flows is harder than following a synchronous call chain
- **Ordering** — messages may arrive out of order (Kafka partitions help)
- **Idempotency** — consumers must handle duplicate messages

**The outbox pattern** is critical for production systems. Without it, you risk the "dual write" problem: DB commit succeeds but message publishing fails (or vice versa), leading to inconsistency.

## ELI5

Imagine a newsroom. In a request-driven system, the editor personally calls every reporter and says "breaking news about X, write a story." If one reporter's phone is off, the editor is stuck waiting. In an event-driven system, the editor posts the news on a bulletin board. Each reporter checks the board and writes their story independently. If a reporter is sick today, they see the news tomorrow and still write the story. The bulletin board is the message queue.
