**What are CI/CD pipelines, and what deployment strategies minimize risk?**

CI/CD is the practice of automating the path from code change to production deployment. It's how modern teams ship quickly and safely.

## Solution

```yaml
# ═══════════════════════════════════
# GITHUB ACTIONS EXAMPLE
# ═══════════════════════════════════

# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  # ─── CI: Test & Build ───
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: test
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck
      - run: npm test
        env:
          DATABASE_URL: postgres://postgres:test@localhost:5432/test

  # ─── Build Docker Image ───
  build:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ─── CD: Deploy ───
  deploy:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy to production
        run: |
          # Example: update ECS service, Kubernetes deployment, etc.
          kubectl set image deployment/app \
            app=ghcr.io/${{ github.repository }}:${{ github.sha }}
          kubectl rollout status deployment/app --timeout=300s
```

```text
DEPLOYMENT STRATEGIES
════════════════════

1. ROLLING DEPLOYMENT (default for most platforms)
   Replace instances one at a time.

   Before:  [v1] [v1] [v1] [v1]
   Step 1:  [v2] [v1] [v1] [v1]   ← First instance updated
   Step 2:  [v2] [v2] [v1] [v1]
   Step 3:  [v2] [v2] [v2] [v1]
   Done:    [v2] [v2] [v2] [v2]

   ✓ Zero downtime
   ✓ Gradual rollout
   ✗ Brief period with mixed versions
   ✗ Slow rollback (must re-deploy v1)

2. BLUE-GREEN DEPLOYMENT
   Run two identical environments, switch traffic instantly.

   Blue (current):  [v1] [v1] [v1]  ← Serving traffic
   Green (new):     [v2] [v2] [v2]  ← Warming up

   Switch: Route traffic Blue → Green
   Rollback: Route traffic Green → Blue (instant!)

   ✓ Instant rollback
   ✓ No mixed versions
   ✗ 2x infrastructure cost
   ✗ Database migrations need careful handling

3. CANARY DEPLOYMENT
   Route a small percentage of traffic to new version.

   [v1] [v1] [v1] [v1] [v1]  ← 95% traffic
   [v2]                        ← 5% traffic (canary)

   Monitor canary for errors, latency, etc.
   If healthy: gradually increase → 25% → 50% → 100%
   If unhealthy: kill canary, 100% stays on v1

   ✓ Minimal blast radius
   ✓ Real-world testing
   ✗ Complex routing setup
   ✗ Need good observability


PIPELINE BEST PRACTICES
═══════════════════════

1. Fast feedback: Run fastest checks first (lint → types → unit → integration)
2. Cache dependencies: npm cache, Docker layer cache, test artifacts
3. Parallelize: Run independent steps concurrently
4. Fail fast: Stop pipeline on first failure
5. Immutable artifacts: Tag images with commit SHA, not "latest"
6. Environment parity: Same Docker image in staging and prod
7. Secrets management: Never commit secrets, use CI secrets store
8. Database migrations: Run BEFORE deploying new code
9. Health checks: New version must pass health check before receiving traffic
10. Automated rollback: If health check fails, revert automatically
```

## Explanation

CI/CD transforms deployment from a risky manual event into a routine automated process. The pipeline gives you confidence that every change is tested before it reaches production.

**The key principle:** Make deployments small, frequent, and reversible. A team that deploys 10 times a day with automated rollback is safer than a team that deploys once a month manually.

**Choosing a deployment strategy:**
- **Rolling**: Good default for most applications. Simple, zero-downtime
- **Blue-Green**: When you need instant rollback (financial systems, critical services)
- **Canary**: When you want to test with real traffic before full rollout (large user bases)

## ELI5

CI/CD is like a factory assembly line with quality checkpoints. Every time you make a change (raw material goes in), it automatically gets inspected (linting), tested (unit tests), and assembled (build). If it passes all checks, it goes to the store (production). If any check fails, the line stops and you fix the issue. Deployment strategies are how you stock the shelves — you can replace all products at once (blue-green), replace them one by one (rolling), or put a few new ones out and see if customers like them first (canary).
