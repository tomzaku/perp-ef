**How do load balancers and reverse proxies work, and when do you need them?**

Load balancers and reverse proxies are fundamental infrastructure components that sit between clients and your application servers.

## Solution

```text
WHAT THEY DO
════════════

Client → [Reverse Proxy / Load Balancer] → [Server 1]
                                          → [Server 2]
                                          → [Server 3]

REVERSE PROXY (single server behind it):
  - SSL/TLS termination (handles HTTPS)
  - Serve static files (images, CSS, JS)
  - Compression (gzip/brotli)
  - Request buffering
  - Rate limiting
  - Caching

LOAD BALANCER (multiple servers behind it):
  - Everything a reverse proxy does, PLUS:
  - Distributes traffic across instances
  - Health checks (removes unhealthy servers)
  - Session affinity (sticky sessions)
  - Automatic failover


LOAD BALANCING ALGORITHMS
═════════════════════════

1. ROUND ROBIN
   Requests go to each server in order: 1 → 2 → 3 → 1 → 2 → 3
   ✓ Simple, works well for stateless apps with similar hardware
   ✗ Doesn't account for server load or request complexity

2. WEIGHTED ROUND ROBIN
   Server 1 (weight: 3): gets 3x more traffic
   Server 2 (weight: 1): gets 1x traffic
   ✓ Useful when servers have different capacities

3. LEAST CONNECTIONS
   Route to the server with fewest active connections
   ✓ Good for varying request durations (some fast, some slow)
   ✗ Slightly more overhead to track connections

4. IP HASH
   Hash client IP → always routes to same server
   ✓ Session affinity without cookies
   ✗ Uneven distribution if some IPs are "hot"

5. CONSISTENT HASHING
   Hash(request key) → server on a hash ring
   ✓ Minimal redistribution when servers are added/removed
   ✓ Good for caching layers (same key → same cache server)


LAYER 4 vs LAYER 7 LOAD BALANCING
══════════════════════════════════

Layer 4 (Transport — TCP/UDP):
  - Routes based on IP + port
  - Very fast (no request inspection)
  - Can't route based on URL, headers, or cookies
  Example: AWS NLB, HAProxy (TCP mode)

Layer 7 (Application — HTTP):
  - Routes based on URL path, headers, cookies, hostname
  - Can modify requests/responses
  - Can do content-based routing
  Example: AWS ALB, Nginx, Caddy, Traefik

  # Route by path:
  /api/*     → backend servers
  /static/*  → CDN / static server
  /ws/*      → WebSocket servers

  # Route by header:
  X-API-Version: 2 → v2 backend
  Default          → v1 backend
```

```nginx
# ═══════════════════════════════════
# NGINX CONFIGURATION
# ═══════════════════════════════════

# Upstream: define backend servers
upstream api_servers {
    least_conn;                          # Load balancing algorithm
    server 10.0.0.1:3000 weight=3;      # More capacity
    server 10.0.0.2:3000;
    server 10.0.0.3:3000;
    server 10.0.0.4:3000 backup;        # Only used if others are down
}

server {
    listen 443 ssl http2;
    server_name api.myapp.com;

    # SSL termination
    ssl_certificate     /etc/ssl/certs/myapp.crt;
    ssl_certificate_key /etc/ssl/private/myapp.key;

    # Compression
    gzip on;
    gzip_types application/json text/css application/javascript;

    # Static files (served directly by Nginx)
    location /static/ {
        root /var/www;
        expires 30d;                     # Browser cache for 30 days
        add_header Cache-Control "public, immutable";
    }

    # API proxy to backend servers
    location /api/ {
        proxy_pass http://api_servers;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Timeouts
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
    }

    # WebSocket support
    location /ws/ {
        proxy_pass http://api_servers;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # Health check endpoint
    location /health {
        access_log off;
        return 200 'OK';
    }
}
```

```text
HEALTH CHECKS
═════════════

Active health checks (load balancer polls servers):
  - GET /health every 10 seconds
  - If 3 consecutive failures → mark unhealthy
  - If 2 consecutive successes → mark healthy again
  - Unhealthy servers receive no traffic

Passive health checks (monitor actual responses):
  - If a server returns 5xx errors → reduce traffic
  - If connection timeouts → mark unhealthy


MODERN ALTERNATIVES
═══════════════════

Caddy (simpler than Nginx):
  - Automatic HTTPS (Let's Encrypt)
  - Simple config file
  - HTTP/3 support built-in

Traefik (container-native):
  - Auto-discovers Docker/Kubernetes services
  - Dynamic configuration
  - Built-in dashboard

Cloud load balancers:
  - AWS ALB/NLB — managed, auto-scaling
  - GCP Cloud Load Balancing — global, anycast
  - Cloudflare — CDN + load balancing + DDoS protection
```

## Explanation

**When you need a reverse proxy:** Almost always in production. Even with a single server, Nginx/Caddy handles SSL, compression, and static files more efficiently than your application server.

**When you need a load balancer:** When one server can't handle the traffic, or when you need high availability (if one server dies, others take over).

**The typical production setup:**
1. DNS → Cloud load balancer (ALB) → multiple Nginx instances → app servers
2. Or simpler: DNS → Cloudflare → single Nginx → app server(s)

**Key decision:** Use cloud-managed load balancers (ALB, Cloud LB) when possible. They handle health checks, auto-scaling, and SSL automatically. Only run your own Nginx/HAProxy when you need custom routing logic or are on-premise.

## ELI5

A reverse proxy is like a receptionist at a company. When you call, you don't talk directly to the employee — the receptionist takes your call, figures out who should handle it, and connects you. A load balancer is a receptionist with multiple employees who can all do the same job — they figure out who's least busy and send the call to that person. If one employee is sick, the receptionist just stops sending calls to them.
