**Dependency Injection (DI)** means a class **receives** its dependencies rather than **creating** them internally. Instead of `this.db = new PostgresDB()` inside the constructor, the database is passed in from outside.

**DI vs DIP:**
- **DIP** (Dependency Inversion *Principle*) — the design rule: high-level modules should depend on abstractions
- **DI** (Dependency *Injection*) — the implementation technique: pass dependencies in from outside
- DI is how you *achieve* DIP in practice

**Three forms of injection:**
1. **Constructor injection** — dependencies passed via constructor (most common, most explicit)
2. **Method injection** — dependency passed as a method argument
3. **Property injection** — dependency set on a property after construction (less preferred)

## Solution

```js
// ─── Without DI — hard to test, tightly coupled ───
class UserService {
  constructor() {
    this.db = new PostgresDatabase();    // hard dependency — can't swap
    this.logger = new FileLogger();      // hard dependency — can't mock
    this.mailer = new SendGridMailer();  // hard dependency — hits real API in tests
  }

  async register(email) {
    const user = await this.db.save({ email });
    this.logger.log(`User registered: ${email}`);
    await this.mailer.send(email, 'Welcome!');
    return user;
  }
}
// Testing this hits a real DB, writes to disk, and sends emails. Painful.


// ─── With DI — testable, flexible ───
class UserService {
  // Dependencies injected — this class has no idea which concrete types they are
  constructor(userRepository, logger, mailer) {
    this.repo = userRepository;
    this.logger = logger;
    this.mailer = mailer;
  }

  async register(email) {
    const user = await this.repo.save({ email });
    this.logger.log(`User registered: ${email}`);
    await this.mailer.send(email, 'Welcome!');
    return user;
  }
}

// Production wiring
const service = new UserService(
  new PostgresUserRepository(db),
  new FileLogger(),
  new SendGridMailer()
);

// Test wiring — all fakes, no real I/O
const service = new UserService(
  new InMemoryUserRepository(),
  { log: jest.fn() },           // inline mock
  { send: jest.fn() }           // inline mock
);


// ─── DI Container (automatic wiring) ───
// Frameworks like NestJS, InversifyJS do this automatically

class Container {
  constructor() { this.bindings = new Map(); }
  register(token, factory) { this.bindings.set(token, factory); }
  resolve(token) {
    const factory = this.bindings.get(token);
    if (!factory) throw new Error(`No binding for ${String(token)}`);
    return factory(this);
  }
}

const container = new Container();
container.register('db',          () => new PostgresDatabase(process.env.DB_URL));
container.register('userRepo',    (c) => new PostgresUserRepository(c.resolve('db')));
container.register('logger',      () => new FileLogger());
container.register('mailer',      () => new SendGridMailer(process.env.SENDGRID_KEY));
container.register('userService', (c) => new UserService(
  c.resolve('userRepo'),
  c.resolve('logger'),
  c.resolve('mailer')
));

const userService = container.resolve('userService');
```

## Explanation

```
Without DI:                        With DI:
───────────────────                ──────────────────────────
UserService                        UserService
  constructor() {                    constructor(repo, logger, mailer) {
    this.db = new Postgres()           this.repo = repo
    this.log = new FileLogger()        this.logger = logger
  }                                    this.mailer = mailer
                                     }
  ↑ UserService CREATES              ↑ UserService USES
    its dependencies                   injected dependencies
    (tightly coupled)                  (loosely coupled)

Test: must set up a real DB        Test: pass { save: jest.fn() }
```

**Why constructor injection is preferred:**
- Dependencies are explicit — you can see everything a class needs just by looking at its constructor
- Object is always in a valid state after construction
- Easy to see in tests exactly what needs to be mocked

## ELI5

You're building a lamp. Without DI, the lamp comes with a built-in, non-removable bulb. If the bulb breaks, you throw away the lamp. With DI, the lamp has a socket — you plug in whatever bulb you want (LED, fluorescent, smart bulb). For testing, you plug in a test bulb that just records whether it was turned on. The lamp doesn't care what kind of bulb it is.
