**MVC (Model-View-Controller)** and **MVVM (Model-View-ViewModel)** are architectural patterns that separate concerns in a UI application. Both divide the app into three layers, but differ in how the layers communicate.

## MVC

- **Model** — data and business logic (fetching, validation, domain rules)
- **View** — renders the UI from the model's data; sends user actions to the controller
- **Controller** — handles user input, updates the model, decides which view to render

```
User → View → Controller → Model → View (re-render)
```

Classic in: Express (routes = controllers), Rails, Angular (older style).

## MVVM

- **Model** — same as MVC
- **View** — declarative UI template; binds to ViewModel properties
- **ViewModel** — exposes observable state and commands; the View reacts automatically to changes

```
View ←─(data binding)─→ ViewModel ←→ Model
```

Classic in: React (state = ViewModel, JSX = View), Vue, SwiftUI, WPF.

## Solution

```js
// ─── MVC Example (Express-style) ───

// Model — pure data logic
class UserModel {
  constructor(db) { this.db = db; }
  async findById(id) { return this.db.query('SELECT * FROM users WHERE id = ?', [id]); }
  async create(data) { return this.db.query('INSERT INTO users ...', [data]); }
}

// View — template rendering (simplified)
const UserView = {
  render: (user) => `<div class="user">${user.name} — ${user.email}</div>`,
  renderError: (msg) => `<div class="error">${msg}</div>`,
};

// Controller — coordinates model + view
class UserController {
  constructor(model) { this.model = model; }

  async show(req, res) {
    try {
      const user = await this.model.findById(req.params.id);
      res.send(UserView.render(user));
    } catch {
      res.status(404).send(UserView.renderError('User not found'));
    }
  }
}

// Route wires them together
// app.get('/users/:id', (req, res) => new UserController(userModel).show(req, res));


// ─── MVVM Example (React-style) ───

// Model — fetches raw data
async function fetchUser(id) {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}

// ViewModel — manages state, exposes data + actions to the View
function useUserViewModel(id) {
  const [user, setUser] = React.useState(null);
  const [loading, setLoading] = React.useState(true);
  const [error, setError] = React.useState(null);

  React.useEffect(() => {
    fetchUser(id)
      .then(setUser)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [id]);

  // Commands
  const updateName = async (name) => {
    const updated = await patchUser(id, { name });
    setUser(updated);
  };

  return { user, loading, error, updateName }; // the "binding surface"
}

// View — pure rendering, binds to ViewModel
function UserCard({ id }) {
  const { user, loading, error, updateName } = useUserViewModel(id); // bind

  if (loading) return <Spinner />;
  if (error)   return <Error message={error.message} />;
  return (
    <div>
      <h2>{user.name}</h2>
      <button onClick={() => updateName('New Name')}>Rename</button>
    </div>
  );
}
```

## Explanation

```
MVC flow:
  1. User clicks button → View calls Controller.handleClick()
  2. Controller calls Model.save()
  3. Model updates
  4. Controller tells View to re-render (or View observes Model)

MVVM flow:
  1. User clicks button → View calls viewModel.updateName()
  2. ViewModel calls Model, updates its own state
  3. View automatically re-renders (two-way or one-way binding)
  → No explicit "tell view to update" — the binding handles it
```

**Key differences:**

| | MVC | MVVM |
|---|---|---|
| Updates View | Controller explicitly | Data binding automatically |
| Testability | Test Controller in isolation | Test ViewModel — no DOM needed |
| Used in | Express, Rails, Angular | React, Vue, SwiftUI |
| Coupling | View knows Controller | View only knows ViewModel interface |

## ELI5

**MVC** — A restaurant. The customer (user) tells the waiter (controller) what they want. The waiter goes to the kitchen (model) to get it, then brings it to the table (view).

**MVVM** — A vending machine. The display (view) shows whatever is in the machine's state (ViewModel). When you press a button, it updates the state, and the display updates automatically — no waiter needed.
