The State pattern allows an object to **change its behaviour when its internal state changes**. Instead of one class with a large `switch (this.state)` in every method, each state becomes its own class that handles the behaviour for that state.

**State vs Strategy:**
- Both delegate behaviour to another object
- **Strategy** — the algorithm is chosen externally by the client and doesn't change the context itself
- **State** — transitions happen *internally*; the state object (or context) decides what comes next

**Frontend use cases:**
- Traffic lights (Red → Green → Yellow → Red)
- Media player (Playing → Paused → Stopped)
- Authentication flow (LoggedOut → Authenticating → LoggedIn → Expired)
- UI modals / wizards (Step1 → Step2 → Confirm → Done)
- Vending machine (Idle → HasCoin → Dispensing)

## Solution

```js
// State interface (each state implements these)
class TrafficLightState {
  next(light) { throw new Error('next() not implemented'); }
  getColor()  { throw new Error('getColor() not implemented'); }
  getDuration() { throw new Error('getDuration() not implemented'); }
}

// Concrete states
class RedState extends TrafficLightState {
  getColor()    { return 'Red'; }
  getDuration() { return 3000; }
  next(light)   { light.setState(new GreenState()); }
}

class GreenState extends TrafficLightState {
  getColor()    { return 'Green'; }
  getDuration() { return 4000; }
  next(light)   { light.setState(new YellowState()); }
}

class YellowState extends TrafficLightState {
  getColor()    { return 'Yellow'; }
  getDuration() { return 1000; }
  next(light)   { light.setState(new RedState()); }
}

// Context — delegates all behaviour to current state
class TrafficLight {
  constructor() {
    this.state = new RedState(); // initial state
  }

  setState(state) {
    this.state = state;
  }

  next()        { this.state.next(this); }
  getColor()    { return this.state.getColor(); }
  getDuration() { return this.state.getDuration(); }
}

const light = new TrafficLight();
console.log(light.getColor()); // Red
light.next();
console.log(light.getColor()); // Green
light.next();
console.log(light.getColor()); // Yellow
light.next();
console.log(light.getColor()); // Red
```

## Explanation

**Without State pattern** — one class, growing switch statements everywhere:
```js
class TrafficLight {
  next() {
    if (this.state === 'Red') this.state = 'Green';
    else if (this.state === 'Green') this.state = 'Yellow';
    else if (this.state === 'Yellow') this.state = 'Red';
  }
  getDuration() {
    if (this.state === 'Red') return 3000;
    else if (this.state === 'Green') return 4000;
    else if (this.state === 'Yellow') return 1000;
  }
  // Every new method needs another switch...
}
```

**With State pattern** — each state owns its own behaviour. Adding a `FlashingYellow` state means adding one class, touching nothing else.

```
Context (TrafficLight)
  │
  └── currentState: RedState
        ├── getColor() → "Red"
        ├── getDuration() → 3000
        └── next() → context.setState(new GreenState())
```

**In React** — the State pattern maps directly to a state machine. Libraries like XState formalise this: each state is a node, each transition is an edge, and impossible transitions simply don't exist in the machine.

## ELI5

Think of a vending machine. When it has no coin, pressing "dispense" does nothing. After you insert a coin, the same button dispenses the item. After dispensing, it resets. Same button, completely different behaviour — because the *state* changed. Each state knows what "press the button" means for that state specifically.
