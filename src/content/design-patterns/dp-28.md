**Pub/Sub (Publish/Subscribe)** is a messaging pattern where publishers emit events to a **broker/event bus**, and subscribers register interest in event types. Publishers and subscribers never reference each other directly.

**Pub/Sub vs Observer:**
- **Observer** — the Subject *directly* notifies its registered Observers. Tight coupling: Subject holds a list of Observer references.
- **Pub/Sub** — a *broker* sits in between. Publishers fire events at the broker; subscribers register with the broker. Publishers and subscribers are completely decoupled — they don't even need to exist at the same time.

**Frontend use cases:**
- Browser `CustomEvent` / `window.dispatchEvent` — window is the broker
- Redux — store is a Pub/Sub broker; components subscribe, action creators publish
- Node.js `EventEmitter`
- Real-time: Redis Pub/Sub, Kafka, WebSocket event rooms
- Micro-frontend communication across independently deployed apps

## Solution

```js
// ─── In-process EventBus (broker) ───
class EventBus {
  constructor() {
    this.channels = new Map(); // topic → Set of handlers
  }

  subscribe(topic, handler) {
    if (!this.channels.has(topic)) this.channels.set(topic, new Set());
    this.channels.get(topic).add(handler);
    // Return unsubscribe function
    return () => this.channels.get(topic)?.delete(handler);
  }

  publish(topic, payload) {
    this.channels.get(topic)?.forEach(handler => {
      try { handler(payload); }
      catch (e) { console.error(`Handler error on topic "${topic}":`, e); }
    });
  }

  publishAsync(topic, payload) {
    this.channels.get(topic)?.forEach(handler =>
      Promise.resolve().then(() => handler(payload))
    );
  }
}

// Global bus (or inject as a dependency)
const bus = new EventBus();

// ─── Publishers — know nothing about subscribers ───
class OrderService {
  async placeOrder(order) {
    // ... save to DB ...
    bus.publish('order.placed', { orderId: order.id, userId: order.userId, total: order.total });
  }
}

// ─── Subscribers — know nothing about publishers ───
class EmailService {
  constructor() {
    bus.subscribe('order.placed', (event) => {
      console.log(`Sending confirmation email to user ${event.userId}`);
    });
  }
}

class InventoryService {
  constructor() {
    bus.subscribe('order.placed', (event) => {
      console.log(`Reserving stock for order ${event.orderId}`);
    });
  }
}

class AnalyticsService {
  constructor() {
    bus.subscribe('order.placed', (event) => {
      console.log(`Recording $${event.total} sale in analytics`);
    });
  }
}

// Wire up — OrderService doesn't import any of these
new EmailService();
new InventoryService();
new AnalyticsService();

const orderService = new OrderService();
await orderService.placeOrder({ id: 1, userId: 42, total: 99.99 });
// → Sends email, reserves stock, records analytics
//    OrderService knows about none of them
```

## Explanation

```
Without Pub/Sub — OrderService tightly coupled:

  OrderService.placeOrder() {
    emailService.sendConfirmation()     // must import EmailService
    inventoryService.reserveStock()     // must import InventoryService
    analyticsService.recordSale()       // must import AnalyticsService
    // add another subscriber → edit OrderService
  }

With Pub/Sub — zero coupling:

  OrderService.placeOrder() {
    bus.publish('order.placed', data)   // that's it
  }
  // Add 10 more subscribers → never touch OrderService
```

**Dead letter / missed events:** A limitation of in-memory Pub/Sub is that subscribers must be registered *before* the event fires. Production systems (Kafka, RabbitMQ) use persistent queues so late subscribers can catch up.

## Diagram

```
[OrderService]──publish('order.placed')──▶[EventBus]──▶[EmailService]
                                                   └──▶[InventoryService]
                                                   └──▶[AnalyticsService]

Publisher has zero knowledge of subscribers.
Subscribers have zero knowledge of each other.
```

## ELI5

Think of a radio station. The DJ (publisher) broadcasts music. Anyone with a radio (subscriber) tuned to that frequency hears it. The DJ has no idea how many people are listening or who they are. New listeners can tune in anytime. The radio station (broker) just broadcasts — it doesn't manage the relationship between DJ and listener.
