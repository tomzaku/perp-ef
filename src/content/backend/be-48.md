**Compare gRPC, REST, and GraphQL — how do they differ architecturally and when should you use each?**

This is a classic system design interview question. Understanding the tradeoffs between API paradigms is essential for backend architects.

```text
// ═══════════════════════════════════
// DECISION FRAMEWORK
// ═══════════════════════════════════

Use REST when:
  ✓ Building a public-facing API (third parties, browsers)
  ✓ Simple CRUD operations
  ✓ HTTP caching is important (GET responses cache well)
  ✓ Team is familiar with HTTP semantics
  ✓ Webhooks and callbacks (standard HTTP POST)

Use gRPC when:
  ✓ Internal microservice-to-microservice calls
  ✓ High throughput, low latency is critical
  ✓ Need real-time bidirectional streaming
  ✓ Polyglot services (generate clients for 10+ languages)
  ✓ Strong typing and schema enforcement at compile time
  ✗ Avoid for browser clients (requires grpc-web + proxy)

Use GraphQL when:
  ✓ Multiple clients (web, iOS, Android) with different data needs
  ✓ Complex, deeply nested data graphs (social networks, e-commerce)
  ✓ Rapid frontend iteration (clients can adjust queries without API changes)
  ✓ Aggregating multiple backend services into one endpoint (BFF pattern)
  ✗ Avoid for simple CRUD (GraphQL adds complexity)
  ✗ Avoid when caching is critical (POST requests don't cache)

// ═══════════════════════════════════
// REAL-WORLD PATTERNS
// ═══════════════════════════════════

Netflix:
  gRPC between 1000s of microservices
  REST/GraphQL at the edge for clients

GitHub:
  REST API v3 (public, backwards-compatible)
  GraphQL API v4 (efficient, client-driven)

Shopify:
  REST for partners/integrations
  GraphQL for their own storefront API (complex product data)

Twitter/X:
  REST for public API
  Internal Finagle (RPC) between services
```

## Explanation

**Why not just use one protocol everywhere?** Each protocol optimizes for a different constraint:

- **REST** optimizes for *universal accessibility* — any HTTP client can consume it, results are cacheable, and the semantics (GET/POST/PUT/DELETE) are widely understood.
- **gRPC** optimizes for *performance and contract strictness* — Protobuf binary encoding, HTTP/2 multiplexing, and generated stubs eliminate entire categories of integration bugs between services.
- **GraphQL** optimizes for *client flexibility* — the server exposes a graph, and clients walk exactly the edges they need, eliminating over-fetching and enabling rapid frontend iteration.

**The hidden costs of GraphQL**: N+1 queries (DataLoader required), complex authorization per field, caching (POST doesn't cache), query depth attacks (need rate limiting by complexity), and a steeper learning curve for the team and API consumers.

**gRPC's hidden limitation**: Browsers cannot speak HTTP/2 directly (the grpc protocol, not just the transport) — you need grpc-web and a proxy like Envoy between browser and gRPC servers. This makes gRPC a poor choice for anything browser-facing.

## ELI5

Imagine ordering food. **REST** is like a menu with fixed combo meals — you order a combo and get everything in it. Simple, everyone knows how it works. **gRPC** is like a private kitchen intercom between chefs — super fast, precise orders in a shorthand code only they understand, but outsiders can't use it. **GraphQL** is like telling the waiter "I want exactly a burger without pickles, fries but no salt, and a Coke but half-sized" — you get exactly what you asked for, but the waiter's job is much harder.
