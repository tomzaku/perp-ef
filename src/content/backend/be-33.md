**What is database replication? Explain master-slave vs master-master and the tradeoffs.**

Replication is how production databases achieve high availability and read scalability. Understanding replication tradeoffs is essential for designing reliable data systems.

## Solution

```text
WHY REPLICATE?
══════════════

1. Read Scaling: distribute read queries across replicas
2. Fault Tolerance: if master dies, promote a replica
3. Geographic Distribution: replicas near users reduce latency
4. Backups: replicas serve as live backups


MASTER-SLAVE (Primary-Replica)
══════════════════════════════

  ┌──────────┐
  │  Master  │ ← All WRITES go here
  │(Primary) │
  └────┬─────┘
       │ Replication (binlog/WAL)
  ┌────┼────────────┐
  ↓    ↓            ↓
┌─────┐ ┌─────┐ ┌─────┐
│Slave│ │Slave│ │Slave│ ← READS distributed here
│  1  │ │  2  │ │  3  │
└─────┘ └─────┘ └─────┘

  Writes: Master only (single point of truth)
  Reads:  Any slave (scale horizontally)
  Best for: read-heavy workloads (most web apps: 90% reads)


MASTER-MASTER (Multi-Primary)
═════════════════════════════

  ┌──────────┐     ┌──────────┐
  │ Master 1 │←───→│ Master 2 │
  │  (R+W)   │     │  (R+W)   │
  └──────────┘     └──────────┘

  Both accept writes → conflict resolution needed!
  
  Conflicts: two masters update same row simultaneously
  Resolution strategies:
    - Last-write-wins (timestamp-based, may lose data)
    - Application-level resolution (merge logic)
    - Avoid: partition writes by region/shard

  Use cases: multi-region active-active, high write availability


SYNC vs ASYNC REPLICATION
═════════════════════════

Synchronous:
  Client → Master → Slave (confirm write) → Master → Client (ACK)
  ✓ Zero data loss (slave always has latest data)
  ✗ Higher latency (wait for slave)
  ✗ Master blocks if slave is slow/down

Asynchronous:
  Client → Master → Client (ACK)
            ↓ (background)
          Slave (eventually gets data)
  ✓ Lower latency (master doesn't wait)
  ✗ Data loss risk (master crashes before slave catches up)
  ✗ Replication lag (slave may be seconds behind)

Semi-synchronous:
  Wait for at least ONE slave to confirm
  Good balance: some durability guarantee + reasonable latency


REPLICATION MECHANISMS
══════════════════════

MySQL: Binary Log (binlog)
  - Statement-based: replicate SQL statements
  - Row-based: replicate actual row changes (more reliable)
  - Mixed: combination of both

PostgreSQL: Write-Ahead Log (WAL)
  - Physical: ship WAL segments (byte-level changes)
  - Logical: ship decoded changes (table-level, more flexible)

Process:
  1. Master writes changes to log (binlog/WAL)
  2. Slave connects and requests log from last position
  3. Slave applies changes sequentially
  4. Slave updates its position pointer


REPLICATION LAG PROBLEMS
═════════════════════════

User writes a comment → reads page → comment not there!

  Time 0: Client writes to Master
  Time 1: Client reads from Slave (hasn't replicated yet)
  Time 2: Slave catches up (too late, user confused)

Solutions:

  1. Read-after-write consistency:
     After a write, read from MASTER (not slave)
     → Simple, but reduces read scaling benefit

  2. Monotonic reads:
     Same user always reads from same replica
     → Prevents going "back in time"

  3. Causal consistency:
     Track which write a read depends on
     → Wait until slave has that write before reading

  4. Application routing:
     Write path:  → Master
     Read path:   → Slave (with acceptable lag)
     Read-own-write: → Master (for N seconds after write)


FAILOVER
════════

When master goes down:

  Automatic failover:
  1. Health check detects master is down
  2. Elect a slave as new master (most up-to-date)
  3. Redirect writes to new master
  4. Other slaves follow new master

  Risks:
  - Split brain: two nodes think they're master
    → Use fencing (STONITH: Shoot The Other Node In The Head)
  - Data loss: promoted slave may be behind
  - Stale reads during transition

  Tools: PostgreSQL Patroni, MySQL Group Replication,
         Redis Sentinel, MongoDB Replica Sets
```

## Explanation

Replication is the foundation of **high availability** in databases. Without replication, a single server failure means downtime and potential data loss. With replication, your system can survive hardware failures and scale read capacity by distributing queries across multiple copies.

The key tradeoff is **consistency vs latency**. Synchronous replication guarantees every replica has the latest data but makes writes slower (must wait for replica confirmation). Asynchronous replication is faster but creates a window where replicas are behind the master — if the master crashes during that window, those writes are lost.

In practice, most production systems use **asynchronous replication with application-level consistency guarantees**. The application knows which operations need strong consistency (read-after-write for user's own data) and routes those reads to the master, while tolerating slightly stale reads for other operations (showing a feed, listing products).

## ELI5

Database replication is like having multiple copies of a notebook. There's one official notebook (master) where you write new entries. Helpers (slaves) copy from it so when someone asks a question, any helper can answer — you don't have to ask the one person with the official notebook. If the person with the official notebook gets sick, one of the helpers becomes the new official notebook keeper. The tricky part is that helpers might not have copied the latest page yet, so they might give slightly outdated answers.
