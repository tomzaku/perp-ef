**What are the different chunking strategies for RAG and how do you choose the right one?**

Chunking — how you split documents into pieces for embedding and retrieval — is one of the most impactful decisions in a RAG pipeline. Poor chunking leads to poor retrieval, regardless of your embedding model or LLM.

## Solution

```text
CHUNKING STRATEGIES
═══════════════════

1. FIXED-SIZE CHUNKING
   Split by token/character count with overlap.

   Document: [===========================================]
   Chunk 1:  [========]
   Chunk 2:       [========]    ← 50 token overlap
   Chunk 3:            [========]

   Pros: Simple, predictable chunk sizes
   Cons: May split mid-sentence or mid-idea

2. RECURSIVE CHARACTER SPLITTING
   Try splitting by larger separators first, fall back to smaller.

   Try: "\n\n" (paragraphs) → "\n" (lines) → ". " (sentences) → " " (words)

   Pros: Respects natural text boundaries
   Cons: Variable chunk sizes

3. SEMANTIC CHUNKING
   Use embedding similarity to detect topic boundaries.

   Sentence embeddings: [s1, s2, s3, s4, s5, s6, s7, s8]
   Similarity:           0.9  0.8  0.3  0.9  0.8  0.2  0.9
                                    ↑                ↑
                              Topic boundary    Topic boundary
   Chunks: [s1,s2,s3] | [s4,s5,s6] | [s7,s8]

   Pros: Preserves semantic coherence
   Cons: Slower, requires embedding model

4. DOCUMENT-STRUCTURE-AWARE
   Split by headers, sections, or HTML/Markdown structure.

   # Chapter 1        → Chunk boundary
   ## Section 1.1     → Chunk boundary
   Paragraph text...

   Pros: Preserves document organization
   Cons: Requires structured input

5. AGENTIC CHUNKING
   Use an LLM to determine optimal split points.
   Pros: Best semantic preservation
   Cons: Expensive, slow


KEY PARAMETERS
══════════════
Parameter    │ Range           │ Impact
─────────────┼─────────────────┼────────────────────────────
Chunk size   │ 256-1024 tokens │ Smaller = precise, Larger = context
Overlap      │ 10-20%          │ Prevents losing context at boundaries
Metadata     │ title, source   │ Enables filtering before vector search


PARENT-CHILD CHUNKING (Advanced)
════════════════════════════════
Embed small chunks for precise retrieval,
return the parent (larger) chunk for context:

Parent chunk (2000 tokens): [Full section of document]
  ├── Child chunk 1 (200 tokens): [embedded, used for search]
  ├── Child chunk 2 (200 tokens): [embedded, used for search]
  └── Child chunk 3 (200 tokens): [embedded, used for search]

Search hits child 2 → Return parent chunk to LLM


DECISION TREE
═════════════
Is your data structured (HTML, Markdown, code)?
  → Yes: Use document-structure-aware chunking
  → No: Is semantic coherence critical?
    → Yes: Use semantic chunking
    → No: Start with recursive character splitting
```

## Explanation

Chunking is the bridge between raw documents and your vector database. The goal is to create chunks that are **small enough to be relevant** to specific queries, but **large enough to contain complete, useful information**.

It's a fundamental tradeoff: smaller chunks give more precise retrieval but may lack context; larger chunks provide more context but may dilute relevance.

**Start simple** (recursive splitting, 512 tokens, 10% overlap), measure retrieval quality, then optimize. Don't over-engineer chunking before you have evaluation metrics.

## ELI5

Imagine you're cutting a textbook into flashcards. Cut too small and each card has only a few words that don't make sense alone. Cut too big and you can't find the specific answer you need. The best flashcards have complete ideas — not too big, not too small — and overlap a little at the edges so you don't lose important connections between ideas.
