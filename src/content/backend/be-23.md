**How does DNS resolution work? Walk through the full lookup process.**

DNS is the internet's phone book — without it, you'd need to memorize IP addresses. Understanding the resolution process is key for debugging, infrastructure design, and security.

## Solution

```text
DNS HIERARCHY
═════════════

                    . (Root)
                   /    |    \
               .com   .org   .io    ← TLD (Top-Level Domain)
              /    \
        google.com  github.com      ← Second-Level Domain
        /       \
    www.google   mail.google        ← Subdomain


FULL DNS RESOLUTION FLOW
═════════════════════════

User types: www.example.com

Step 1: Browser Cache
  → Check browser's DNS cache (Chrome: chrome://net-internals/#dns)
  → If found and TTL not expired → use cached IP ✓

Step 2: OS Cache
  → Check operating system's DNS cache
  → Also checks /etc/hosts file (local overrides)
  → Linux: systemd-resolved | Mac: mDNSResponder

Step 3: Resolver (Recursive DNS Server)
  → Usually your ISP's DNS or public DNS (8.8.8.8, 1.1.1.1)
  → Has its own cache — if found, return immediately ✓
  → If not cached, starts the recursive lookup:

Step 4: Root Nameserver (.)
  Resolver → "Where is www.example.com?"
  Root     → "I don't know, but .com is handled by these TLD servers"
  (13 root server clusters worldwide, identified as a.root-servers.net)

Step 5: TLD Nameserver (.com)
  Resolver → "Where is www.example.com?"
  TLD      → "example.com is handled by ns1.example.com (1.2.3.4)"

Step 6: Authoritative Nameserver
  Resolver → "What is the IP for www.example.com?"
  Auth NS  → "www.example.com = 93.184.216.34, TTL=3600"

Step 7: Response flows back
  Auth NS → Resolver (caches for TTL) → OS → Browser (caches) → App


DNS RECORD TYPES
════════════════

┌────────┬─────────────────────────────────────────────────┐
│ Type   │ Purpose                                         │
├────────┼─────────────────────────────────────────────────┤
│ A      │ Domain → IPv4 address (93.184.216.34)           │
│ AAAA   │ Domain → IPv6 address                           │
│ CNAME  │ Domain → another domain (alias)                 │
│        │ www.example.com → example.com                   │
│ MX     │ Mail server for domain (priority + server)      │
│ NS     │ Authoritative nameservers for domain            │
│ TXT    │ Arbitrary text (SPF, DKIM, domain verification) │
│ SOA    │ Start of Authority (zone admin info)            │
│ SRV    │ Service location (host + port for a service)    │
│ PTR    │ Reverse DNS: IP → domain name                   │
└────────┴─────────────────────────────────────────────────┘


WHY DNS USES UDP (MOSTLY)
═════════════════════════

- DNS queries are small (< 512 bytes typically)
- UDP is faster: no handshake overhead
- Single request-response pattern fits UDP perfectly
- Falls back to TCP when:
  → Response > 512 bytes (or 4096 with EDNS)
  → Zone transfers between DNS servers (AXFR)
  → DNS over TLS (DoT) on port 853


DNS CACHING & TTL
═════════════════

TTL (Time To Live) = how long to cache the record (in seconds)

example.com  A  93.184.216.34  TTL=3600  ← cache for 1 hour

Low TTL (60s):   Faster DNS changes, more DNS queries
High TTL (86400): Fewer queries, slower propagation of changes

Common strategy:
  - Normal: TTL=3600 (1 hour)
  - Before migration: Lower TTL to 60s days in advance
  - After migration: Update IP, wait for old TTL to expire
  - After stable: Raise TTL back to 3600


DNS FOR LOAD BALANCING
══════════════════════

Round-robin DNS:
  example.com  A  1.2.3.4
  example.com  A  5.6.7.8
  example.com  A  9.10.11.12
  → Resolver returns different order each time

GeoDNS:
  → Returns different IPs based on client location
  → US users → US datacenter, EU users → EU datacenter

Limitations:
  - No health checking (returns IPs of dead servers)
  - Caching ignores changes until TTL expires
  → Use with actual load balancers for production
```

## Explanation

DNS is a **hierarchical, distributed database** designed for high availability and performance. No single server knows every domain — the responsibility is delegated down the hierarchy. Root servers know about TLDs, TLD servers know about domain registrars, and authoritative servers know the actual IP addresses.

**Caching is critical** to DNS performance. Without it, every web request would require 4+ round trips to resolve a domain. The TTL mechanism lets domain owners control the tradeoff between fast updates and reduced DNS traffic. In practice, most DNS lookups are served from cache in under 1ms.

DNS is also a common interview topic for **security**: DNS spoofing/poisoning attacks can redirect users to malicious servers. DNSSEC adds cryptographic signatures to prevent tampering. DNS over HTTPS (DoH) and DNS over TLS (DoT) encrypt queries to prevent eavesdropping.

## ELI5

DNS is like a phone book for the internet. When you want to call your friend "google.com," you don't know their phone number (IP address). So you ask the phone book service. It doesn't know everyone's number either, so it asks the "area code" office (.com), which points you to Google's personal secretary (authoritative nameserver), who gives you the actual number. Once you know the number, you write it on a sticky note (cache) so you don't have to look it up again for a while.
