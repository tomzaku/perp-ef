**Explain concurrency vs parallelism. What are race conditions, and how do you prevent deadlocks?**

Concurrency bugs are among the hardest to debug because they're non-deterministic. Understanding locking primitives and deadlock conditions is essential for any backend developer.

## Solution

```text
CONCURRENCY vs PARALLELISM
═══════════════════════════

Concurrency (1 core, time-slicing):
  Thread A: ██░░██░░██
  Thread B: ░░██░░██░░
  → Tasks INTERLEAVE on single core

Parallelism (multiple cores):
  Core 1: ████████  Thread A
  Core 2: ████████  Thread B
  → Tasks execute SIMULTANEOUSLY

Concurrency = dealing with multiple things at once (structure)
Parallelism = doing multiple things at once (execution)


RACE CONDITION
══════════════

Two threads increment a shared counter (counter = 0):

  Thread A                  Thread B
  read counter (0)
                            read counter (0)
  increment (0 → 1)
                            increment (0 → 1)
  write counter (1)
                            write counter (1)

  Expected: 2    Got: 1    ← Race condition!

The CRITICAL SECTION is the code that accesses shared data.
Only one thread should be in the critical section at a time.


LOCKING PRIMITIVES
══════════════════

1. MUTEX (Mutual Exclusion)
   - Binary lock: locked or unlocked
   - Only the thread that locked it can unlock it
   - If locked, other threads SLEEP (yield CPU)

   mutex.lock()
   counter++          // critical section
   mutex.unlock()

   Best for: protecting shared data, general-purpose locking


2. SEMAPHORE (Counting)
   - Counter-based: allows N concurrent accesses
   - Any thread can signal (release), not just the acquirer
   - Binary semaphore (N=1) similar to mutex

   semaphore = Semaphore(3)   // allow 3 concurrent
   semaphore.acquire()        // counter-- (block if 0)
   use_limited_resource()
   semaphore.release()        // counter++

   Best for: limiting concurrent access (connection pool, rate limit)


3. SPINLOCK (Busy Wait)
   - Thread loops ("spins") checking lock instead of sleeping
   - No context switch overhead
   - Wastes CPU cycles while waiting

   while (!lock.tryAcquire()) { /* spin */ }
   counter++
   lock.release()

   Best for: very short critical sections (nanoseconds)
   Worst for: long waits (wastes CPU)


4. READ-WRITE LOCK
   - Multiple readers OR one writer (not both)
   - Readers don't block each other

   rwlock.readLock()    // multiple threads can hold
   data = read_shared()
   rwlock.readUnlock()

   rwlock.writeLock()   // exclusive access
   write_shared(data)
   rwlock.writeUnlock()

   Best for: read-heavy workloads (config, caches)


COMPARISON TABLE
════════════════

┌────────────┬──────────┬──────────┬────────────┬───────────┐
│ Primitive  │ Blocking │ Count    │ Ownership  │ Use Case  │
├────────────┼──────────┼──────────┼────────────┼───────────┤
│ Mutex      │ Sleep    │ Binary   │ Yes        │ General   │
│ Semaphore  │ Sleep    │ N        │ No         │ Limiting  │
│ Spinlock   │ Busy-wait│ Binary   │ Yes        │ Short CS  │
│ RW Lock    │ Sleep    │ N read/  │ Yes        │ Read-heavy│
│            │          │ 1 write  │            │           │
└────────────┴──────────┴──────────┴────────────┴───────────┘


DEADLOCK
════════

Four conditions (ALL must hold for deadlock):

1. Mutual Exclusion   → Resource held exclusively
2. Hold and Wait      → Thread holds one, waits for another
3. No Preemption      → Can't force-take a resource
4. Circular Wait      → A waits for B, B waits for A

Example:
  Thread A: lock(mutex1) → wants lock(mutex2) → BLOCKED
  Thread B: lock(mutex2) → wants lock(mutex1) → BLOCKED
  → Both wait forever = DEADLOCK

Prevention (break any one condition):
  1. Lock ordering: always acquire locks in same order
     → Breaks circular wait
     → Most common solution

  2. Try-lock with timeout:
     if (!mutex2.tryLock(timeout)) {
       mutex1.unlock()  // release and retry
     }
     → Breaks hold-and-wait

  3. Lock-free data structures:
     → Use atomic operations (CAS - Compare And Swap)
     → Breaks mutual exclusion

Detection:
  - Build a wait-for graph
  - If cycle exists → deadlock
  - Database engines do this (detect + abort one transaction)


PRACTICAL EXAMPLES
══════════════════

// JavaScript/Node.js (single-threaded but async race conditions!)
let balance = 100
// Both run concurrently via event loop:
async function withdraw(amount) {
  const current = await getBalance()  // both read 100
  if (current >= amount) {
    await setBalance(current - amount) // both set to 50
  }
}
// Fix: use atomic operations or database transactions

// Go (goroutines + mutex)
var mu sync.Mutex
var counter int

func increment() {
    mu.Lock()
    defer mu.Unlock()
    counter++
}

// Java (synchronized)
synchronized void increment() {
    counter++;
}
```

## Explanation

Concurrency bugs are notoriously hard to debug because they depend on **timing** — a race condition might occur once in a million runs but cause catastrophic data corruption. The key is to identify **every** shared mutable state and protect it with appropriate synchronization.

**Choosing the right primitive matters for performance.** A mutex is the safe default. Use a spinlock only for critical sections shorter than a context switch (~microseconds). Use a read-write lock when reads vastly outnumber writes. Use a semaphore when you need to limit concurrency (like a connection pool with 10 max connections).

**Deadlock prevention** is simpler than deadlock detection. The most practical approach is **lock ordering** — establish a global order for all locks and always acquire them in that order. This makes circular wait impossible. In database systems, deadlock detection (finding cycles in a wait-for graph) is more common because you can't always control lock ordering across distributed transactions.

## ELI5

Imagine two kids each holding one shoe from a pair. Kid A has the left shoe and wants the right. Kid B has the right shoe and wants the left. Neither will give up their shoe until they get the other one. They're stuck forever — that's a deadlock! The fix: make a rule that everyone picks up the left shoe first, then the right. A race condition is like two kids running to grab the last cookie — whoever gets there first wins, and the result is unpredictable.
