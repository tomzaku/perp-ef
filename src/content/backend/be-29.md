**How do processes communicate with each other? Explain the main IPC mechanisms and when to use each.**

Since processes have isolated memory spaces, they need explicit mechanisms to share data. IPC is fundamental to system design — from shell pipelines to microservice architectures.

PIPES
═════

Unnamed pipe: parent ←→ child only
  int fd[2];
  pipe(fd);         // fd[0]=read, fd[1]=write
  if (fork() == 0) {
    close(fd[1]);   // child closes write end
    read(fd[0], buf, size);  // reads from parent
  } else {
    close(fd[0]);   // parent closes read end
    write(fd[1], data, size); // writes to child
  }

Shell example: ls | grep ".txt" | wc -l
  → 3 processes connected by 2 pipes

Named pipe (FIFO): any processes, persists as file
  mkfifo /tmp/myfifo
  echo "hello" > /tmp/myfifo   # writer (blocks until reader)
  cat /tmp/myfifo               # reader


SHARED MEMORY
═════════════

Fastest IPC: no kernel involvement after setup

  Process A                    Process B
  ┌──────────┐                ┌──────────┐
  │ Virtual  │                │ Virtual  │
  │ Memory   │                │ Memory   │
  │    ↓     │                │    ↓     │
  └────┬─────┘                └────┬─────┘
       │                           │
       └─────→ Physical RAM ←─────┘
              (shared region)

  // Create shared memory
  int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);
  ftruncate(fd, 4096);
  void* ptr = mmap(NULL, 4096, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);

  // Now both processes read/write through ptr
  // MUST use synchronization (mutex/semaphore) to avoid races!


SIGNALS
═══════

Asynchronous notifications to a process:
  SIGTERM (15) → graceful shutdown request
  SIGKILL (9)  → force kill (cannot be caught)
  SIGINT (2)   → Ctrl+C interrupt
  SIGUSR1/2    → user-defined signals
  SIGCHLD      → child process terminated

  kill -TERM <pid>    // send SIGTERM
  signal(SIGTERM, handler_function)  // register handler


MESSAGE QUEUES
══════════════

Structured, asynchronous message passing:
  - Messages have types/priorities
  - Persist until consumed
  - Decouples sender and receiver

  POSIX message queue:
  mqd_t mq = mq_open("/myqueue", O_CREAT | O_WRONLY, 0666, &attr);
  mq_send(mq, message, strlen(message), priority);
  mq_receive(mq, buffer, MAX_SIZE, &priority);

  Modern equivalent: Redis, RabbitMQ, Kafka (cross-machine)


FORK & COPY-ON-WRITE (COW)
═══════════════════════════

fork() creates a child process:

  Before fork:
  Parent: [Code | Data | Heap | Stack]

  After fork:
  Parent: [Code | Data | Heap | Stack] ──→ Physical Pages
  Child:  [Code | Data | Heap | Stack] ──→ Same Physical Pages!
                                            (shared, read-only)

  Copy-on-Write:
  - Child gets copy of parent's page table (not memory!)
  - Pages marked read-only
  - When either writes → page fault → OS copies just that page
  - Huge savings: fork() is nearly instant even for large processes

  Child inherits: memory (COW), file descriptors, signal handlers
  Child gets new: PID, own copy of modified pages

  Dirty COW (CVE-2016-5195):
  - Race condition in COW implementation
  - Allowed unprivileged users to write to read-only files
  - Major Linux kernel vulnerability (2016)


FILE DESCRIPTOR INHERITANCE
═══════════════════════════

Child inherits parent's open file descriptors:

  Parent opens socket on fd=5
  fork()
  Child also has fd=5 pointing to same socket!

  Used by: web servers (parent accepts, child handles)
  Gotcha: both parent and child must close fd to actually close it

## Explanation

IPC is necessary because processes are **intentionally isolated** — one process can't read another's memory. This isolation is great for security and stability, but sometimes processes need to cooperate. The IPC mechanism you choose depends on your requirements.

**Pipes** are the simplest and most Unix-like approach, perfect for streaming data between related processes (like shell commands). **Shared memory** is the fastest because data isn't copied through the kernel — both processes read/write the same physical memory directly. But it requires careful synchronization to prevent race conditions.

**Copy-on-Write** is a critical optimization to understand. When `fork()` is called, the OS doesn't duplicate the parent's entire memory — it shares the same physical pages and only copies when one process writes. This makes fork() nearly free regardless of process size, which is why fork-based web servers (like traditional Apache/PHP) can work efficiently.

## ELI5

Processes are like people living in separate apartments — they can't see into each other's rooms. IPC is how they communicate. A **pipe** is like a tube between apartments — you push a message in one end, they pull it out the other. **Shared memory** is like a shared whiteboard in the hallway — both can read and write, but they need to take turns. **Signals** are like knocking on the door — simple notifications. And **fork** is like cloning yourself — the clone starts with copies of all your stuff, but after that, you each go your own way.
