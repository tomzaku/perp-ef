**How should you store passwords, API keys, and other secrets in a production system?**

Secret management is where security theory meets daily practice. Every backend developer handles credentials — doing it wrong leads to data breaches.

## Solution

```text
NEVER DO THIS
══════════════

  // Hardcoded in source code
  const DB_PASSWORD = "supersecret123"
  const API_KEY = "sk_live_abc123"

  // In committed .env file
  git add .env  → now in git history FOREVER

  // Plaintext in database
  INSERT INTO users (email, password) VALUES ('a@b.com', 'mypassword')

  // MD5/SHA for passwords
  const hash = md5(password)  // cracked in milliseconds


PASSWORD STORAGE
════════════════

Level 0 (WORST): Plaintext
  → Database breach = all passwords exposed

Level 1 (BAD): MD5/SHA hash
  → Rainbow table attack cracks most passwords instantly
  → GPU: 10 billion SHA-256/second

Level 2 (BETTER): Salted hash
  → Unique salt per user prevents rainbow tables
  → But fast hash still allows brute force

Level 3 (CORRECT): bcrypt/argon2
  → Intentionally slow (configurable cost factor)
  → Built-in unique salt
  → Resistant to GPU/ASIC attacks
  → Industry standard

  // Node.js
  const bcrypt = require('bcrypt')
  const hash = await bcrypt.hash(password, 12)  // 12 rounds
  const valid = await bcrypt.compare(input, hash)

  // Stored in DB:
  // $2b$12$LJ3m4yPwsSRN5GEZLY3jie.wcwRCHNWi2jCvfPSd3FEdK.W0BFWC
  //  ↑  ↑   ↑ salt (22 chars)        ↑ hash (31 chars)
  // algo cost


APPLICATION SECRETS
═══════════════════

Hierarchy (least to most secure):

1. Environment Variables (minimum acceptable)
   + Not in source code
   + Easy to set per environment
   - Visible in process listing (ps auxe)
   - No audit trail
   - No rotation support

   DATABASE_URL=postgres://user:pass@host:5432/db
   API_KEY=sk_live_abc123

2. .env Files (development only)
   + Convenient for local development
   - MUST be in .gitignore
   - No encryption at rest
   - No rotation support

   # .gitignore
   .env
   .env.local
   .env.production

3. Secret Managers (production standard)
   + Encrypted at rest
   + Access control (IAM)
   + Audit logging (who accessed what when)
   + Automatic rotation
   + Version history

   Tools:
     HashiCorp Vault    → self-hosted, most flexible
     AWS Secrets Manager → AWS-native, auto-rotation
     GCP Secret Manager → GCP-native
     Azure Key Vault    → Azure-native
     Doppler            → cross-platform SaaS

   // AWS Secrets Manager example
   const secret = await secretsManager.getSecretValue({
     SecretId: 'prod/database/credentials'
   }).promise()
   const { username, password } = JSON.parse(secret.SecretString)


SECRET ROTATION
═══════════════

Rotate secrets regularly to limit blast radius of leaks:

  API Keys:
    1. Generate new key
    2. Deploy app with new key
    3. Verify new key works
    4. Revoke old key

  Database Credentials:
    1. Create new DB user with new password
    2. Update secret manager
    3. App picks up new credentials (no restart)
    4. Drop old DB user

  AWS Secrets Manager auto-rotation:
    Supports: RDS, DocumentDB, Redshift
    Lambda function handles rotation automatically
    Configurable schedule (30, 60, 90 days)


12-FACTOR APP: CONFIG
═════════════════════

Principle: strict separation of config from code

  Code:   same across all environments
  Config: changes between environments
    → dev DB, staging DB, prod DB = different configs, same code

  Config should be injected via:
    1. Environment variables (basic)
    2. Config files (mounted, not committed)
    3. Secret managers (best)

  NEVER: feature flags, API keys, or DB urls in code


DEFENSE IN DEPTH
════════════════

┌─────────────────────────────────────────┐
│ Layer 1: Prevention                     │
│   - .gitignore for .env files           │
│   - Pre-commit hooks scan for secrets   │
│   - Secret manager for all credentials  │
│   - Least privilege (minimal DB perms)  │
├─────────────────────────────────────────┤
│ Layer 2: Detection                      │
│   - Git secret scanning (GitHub, GitLab)│
│   - truffleHog / gitleaks in CI/CD      │
│   - Audit logs on secret access         │
│   - Anomaly detection on API usage      │
├─────────────────────────────────────────┤
│ Layer 3: Response                       │
│   - Automated key rotation on detection │
│   - Incident response runbook           │
│   - Git history rewrite (if committed)  │
│   - Revoke + regenerate immediately     │
└─────────────────────────────────────────┘


GIT SECRET DETECTION
════════════════════

Pre-commit hook (.pre-commit-config.yaml):
  - repo: https://github.com/gitleaks/gitleaks
    hooks:
      - id: gitleaks

CI/CD pipeline:
  gitleaks detect --source . --verbose

If secret already committed:
  1. Immediately revoke/rotate the secret
  2. git filter-branch or BFG Repo-Cleaner to remove from history
  3. Force push (destructive — coordinate with team)
  4. Assume the secret is compromised (even if "removed")


PRACTICAL CHECKLIST
═══════════════════

  [ ] .env in .gitignore
  [ ] No secrets in source code (use env vars minimum)
  [ ] Passwords hashed with bcrypt/argon2
  [ ] Secret manager for production credentials
  [ ] Pre-commit hooks scanning for secrets
  [ ] CI/CD scanning for leaked secrets
  [ ] Secret rotation policy in place
  [ ] Least-privilege database users
  [ ] Audit logging on secret access
  [ ] Incident response plan for leaked secrets
```

## Explanation

Secret management failures are behind many of the worst data breaches. The core principle is **separation of secrets from code** — secrets change across environments and must be rotatable without code changes. Hardcoded secrets are discovered when code is open-sourced, shared in screenshots, or accessed by compromised developer machines.

**Password hashing deserves special attention** because it protects your users directly. The difference between bcrypt and MD5 isn't theoretical — it's the difference between "attackers crack all passwords in hours" and "attackers can't crack any passwords in years." bcrypt/argon2's configurable cost factor means you can increase difficulty over time as hardware gets faster.

For **application secrets** (API keys, database credentials, encryption keys), the gold standard is a dedicated secret manager with automatic rotation, access control, and audit logging. Even if an attacker gains access to your application server, they can't extract secrets that are fetched at runtime from a separate service with its own authentication. This defense-in-depth approach means compromising one layer doesn't compromise everything.

## ELI5

Storing passwords in plaintext is like writing everyone's house key code on a whiteboard in the lobby. Hashing with bcrypt is like putting each code through a scrambler that takes 10 seconds per attempt — a thief would need years to try enough codes. For API keys, imagine you have a safe deposit box at a bank (secret manager) instead of keeping your valuables under the mattress (.env file). The bank has security cameras (audit logs), requires ID to access (IAM), and you can change the lock anytime (rotation).
