**What is Docker, how do containers work, and how do you use them in development and production?**

Containers are the standard unit of deployment for modern backend applications. Understanding Docker is essential for any full-stack developer.

## Solution

```text
CONTAINERS vs VMs
═════════════════

Virtual Machine:
  ┌──────────┐ ┌──────────┐
  │  App A   │ │  App B   │
  │  Libs    │ │  Libs    │
  │  Guest OS│ │  Guest OS│  ← Full OS per VM (heavy)
  ├──────────┴─┴──────────┤
  │      Hypervisor       │
  │      Host OS          │
  └───────────────────────┘

Container:
  ┌──────────┐ ┌──────────┐
  │  App A   │ │  App B   │
  │  Libs    │ │  Libs    │  ← Shares host OS kernel (lightweight)
  ├──────────┴─┴──────────┤
  │    Container Runtime   │
  │      Host OS          │
  └───────────────────────┘

Containers: Start in seconds, MBs, share OS kernel
VMs: Start in minutes, GBs, full OS isolation


DOCKERFILE — BUILD INSTRUCTIONS
═══════════════════════════════
```

```dockerfile
# ─── Node.js Production Dockerfile ───

# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                          # Install deps (deterministic)
COPY . .
RUN npm run build                   # Build TypeScript / bundle

# Stage 2: Production (smaller image)
FROM node:20-alpine AS runner
WORKDIR /app

# Don't run as root
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./

USER appuser
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

```text
MULTI-STAGE BUILD BENEFITS
══════════════════════════
- Builder stage has dev deps (TypeScript, build tools) — ~500MB
- Runner stage only has production code — ~100MB
- Source code not included in final image (security)


DOCKER COMPOSE — MULTI-CONTAINER
═════════════════════════════════
```

```yaml
# docker-compose.yml
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/myapp
      REDIS_URL: redis://redis:6379
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data    # Persist data
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d myapp"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  pgdata:                                   # Named volume
```

```text
# Commands
docker compose up -d        # Start all services (detached)
docker compose down         # Stop all services
docker compose logs app     # View app logs
docker compose exec app sh  # Shell into running container


KEY CONCEPTS
════════════

IMAGE: Read-only template (blueprint)
  - Built from Dockerfile
  - Stored in registry (Docker Hub, ECR, GCR)
  - Tagged: myapp:latest, myapp:v1.2.3

CONTAINER: Running instance of an image
  - Has its own filesystem, network, process space
  - Ephemeral — data is lost when container is removed
  - Use volumes for persistent data

VOLUME: Persistent storage that outlives containers
  - Named volumes: managed by Docker
  - Bind mounts: maps host directory into container

NETWORK: Containers communicate by service name
  - "db" resolves to the database container's IP
  - Services in same compose file share a network


DOCKERFILE BEST PRACTICES
═════════════════════════

1. Order matters for layer caching:
   COPY package*.json ./     ← Changes rarely (cached)
   RUN npm ci                ← Only reruns if package.json changed
   COPY . .                  ← Changes often (rebuilds from here)

2. Use .dockerignore:
   node_modules
   .git
   .env
   dist

3. Pin versions:
   FROM node:20.11-alpine    ← Not just "node:latest"

4. Don't run as root:
   USER appuser

5. Use multi-stage builds for smaller images

6. One process per container:
   ✗ Don't run app + nginx + redis in one container
   ✓ Each service gets its own container
```

## Explanation

Docker solves three core problems:
1. **"Works on my machine"** — Containers include everything the app needs, identical everywhere
2. **Environment parity** — Dev, staging, and prod run the same container
3. **Isolation** — Each service has its own dependencies, no conflicts

**For development:** Docker Compose gives you a full stack (app + database + cache + queue) with one command. New developers clone the repo, run `docker compose up`, and they're ready.

**For production:** Container images are the deployment artifact. Build once, push to registry, deploy to any environment. Kubernetes orchestrates containers at scale.

**Key mental model:** Think of a Docker image as a snapshot of your application at a point in time — code, dependencies, runtime, config — all frozen together.

## ELI5

Docker is like a lunchbox. Instead of saying "here's the recipe, good luck finding the ingredients," you pack the entire meal — food, utensils, napkin, everything — into a box. Anyone who gets the box has the exact same lunch. Docker Compose is like packing multiple lunchboxes for a picnic — one for the main course, one for drinks, one for dessert — and they all go together.
