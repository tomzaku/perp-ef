**How do sockets work? How can a server handle multiple clients on a single port?**

Sockets are the fundamental building block of all network communication. Every HTTP request, database connection, and WebSocket chat goes through sockets.

## Solution

```text
WHAT IS A SOCKET?
═════════════════

A socket = IP address + Port number
  → An endpoint for sending/receiving data

  Client socket:  192.168.1.10:54321  (ephemeral port)
  Server socket:  10.0.0.1:443        (well-known port)

In Linux: a socket is a file descriptor (integer)
  → read(), write(), close() work on sockets just like files


SOCKET LIFECYCLE (SERVER)
═════════════════════════

  socket()    → Create a socket (get file descriptor)
      ↓
  bind()      → Assign IP + port to the socket
      ↓
  listen()    → Mark socket as passive (ready for connections)
      ↓
  accept()    → Wait for client connection → returns NEW socket fd
      ↓           (original socket keeps listening)
  read/write  → Exchange data on the new socket
      ↓
  close()     → Tear down connection


HOW MULTIPLE CLIENTS ON ONE PORT?
═════════════════════════════════

Server listens on port 80. Three clients connect:

  Connection 1: (Client_A:50001, Server:80)  → fd=5
  Connection 2: (Client_B:50002, Server:80)  → fd=6
  Connection 3: (Client_A:50003, Server:80)  → fd=7

Each connection is a unique 4-tuple:
  (source IP, source port, destination IP, destination port)

The listening socket (fd=4) stays on port 80.
accept() creates a NEW fd for each client connection.
Server reads/writes to the specific fd for each client.


CONNECTION HANDLING MODELS
══════════════════════════

1. PROCESS-PER-CONNECTION (Apache prefork)
   accept() → fork() → child handles client
   + Simple isolation
   - Heavy: each process uses MB of memory
   - Max thousands of connections

2. THREAD-PER-CONNECTION (Java traditional)
   accept() → new thread handles client
   + Lighter than processes
   - Still limited to ~10K threads
   - Context switching overhead

3. EVENT-DRIVEN / I/O MULTIPLEXING (Node.js, Nginx)
   Single thread monitors many sockets:

   while (true) {
     ready_fds = epoll_wait(epoll_fd)  // blocks until activity
     for (fd in ready_fds) {
       if (fd == server_fd) accept()   // new connection
       else handle_data(fd)            // read/write
     }
   }

   + Handles 100K+ connections (C10K problem solved)
   - Complex programming model

   I/O multiplexing APIs:
     select()  → oldest, limited to 1024 fds
     poll()    → no fd limit, but O(n) scanning
     epoll()   → Linux, O(1) for ready fds (best)
     kqueue()  → macOS/BSD equivalent of epoll


EPHEMERAL PORTS
═══════════════

Client port assigned automatically by OS
  → Range: 49152-65535 (configurable)
  → Limits: ~16K connections per (client IP, server IP, server port)

Multiple browser tabs to same server:
  Tab 1: (your_ip:52000, google:443)
  Tab 2: (your_ip:52001, google:443)
  → Different source ports = different connections


UNIX SOCKETS vs TCP SOCKETS
════════════════════════════

TCP Socket:  Communicates over network (IP + port)
Unix Socket: Communicates on same machine (file path)

  TCP:  mysql -h 127.0.0.1 -P 3306    → through network stack
  Unix: mysql --socket=/tmp/mysql.sock → bypasses network stack

Unix sockets are faster for local communication:
  - No TCP overhead (no handshake, checksums, routing)
  - Used by: Docker, PostgreSQL, Redis, Nginx to PHP-FPM


SOCKET BUFFERS
══════════════

Each socket has a send buffer and receive buffer in kernel:

  App write() → [Send Buffer] → Network → [Recv Buffer] → App read()

  - write() copies data to send buffer and returns immediately
  - If send buffer full → write() blocks (TCP backpressure)
  - If recv buffer full → TCP tells sender to slow down (flow control)
  - Buffer sizes: typically 64KB-256KB (tunable via setsockopt)


MAX CONNECTIONS ON A SERVER
═══════════════════════════

Theoretical: ~2^48 (all combinations of client IP + port)
Practical limits:
  - File descriptors: default 1024, tunable to ~1M (ulimit -n)
  - Memory: each connection uses few KB (kernel + app state)
  - CPU: handling connections takes processing time

Real-world:
  - Nginx: routinely handles 100K+ concurrent connections
  - WhatsApp: 2M+ connections per server (Erlang)
```

## Explanation

Sockets are the **interface between your application and the network stack**. In Linux, they follow the "everything is a file" philosophy — a socket is just a file descriptor that you read from and write to. This unification means the same system calls (read, write, close) work for files, pipes, and network connections.

The key insight for handling multiple clients is the **4-tuple uniqueness**. Many developers think "one port = one connection," but that's wrong. A server on port 80 can handle millions of connections because each is uniquely identified by the combination of source IP, source port, destination IP, and destination port. The `accept()` call creates a new file descriptor for each connection while the original socket continues listening.

The evolution from **process-per-connection to event-driven** architectures (epoll/kqueue) solved the C10K problem. Node.js, Nginx, and Redis all use this model — a single thread efficiently manages thousands of connections by only processing sockets that have data ready, rather than blocking on each one.

## ELI5

A socket is like a phone line. The server is a business with one published phone number (port 80). When someone calls, the receptionist (listening socket) answers and transfers them to a new line (new socket). The receptionist goes back to waiting for the next call. That's how one phone number handles many callers at once — each caller gets their own line, but they all dialed the same number.
