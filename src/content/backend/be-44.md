**How do you implement caching in front of a database? Compare caching strategies and explain how to handle invalidation.**

Caching is one of the most asked performance topics because every production system uses it. Interviewers want to see you understand patterns, trade-offs, and failure modes — not just "put Redis in front of it."

## Solution

```text
WHY CACHE?
══════════

Database query:  ~5-50ms  (with index)
Redis GET:       ~0.5ms   (10-100x faster)
In-memory:       ~0.001ms (1000x faster)

Caching is the #1 way to reduce database load
and improve response times for read-heavy workloads.


CACHING LAYERS
══════════════

┌────────────────────────────────────────────────┐
│ Client (Browser)                                │
│  └─ HTTP Cache (Cache-Control, ETag)           │
├────────────────────────────────────────────────┤
│ CDN (CloudFront, Cloudflare)                   │
│  └─ Static assets, API responses               │
├────────────────────────────────────────────────┤
│ Application (In-Memory)                        │
│  └─ LRU cache, computed values                 │
├────────────────────────────────────────────────┤
│ Distributed Cache (Redis, Memcached)           │

├────────────────────────────────────────────────┤
│ Database (Query Cache, Buffer Pool)            │
│  └─ Materialized views, connection pool        │
└────────────────────────────────────────────────┘

Each layer reduces load on the layer below it.


═══════════════════════════════════════════════════════
PATTERN 1: CACHE-ASIDE (Lazy Loading)
═══════════════════════════════════════════════════════

Most common pattern. Application manages the cache.

  Read:                      Write:
  ┌─────┐  1. check  ┌─────┐   ┌─────┐  1. write  ┌────┐
  │ App │──────────→│Cache │   │ App │───────────→│ DB │
  │     │←──────────│     │   │     │  2. delete  │    │
  │     │  (hit?)   └─────┘   │     │──────────→ │    │
  │     │                     └─────┘   ┌─────┐  └────┘
  │     │  2. query   ┌────┐            │Cache │
  │     │───────────→│ DB │            │(del) │
  │     │←───────────│    │            └─────┘
  │     │  3. set     └────┘
  │     │──────────→┌─────┐
  └─────┘           │Cache │
                    └─────┘
```

```javascript
// Cache-Aside implementation
const Redis = require('ioredis');
const redis = new Redis();

async function getUser(userId) {
  const cacheKey = `user:${userId}`;

  // 1. Check cache
  const cached = await redis.get(cacheKey);
  if (cached) {
    return JSON.parse(cached);  // Cache HIT
  }

  // 2. Cache MISS → query database
  const user = await db.query(
    'SELECT * FROM users WHERE id = $1', [userId]
  );

  // 3. Populate cache with TTL
  await redis.set(cacheKey, JSON.stringify(user), 'EX', 300);

  return user;
}

async function updateUser(userId, data) {
  // 1. Update database (source of truth)
  await db.query(
    'UPDATE users SET name = $1 WHERE id = $2',
    [data.name, userId]
  );

  // 2. Invalidate cache (NOT update — avoids race conditions)
  await redis.del(`user:${userId}`);
}

// Pros: Only caches data that's actually requested
// Cons: Cache miss = slower (DB query + cache write)
//       Stale data possible between DB write and cache delete
```

```text
═══════════════════════════════════════════════════════
PATTERN 2: READ-THROUGH / WRITE-THROUGH
═══════════════════════════════════════════════════════

Cache sits between app and DB. Cache handles DB reads/writes.

  ┌─────┐  read/write  ┌─────┐  auto-sync  ┌────┐
  │ App │──────────────→│Cache│─────────────→│ DB │
  │     │←──────────────│     │←─────────────│    │
  └─────┘               └─────┘             └────┘

  App only talks to the cache layer.
  Cache is always consistent with DB.

  Pros: Simpler app code, cache always fresh
  Cons: Initial read is slow (must load from DB)
        Write latency includes both cache + DB


═══════════════════════════════════════════════════════
PATTERN 3: WRITE-BEHIND (Write-Back)
═══════════════════════════════════════════════════════

Cache buffers writes and flushes to DB asynchronously.

  ┌─────┐  write  ┌─────┐  async batch  ┌────┐
  │ App │────────→│Cache│──────────────→│ DB │
  │     │  (fast) │     │  (background)  │    │
  └─────┘         └─────┘               └────┘

  Pros: Fastest writes (app doesn't wait for DB)
        Batch DB writes reduce load
  Cons: Data loss risk if cache crashes before flush
        Complex to implement correctly
  Use for: Analytics counters, view counts, activity logs
```

```javascript
// Write-behind example: View counter
// Don't write to DB on every page view — buffer in Redis

async function recordPageView(pageId) {
  // Increment in Redis (fast, in-memory)
  await redis.hincrby('page_views', pageId, 1);
}

// Background job: Flush to DB every 60 seconds
async function flushViewCounts() {
  const counts = await redis.hgetall('page_views');
  if (Object.keys(counts).length === 0) return;

  // Batch update DB
  const values = Object.entries(counts)
    .map(([id, count]) => `('${id}', ${count})`)
    .join(',');

  await db.query(`
    INSERT INTO page_views (page_id, view_count)
    VALUES ${values}
    ON CONFLICT (page_id)
    DO UPDATE SET view_count = page_views.view_count + EXCLUDED.view_count
  `);

  // Clear flushed counts
  await redis.del('page_views');
}
```

```text
═══════════════════════════════════════════════════════
CACHE INVALIDATION STRATEGIES
═══════════════════════════════════════════════════════

"There are only two hard things in computer science:
 cache invalidation and naming things." — Phil Karlton


Strategy 1: TTL (Time-To-Live)
──────────────────────────────
  SET key value EX 300   -- expires in 5 minutes

  Simple. Guarantees staleness is bounded.
  Use for: product listings, search results, configs
  Risk: Data is stale for up to TTL duration


Strategy 2: Event-Driven Invalidation
─────────────────────────────────────
  On data change → delete/update cache immediately

  App code:    UPDATE users SET ... → redis.del("user:123")
  CDC/Events:  DB change → Kafka → cache invalidation service
  Pub/Sub:     redis.publish("invalidate", "user:123")

  Most consistent. Use for: user profiles, cart, permissions
  Risk: Cache delete can fail → add TTL as safety net


Strategy 3: Versioned Keys
──────────────────────────
  Cache key includes a version number:
    "user:123:v5" → { name: "Alice", ... }

  On update: increment version in a fast counter
    redis.incr("user:123:version")  →  v6
    New reads build key "user:123:v6" → cache miss → fresh data
    Old "user:123:v5" expires via TTL

  Use for: objects that change frequently
  Benefit: No explicit delete needed


═══════════════════════════════════════════════════════
CACHE FAILURE MODES & SOLUTIONS
═══════════════════════════════════════════════════════


Problem 1: CACHE STAMPEDE (Thundering Herd)
───────────────────────────────────────────
  Popular key expires → 1000 requests hit DB simultaneously

  ┌──────┐     cache miss!     ┌────┐
  │Req 1 │─────────────────────│ DB │  ← 1000 identical
  │Req 2 │─────────────────────│    │     queries at once
  │Req 3 │─────────────────────│    │
  │ ...  │─────────────────────│    │
  └──────┘                     └────┘
```

```javascript
// Solution A: Mutex lock (only one request queries DB)
async function getWithLock(key, fetchFn, ttl = 300) {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const lockKey = `lock:${key}`;
  const acquired = await redis.set(lockKey, '1', 'NX', 'EX', 5);

  if (acquired) {
    // Winner: fetch from DB and populate cache
    const data = await fetchFn();
    await redis.set(key, JSON.stringify(data), 'EX', ttl);
    await redis.del(lockKey);
    return data;
  } else {
    // Losers: wait briefly, then retry from cache
    await new Promise(r => setTimeout(r, 100));
    return getWithLock(key, fetchFn, ttl);
  }
}

// Solution B: Stale-while-revalidate
// Serve stale data immediately, refresh in background
async function getWithSWR(key, fetchFn, ttl = 300, staleTTL = 600) {
  const cached = await redis.get(key);
  const staleAt = await redis.get(`${key}:stale_at`);

  if (cached) {
    if (staleAt && Date.now() > parseInt(staleAt)) {
      // Data is stale — refresh in background
      refreshInBackground(key, fetchFn, ttl, staleTTL);
    }
    return JSON.parse(cached);  // Return immediately (maybe stale)
  }

  // No cache at all — must wait for DB
  const data = await fetchFn();
  await redis.set(key, JSON.stringify(data), 'EX', staleTTL);
  await redis.set(`${key}:stale_at`, Date.now() + ttl * 1000, 'EX', staleTTL);
  return data;
}
```

```text
Problem 2: CACHE PENETRATION
─────────────────────────────
  Requests for non-existent keys always miss cache → hit DB

  Solution: Cache the "not found" result with short TTL
    redis.set("user:999999", "NOT_FOUND", "EX", 60)

  Solution: Bloom filter (probabilistic "definitely not in DB")


Problem 3: HOT KEY
──────────────────
  One key gets millions of reads (viral post, flash sale)
  Single Redis node becomes bottleneck.

  Solution: Local cache + Redis (two-tier)
    - Keep hot keys in process memory (LRU, 10s TTL)
    - Falls back to Redis on local cache miss
    - 95% of reads never hit Redis

  Solution: Key replication
    - Spread across "product:123:shard1", "product:123:shard2"
    - Client randomly picks a shard to read from
```

## Explanation

**Cache-aside** is the default choice for most applications — it's simple, well-understood, and only caches data that's actually read. The main decisions are:

1. **TTL selection**: Short TTL (30s-5min) for frequently changing data, longer (15min-1hr) for stable data. When in doubt, start short and increase.
2. **Invalidation**: Always invalidate (delete) on write rather than updating the cache — this avoids race conditions where two concurrent writes leave stale data in cache.
3. **Serialization**: JSON is simple but slow for large objects. Consider MessagePack or Protocol Buffers for high-throughput caches.

**The golden rule**: Your application must work correctly *without* the cache. The cache is a performance optimization, not a data store. If Redis goes down, your app should be slower but still functional (except for Redis-native features like rate limiting).

## ELI5

Caching is like keeping sticky notes on your desk. Instead of walking to the filing cabinet (database) every time someone asks a question, you write the answer on a sticky note (cache). The tricky parts are: knowing when to throw away old sticky notes (invalidation), what happens when 100 people ask the same question at once and your note just fell off (stampede), and making sure your sticky notes match what's actually in the filing cabinet (consistency).
