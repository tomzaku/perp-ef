**What are the OWASP Top 10 security risks, and how do you prevent them in backend applications?**

Security is not optional. Understanding the most common attack vectors helps you build applications that protect user data and resist exploitation.

## Solution

```text
OWASP TOP 10 (2021) — BACKEND FOCUS
════════════════════════════════════

1. BROKEN ACCESS CONTROL (#1 most common)
   User accesses data/actions they shouldn't be able to.

   Attack:
     GET /api/users/456/orders   ← User 123 accessing user 456's orders
     DELETE /api/admin/users/789  ← Regular user calling admin endpoint

   Prevention:
     ✓ Check authorization on EVERY endpoint (not just frontend)
     ✓ Deny by default — explicitly allow, don't explicitly deny
     ✓ Validate resource ownership (does this order belong to this user?)
```

```typescript
// BAD — only checks authentication, not authorization:
app.get('/api/orders/:id', authenticate, async (req, res) => {
  const order = await db.order.findUnique({ where: { id: req.params.id } });
  res.json(order); // Any authenticated user can see ANY order!
});

// GOOD — checks ownership:
app.get('/api/orders/:id', authenticate, async (req, res) => {
  const order = await db.order.findUnique({
    where: { id: req.params.id, userId: req.user.id } // ← Ownership check
  });
  if (!order) return res.status(404).json({ error: 'Not found' });
  res.json(order);
});
```

```text
2. INJECTION (#3)
   Untrusted data sent as part of a command/query.

   SQL Injection:
     Input: ' OR 1=1; DROP TABLE users; --
     Query: SELECT * FROM users WHERE email = '' OR 1=1; DROP TABLE users;--'
```

```typescript
// BAD — string concatenation:
const result = await db.query(
  `SELECT * FROM users WHERE email = '${req.body.email}'`
);

// GOOD — parameterized query:
const result = await db.query(
  'SELECT * FROM users WHERE email = $1',
  [req.body.email]
);

// GOOD — ORM (automatically parameterizes):
const user = await prisma.user.findUnique({
  where: { email: req.body.email }
});
```

```text
3. CRYPTOGRAPHIC FAILURES (#2)
   Sensitive data exposed due to weak/missing encryption.

   Prevention:
     ✓ HTTPS everywhere (TLS 1.2+)
     ✓ Hash passwords with bcrypt/argon2 (NEVER MD5/SHA1)
     ✓ Encrypt sensitive data at rest (AES-256)
     ✓ Don't store secrets in code/git (use env vars, vault)
     ✓ Don't log sensitive data (tokens, passwords, PII)

4. INSECURE DESIGN (#4)
   Missing security controls in the design phase.

   Prevention:
     ✓ Threat modeling before building
     ✓ Rate limiting on sensitive endpoints
     ✓ Account lockout after failed login attempts
     ✓ CAPTCHA for public-facing forms

5. SECURITY MISCONFIGURATION (#5)
   Default credentials, unnecessary features enabled, verbose errors.

   Prevention:
     ✓ Remove default accounts and passwords
     ✓ Disable stack traces in production
     ✓ Set security headers (Helmet.js)
     ✓ Remove unnecessary HTTP methods
     ✓ Keep dependencies updated
```

```typescript
// Security headers with Helmet.js
import helmet from 'helmet';

app.use(helmet());
// Sets: X-Content-Type-Options, X-Frame-Options,
//       Strict-Transport-Security, Content-Security-Policy, etc.

// Don't expose stack traces in production
app.use((err, req, res, next) => {
  res.status(err.status || 500).json({
    error: process.env.NODE_ENV === 'production'
      ? 'Internal server error'     // Safe for production
      : err.message                 // Detailed for development
  });
});
```

```text
6. VULNERABLE COMPONENTS (#6)
   Using libraries with known vulnerabilities.

   Prevention:
     ✓ npm audit / yarn audit regularly
     ✓ Dependabot / Renovate for auto-updates
     ✓ Pin dependency versions
     ✓ Monitor CVE databases

7. AUTHENTICATION FAILURES (#7)
   Weak passwords, credential stuffing, session issues.

   Prevention:
     ✓ Strong password requirements
     ✓ Multi-factor authentication (MFA)
     ✓ Rate limit login attempts
     ✓ Secure session management (HttpOnly, Secure, SameSite cookies)
     ✓ Use established auth libraries (don't roll your own)

8. DATA INTEGRITY FAILURES (#8)
   Trusting untrusted data (deserialization, unsigned updates).

   Prevention:
     ✓ Verify software integrity (checksums, signatures)
     ✓ Use signed JWTs (not just encoded)
     ✓ Validate CI/CD pipeline integrity

9. LOGGING & MONITORING FAILURES (#9)
   Not detecting attacks due to missing logs.

   Prevention:
     ✓ Log authentication events (success and failure)
     ✓ Log access control failures
     ✓ Alert on suspicious patterns
     ✓ Don't log sensitive data (tokens, passwords)

10. SERVER-SIDE REQUEST FORGERY (SSRF) (#10)
    Server makes requests to unintended destinations.

    Attack:
      POST /api/fetch-url
      { "url": "http://169.254.169.254/latest/meta-data/" }
      ← Accesses AWS metadata from inside the network!

    Prevention:
      ✓ Validate and sanitize user-provided URLs
      ✓ Block requests to internal IPs (10.x, 169.254.x, localhost)
      ✓ Use allowlists for permitted domains
      ✓ Don't allow redirects to internal resources


SECURITY CHECKLIST FOR EVERY BACKEND
════════════════════════════════════

□ Parameterized queries (no string concatenation in SQL)
□ Input validation on all endpoints (Zod, Joi)
□ Authentication on all protected routes
□ Authorization checks (resource ownership)
□ Rate limiting on auth endpoints
□ Security headers (Helmet.js)
□ HTTPS only (HSTS header)
□ Password hashing (bcrypt/argon2)
□ No secrets in code/git
□ Dependency auditing (npm audit)
□ Error messages don't leak internals
□ Structured security logging
```

## Explanation

Security is about **defense in depth** — multiple layers of protection so that if one fails, others still protect you.

**The most impactful security practices for backend developers:**
1. **Parameterized queries** — Prevents 100% of SQL injection (the oldest and still common attack)
2. **Authorization on every endpoint** — The #1 vulnerability is broken access control
3. **Input validation** — Never trust client data. Validate type, length, format, range
4. **Keep dependencies updated** — Most breaches exploit known vulnerabilities in outdated packages

**Mental model:** Treat every request as potentially malicious. Validate at the boundary, authorize at every step, and log everything.

## ELI5

Web security is like protecting a house. Broken access control is like having locks but leaving windows open. Injection is like someone slipping a fake instruction to your mail carrier. SSRF is like tricking someone inside the house into unlocking doors for a stranger. Good security means checking every door, window, and delivery — not just the front door.
