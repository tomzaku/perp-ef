**What is the difference between stack and heap memory? How does garbage collection work?**

Memory management is fundamental to understanding performance, debugging crashes, and writing efficient code in any language.

## Solution

```text
STACK vs HEAP
═════════════

┌──────────────────┬──────────────────┬──────────────────┐
│ Aspect           │ Stack            │ Heap             │
├──────────────────┼──────────────────┼──────────────────┤
│ Allocation       │ Automatic (LIFO) │ Dynamic (manual  │
│                  │                  │ or GC)           │
│ Speed            │ Very fast (just  │ Slower (find     │
│                  │ move pointer)    │ free block)      │
│ Size             │ Small (1-8 MB)   │ Large (GBs)     │
│ Lifetime         │ Function scope   │ Until freed/GC'd │
│ Management       │ Compiler         │ Programmer or GC │
│ Fragmentation    │ None             │ Yes              │
│ Growth direction │ Downward ↓       │ Upward ↑         │
│ Thread safety    │ Each thread has  │ Shared across    │
│                  │ own stack        │ threads (needs   │
│                  │                  │ synchronization) │
└──────────────────┴──────────────────┴──────────────────┘


WHAT GOES WHERE?
════════════════

Stack:
  - Local variables (int x = 5)
  - Function parameters
  - Return addresses
  - Fixed-size data known at compile time

Heap:
  - Objects created with new/malloc
  - Data with dynamic size (resizable arrays)
  - Data that outlives the function that created it
  - Closures/captured variables


FUNCTION CALL & STACK FRAMES
═════════════════════════════

  function main() {
    int a = 1;
    foo(a);         // push foo's frame
  }
  function foo(int x) {
    int b = 2;
    bar(b);         // push bar's frame
  }
  function bar(int y) {
    int c = 3;      // top of stack
  }

  Stack (grows downward):
  ┌──────────────┐ ← High address
  │ main()       │
  │  a = 1       │
  │  return addr │
  ├──────────────┤
  │ foo(x=1)     │
  │  b = 2       │
  │  return addr │
  ├──────────────┤
  │ bar(y=2)     │ ← Stack Pointer (SP)
  │  c = 3       │
  └──────────────┘ ← Low address

  When bar() returns → SP moves up, frame is "freed" instantly


STACK OVERFLOW
══════════════

Cause: stack grows beyond its limit (typically 1-8 MB)

  function recursive(n) {
    return recursive(n + 1)  // infinite recursion
  }
  // Each call adds a frame → eventually runs out of stack space
  // → Stack Overflow error / Segmentation fault

  Also caused by: very large local arrays
  int arr[10000000];  // 40MB on stack → overflow


PASS BY VALUE vs PASS BY REFERENCE
═══════════════════════════════════

  Pass by value: copy the value onto the stack
    void foo(int x) { x = 5; }  // original unchanged
    int a = 3; foo(a); // a is still 3

  Pass by reference: copy the reference/pointer onto stack
    void foo(int* x) { *x = 5; }  // original changed
    int a = 3; foo(&a); // a is now 5

  Java/JS/Python: objects passed by "reference value"
    → The reference is copied (value), but it points to same object
    → Reassigning param doesn't affect caller
    → Mutating the object DOES affect caller


GARBAGE COLLECTION
══════════════════

Problem: heap memory must be freed, but manual free is error-prone
  - Forget to free → memory leak
  - Free too early → use-after-free (crash/security bug)
  - Free twice → double-free (crash)

Solution: automatic garbage collection

1. REFERENCE COUNTING
   Each object tracks how many references point to it
   When count reaches 0 → free

   obj = new Object()  // refcount = 1
   ref = obj           // refcount = 2
   ref = null          // refcount = 1
   obj = null          // refcount = 0 → FREE

   Problem: circular references!
   a.next = b; b.prev = a  // both refcount=1 after removing
                            // external refs → never freed!

   Used by: Python (+ cycle detector), Swift (ARC), Rust (Rc)

2. MARK AND SWEEP
   Phase 1 (Mark): Start from roots (stack, globals)
     → Traverse all reachable objects → mark as "alive"
   Phase 2 (Sweep): Free all unmarked objects

   Roots: [stack vars, globals]
      ↓
   [A] → [B] → [C]    ← all marked (reachable)
   [D] → [E]           ← unmarked (unreachable) → FREE

   Handles circular references!
   Problem: stop-the-world pauses

3. GENERATIONAL GC (modern approach)
   Observation: most objects die young

   Young Generation (nursery):
     - New objects allocated here
     - Collected frequently (minor GC)
     - Survivors promoted to old generation

   Old Generation:
     - Long-lived objects
     - Collected rarely (major GC, more expensive)

   ┌─────────────────┬────────────────┐
   │ Young Gen       │ Old Gen        │
   │ (collect often) │ (collect rare) │
   │ [new] [new]     │ [old] [old]    │
   │ Most die here   │ Stable objects │
   └─────────────────┴────────────────┘

   Used by: Java (G1GC, ZGC), .NET, Go, V8 (JavaScript)


MEMORY LEAKS (even with GC!)
═════════════════════════════

GC prevents most leaks but not all:

  - Event listeners not removed
  - Closures capturing large objects
  - Growing caches without eviction
  - Global variables accumulating data
  - Detached DOM nodes (in browsers)

// JavaScript memory leak example:
const cache = {};
function process(key, data) {
  cache[key] = data;  // cache grows forever!
}
// Fix: use WeakMap, or add eviction (LRU)
```

## Explanation

The stack and heap serve complementary roles. The **stack** is blazingly fast because allocation is just moving a pointer — push when entering a function, pop when returning. But it's limited in size and lifetime: data dies when the function returns. The **heap** is flexible — objects can live as long as needed and grow to any size — but allocation requires finding free space and deallocation requires either manual management or garbage collection.

**Garbage collection** is the runtime's way of handling heap cleanup automatically. Modern GCs like Java's G1 or V8's Orinoco use **generational** and **concurrent** strategies to minimize pauses. The "generational hypothesis" (most objects die young) means the GC can focus its effort on the young generation, which is small and fast to collect.

Even with GC, **memory leaks are possible** when objects remain reachable but unused. The GC can only free objects with no references — if your code holds references to objects it no longer needs (event listeners, caches, closures), those objects live forever. Profiling tools (Chrome DevTools, Java VisualVM) help find these leaks by showing heap snapshots and allocation patterns.

## ELI5

The **stack** is like a stack of plates. You put a plate on top when a function starts and take it off when it ends. It's fast and automatic, but you can't reach plates in the middle. The **heap** is like a messy desk — you can put things anywhere and they stay until you clean up. If you never clean up, the desk overflows (memory leak). **Garbage collection** is like having a cleaning robot that periodically checks what you're still using and throws away the rest.
