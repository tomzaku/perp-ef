Modern JavaScript apps consist of hundreds of files. A **bundler** takes all your modules, resolves their dependencies, transforms them (TypeScript → JS, JSX → JS), and produces optimized output files for the browser.

**Why bundlers exist:**
- Browsers historically couldn't load thousands of separate files efficiently
- Modules need transformation (TypeScript, JSX, CSS Modules)
- Production needs optimization: minification, tree-shaking, code splitting

**Key concepts:**
- **Entry point:** Where the bundler starts building the dependency graph
- **Tree shaking:** Removing unused exports from the final bundle
- **Code splitting:** Breaking the bundle into smaller chunks loaded on demand
- **HMR (Hot Module Replacement):** Update modules in dev without full page reload
- **Source maps:** Map minified code back to original source for debugging

Compare Webpack, Vite, Rollup, and esbuild. Explain tree shaking, code splitting, and HMR.

## Explanation

BUNDLER COMPARISON:

```
                Webpack         Vite            Rollup          esbuild
Language:       JS              TS              JS              Go
Dev server:     Bundles all     Native ESM      N/A (build only) Fast bundle
HMR speed:      Slower          Instant         N/A             Very fast
Build speed:    Moderate        Fast (esbuild)  Moderate        Fastest
Tree shaking:   ✓               ✓ (Rollup)      ✓ (best)        ✓
Code splitting: ✓               ✓               ✓               ✓
Config:         Complex         Simple          Minimal         Minimal
Ecosystem:      Huge (loaders)  Growing         Moderate        Limited
Best for:       Large apps      Modern SPAs     Libraries       CI/scripts
```

HOW VITE WORKS IN DEV VS PRODUCTION:

```
Development (native ESM):
  Browser requests index.html
  → requests /src/main.tsx
  → Vite transforms ONLY that file (TypeScript → JS)
  → browser requests its imports as separate files
  → each file transformed on demand

  First load: slightly slower (resolves module graph)
  HMR: only changed file re-transforms → near-instant update

Production (Rollup):
  vite build → Rollup bundles everything
  → tree-shaking + code splitting + minification
  → hashed filenames (main.a1b2c3.js) for cache busting
```

TREE SHAKING REQUIREMENTS:

```
✓ Works:
  import { add } from './math.js'        // ES modules — static
  export function add() {}               // named export

✗ Doesn't work:
  const { add } = require('./math.js')   // CommonJS — dynamic
  export default { add, subtract }       // default object export (whole object included)
  if (x) { import('./math.js') }        // dynamic import (runtime)
```

CODE SPLITTING STRATEGIES:

```
Route-based (most impactful):
  Home page:       home.chunk.js      loaded immediately
  Dashboard:       dashboard.chunk.js loaded when /dashboard visited
  Admin:           admin.chunk.js     loaded when /admin visited

Vendor splitting (for long-term caching):
  vendor.chunk.js    — react, react-dom, etc. (rarely changes)
  app.chunk.js       — your code (changes often)

  Browser caches vendor chunk across deploys (different hashes)
  Only app chunk re-downloads on new deploy ✓
```

## ELI5

Imagine you're writing a book with 500 chapters, each in a separate document. Before sending it to a printer, you need to:
1. Combine all chapters into one file
2. Remove chapters nobody reads (tree shaking)
3. Split it into volumes so readers don't carry all 500 pages (code splitting)
4. Compress repetitive words (minification)

**That's what a bundler does with your JavaScript files.**

```
Your project:
  src/
    main.tsx         → imports App
    App.tsx          → imports Header, Router, Dashboard
    Dashboard.tsx    → imports Chart, Table, utils
    utils.ts         → exports format, parse, validate
    ... (500 more files)

Bundler output:
  dist/
    index.html
    main.a1b2c3.js          (your app code, minified)
    vendor.d4e5f6.js        (react, react-dom — cached separately)
    dashboard.g7h8i9.js     (only loaded when user visits /dashboard)
```

**Webpack vs Vite:**

```
Webpack (the old workhorse):
  Dev: "Let me bundle ALL your code first, then start the server"
  → 30 seconds to start, 2-3 seconds per HMR update
  → Powerful but slow for large apps

Vite (the new speedster):
  Dev: "I'll just start immediately. Each file transforms on demand."
  → 0.5 seconds to start, <100ms HMR updates
  → Uses the browser's native ES module system

  How? Modern browsers can import ES modules natively:
    <script type="module" src="/src/main.tsx">
  Vite transforms files only when the browser asks for them.
```

**Tree shaking** is like cleaning out your closet: you only keep what you actually wear.

```
You import: import { format } from 'date-fns'
date-fns has 200 functions.
Tree shaker checks: "Do you use parse? No. Remove it."
                    "Do you use addDays? No. Remove it."
                    "Do you use format? YES. Keep it."
Final bundle: only format() included → 50KB instead of 300KB
```

**HMR (Hot Module Replacement)** is like changing one Lego brick in a model without breaking the whole thing down.

```
Without HMR:
  Edit Button.tsx → wait for full rebuild → browser refreshes
  → all state lost (form values, scroll position, open modals)

With HMR:
  Edit Button.tsx → only Button.tsx re-transforms → injected into live app
  → state preserved, change visible instantly
  → like swapping a Lego brick while the model stays intact
```
