Compare pagination strategies: **offset-based**, **cursor-based**, and **keyset**. Each has different trade-offs for performance, consistency, and implementation complexity.

**Offset-based:** `?page=3&limit=20` — Simple but slow for deep pages (OFFSET skips rows).

**Cursor-based:** `?cursor=abc123&limit=20` — Uses an opaque cursor (encoded last-seen ID). Fast, consistent even with new inserts.

**Keyset:** `?after_id=150&limit=20` — Like cursor but uses actual column values. Very fast with proper indexes.

Implement cursor-based pagination with infinite scroll.

## Solution

```js
// ════════════════════════════════════════════
// OFFSET-BASED PAGINATION
// ════════════════════════════════════════════

// GET /api/posts?page=3&limit=20
// SQL: SELECT * FROM posts ORDER BY id LIMIT 20 OFFSET 40

async function fetchOffset(page, limit) {
  const res = await fetch('/api/posts?page=' + page + '&limit=' + limit);
  const { data, total } = await res.json();
  return {
    data,
    total,
    totalPages: Math.ceil(total / limit),
    currentPage: page,
    hasNext: page * limit < total,
  };
}

// Pros: Simple, supports "jump to page 50"
// Cons: Slow for deep pages (OFFSET 1000000), inconsistent with inserts

// ════════════════════════════════════════════
// CURSOR-BASED PAGINATION (recommended)
// ════════════════════════════════════════════

// GET /api/posts?cursor=eyJpZCI6MTUwfQ&limit=20
// Cursor is base64({ id: 150 }) — opaque to client

function encodeCursor(data) {
  return btoa(JSON.stringify(data));
}

function decodeCursor(cursor) {
  return JSON.parse(atob(cursor));
}

// Server-side handler
function handleCursorPagination(cursor, limit) {
  const decoded = cursor ? decodeCursor(cursor) : null;

  // SQL: SELECT * FROM posts WHERE id > :lastId ORDER BY id LIMIT :limit + 1
  const query = decoded
    ? 'SELECT * FROM posts WHERE id > ' + decoded.id + ' ORDER BY id LIMIT ' + (limit + 1)
    : 'SELECT * FROM posts ORDER BY id LIMIT ' + (limit + 1);

  // Fetch limit + 1 to check if there's a next page
  const results = []; // ... execute query
  const hasMore = results.length > limit;
  const data = hasMore ? results.slice(0, limit) : results;
  const nextCursor = hasMore
    ? encodeCursor({ id: data[data.length - 1].id })
    : null;

  return { data, nextCursor, hasMore };
}

// ════════════════════════════════════════════
// CLIENT: Infinite Scroll with Cursor Pagination
// ════════════════════════════════════════════

class InfiniteScroll {
  #cursor = null;
  #hasMore = true;
  #loading = false;
  #items = [];
  #limit = 20;
  #onUpdate = null;

  constructor(fetchFn, onUpdate) {
    this.#fetchFn = fetchFn;
    this.#onUpdate = onUpdate;
    this.#setupScrollListener();
  }

  #fetchFn;

  async loadMore() {
    if (this.#loading || !this.#hasMore) return;
    this.#loading = true;

    try {
      const { data, nextCursor, hasMore } = await this.#fetchFn(
        this.#cursor,
        this.#limit
      );
      this.#items = [...this.#items, ...data];
      this.#cursor = nextCursor;
      this.#hasMore = hasMore;
      this.#onUpdate?.(this.getState());
    } finally {
      this.#loading = false;
    }
  }

  #setupScrollListener() {
    // Use IntersectionObserver for better performance
    // than scroll event listener
    if (typeof IntersectionObserver === 'undefined') return;

    this.observer = new IntersectionObserver(
      (entries) => {
        if (entries[0].isIntersecting) {
          this.loadMore();
        }
      },
      { rootMargin: '200px' } // Load before reaching bottom
    );
  }

  // Call this with the sentinel element at the bottom of the list
  observe(element) {
    this.observer?.observe(element);
  }

  getState() {
    return {
      items: this.#items,
      loading: this.#loading,
      hasMore: this.#hasMore,
    };
  }
}

// Usage
async function fetchPosts(cursor, limit) {
  const params = new URLSearchParams({ limit });
  if (cursor) params.set('cursor', cursor);
  const res = await fetch('/api/posts?' + params);
  return res.json();
}

const scroll = new InfiniteScroll(fetchPosts, (state) => {
  console.log('Items:', state.items.length, 'Has more:', state.hasMore);
});

// Initial load
scroll.loadMore();
```

## ELI5

Imagine reading a very long book. You don't read all 500 pages at once — you read one chapter, put a bookmark in, and come back later.

**Offset-based pagination** is like using page numbers:

```
"Give me page 3 of 20 items per page"
= "Skip the first 40 items, give me items 41-60"

Problem with page numbers:
  While you're on page 3, someone adds 5 new posts at the top.
  Now what was on page 3 is different from when you started!
  Also: "skip 10,000 rows" is slow in a database.
```

**Cursor-based pagination** is a smarter bookmark — it remembers exactly where you stopped:

```
"Give me the next 20 items after item ID 150"
= Your bookmark says "I was at item #150"

  First load:  no cursor → get items 1-20, next cursor = "item #20"
  Next page:   cursor="item #20" → get items 21-40, cursor = "item #40"
  Next page:   cursor="item #40" → get items 41-60, cursor = "item #60"

  New posts added? Doesn't matter — you're always "after item #60"
  Database query: WHERE id > 60 ORDER BY id LIMIT 20 → very fast with index!
```

**Infinite scroll** vs **next page button** — same pagination logic, different UX:

```
Infinite scroll:
  Page loads → shows first 20 posts
  User scrolls down → "sentinel" element enters view
  → IntersectionObserver fires → loadMore() called automatically
  → Next 20 posts appended to the list

  Sentinel element: invisible div at the bottom of the list
  IntersectionObserver: fires when that div becomes visible
  → triggers fetch before user hits the very bottom (200px early)
```
