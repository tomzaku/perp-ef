**What are vector databases, time-series databases, and search engines? When would you use each instead of a traditional SQL or NoSQL database?**

As AI, IoT, and real-time analytics become mainstream, specialized databases are increasingly common in interviews — especially at companies building AI features, monitoring systems, or recommendation engines.

```text
1. VECTOR DATABASES
═══════════════════════════════════════════════════════

WHAT: Store and search high-dimensional vectors
      (embeddings from AI/ML models).

HOW IT WORKS:
  Text/Image → AI Model → Embedding Vector → Vector DB

  "How do I reset my password?"
       ↓ (OpenAI embedding model)
  [0.023, -0.456, 0.789, ..., 0.012]  (1536 dimensions)

  Similarity search finds the closest vectors using:
  - Cosine similarity (angle between vectors)
  - Euclidean distance (straight-line distance)
  - Dot product (magnitude + direction)

  Unlike SQL WHERE exact match, vector search finds
  "semantically similar" results.

USE CASES:
  - AI chatbot retrieval (RAG — find relevant docs)
  - Semantic search ("comfortable shoes" → finds "cozy sneakers")
  - Recommendation engines (similar products/content)
  - Image similarity (reverse image search)
  - Anomaly detection (find outlier patterns)

POPULAR OPTIONS:
  ┌─────────────┬────────────────────────────────────┐
  │ Pinecone    │ Managed, serverless, purpose-built  │
  │ Weaviate    │ Open-source, multi-modal            │
  │ Qdrant      │ Open-source, Rust-based, fast       │
  │ Milvus      │ Open-source, massive scale           │
  │ pgvector    │ PostgreSQL extension (good start)   │
  │ ChromaDB    │ Lightweight, good for prototyping    │
  └─────────────┴────────────────────────────────────┘
```

```python
# ─── pgvector (PostgreSQL Extension) ─────────────
# Great starting point — no new infrastructure needed

import psycopg2
import openai

# Setup: Enable pgvector in PostgreSQL
# CREATE EXTENSION vector;
# CREATE TABLE documents (
#   id SERIAL PRIMARY KEY,
#   content TEXT,
#   embedding vector(1536)  -- OpenAI ada-002 dimension
# );
# CREATE INDEX ON documents
#   USING ivfflat (embedding vector_cosine_ops)
#   WITH (lists = 100);

# Step 1: Generate embedding
def get_embedding(text):
    response = openai.embeddings.create(
        model="text-embedding-ada-002",
        input=text
    )
    return response.data[0].embedding

# Step 2: Store document with embedding
def store_document(content):
    embedding = get_embedding(content)
    cur.execute(
        "INSERT INTO documents (content, embedding) VALUES (%s, %s)",
        (content, embedding)
    )

# Step 3: Similarity search (find 5 most relevant docs)
def search(query, limit=5):
    query_embedding = get_embedding(query)
    cur.execute("""
        SELECT content, 1 - (embedding <=> %s::vector) AS similarity
        FROM documents
        ORDER BY embedding <=> %s::vector  -- cosine distance
        LIMIT %s
    """, (query_embedding, query_embedding, limit))
    return cur.fetchall()

# Example: RAG (Retrieval-Augmented Generation)
docs = search("How do I reset my password?")
# Returns semantically similar docs like:
#   "Password recovery steps: Go to Settings > Security..."
#   "Account access troubleshooting guide..."
# Even though they don't contain the exact words!
```

```python
from pinecone import Pinecone

pc = Pinecone(api_key="your-key")
index = pc.Index("products")

# Upsert vectors with metadata
index.upsert(vectors=[
    {
        "id": "prod-1",
        "values": get_embedding("Comfortable running shoes"),
        "metadata": {
            "category": "footwear",
            "price": 129.99,
            "brand": "Nike"
        }
    }
])

# Semantic search with metadata filtering
results = index.query(
    vector=get_embedding("cozy sneakers for jogging"),
    top_k=10,
    filter={
        "category": "footwear",
        "price": {"$lt": 150}
    },
    include_metadata=True
)
# Finds "Comfortable running shoes" even though
# query used different words!
```

```text
2. TIME-SERIES DATABASES
═══════════════════════════════════════════════════════

WHAT: Optimized for timestamped data points —
      metrics, events, sensor readings.

WHY NOT POSTGRES?
  - Ingestion: TSDB handles millions of writes/sec
  - Compression: 10-20x better for time-series data
  - Queries: Built-in downsampling, rollups, retention
  - Postgres struggles at >100M time-series rows

CHARACTERISTICS:
  - Write-heavy (append-mostly, rarely update)
  - Time-ordered queries (last hour, last 7 days)
  - Aggregation-heavy (avg, p95, rate, rollups)
  - Auto-expiry (delete data older than 30 days)

USE CASES:
  - Application metrics & monitoring (CPU, memory, latency)
  - IoT sensor data (temperature, humidity, pressure)
  - Financial tick data (stock prices every millisecond)
  - Real-time analytics dashboards
  - DevOps observability (Grafana + InfluxDB/Prometheus)

POPULAR OPTIONS:
  ┌──────────────┬──────────────────────────────────────┐
  │ InfluxDB     │ Purpose-built TSDB, InfluxQL + Flux   │
  │ TimescaleDB  │ PostgreSQL extension (familiar SQL!)  │
  │ Prometheus   │ Pull-based metrics, Kubernetes native │
  │ ClickHouse   │ Columnar OLAP, blazing-fast analytics │
  │ QuestDB      │ SQL-native TSDB, fast ingestion       │
  │ Amazon       │ Timestream — managed, serverless      │
  │ Timestream   │                                       │
  └──────────────┴──────────────────────────────────────┘


DATA MODEL:

  Measurement: cpu_usage
  Tags (indexed): host=server-1, region=us-east  ← metadata
  Fields (values): usage_percent=73.5            ← what you measure
  Timestamp: 2024-06-15T10:30:00Z               ← when

  ┌───────────────────┬──────────┬─────────┬──────────────┐
  │ time              │ host     │ region  │ usage_percent │
  ├───────────────────┼──────────┼─────────┼──────────────┤
  │ 2024-06-15 10:30  │ server-1 │ us-east │ 73.5         │
  │ 2024-06-15 10:30  │ server-2 │ us-west │ 45.2         │
  │ 2024-06-15 10:31  │ server-1 │ us-east │ 78.1         │
  │ 2024-06-15 10:31  │ server-2 │ us-west │ 42.8         │
  └───────────────────┴──────────┴─────────┴──────────────┘
```

```sql
-- ─── TimescaleDB (PostgreSQL Extension) ──────────
-- Use familiar SQL! No new query language to learn.

-- Create a regular table, then convert to hypertable
CREATE TABLE metrics (
  time        TIMESTAMPTZ NOT NULL,
  host        TEXT NOT NULL,
  cpu_percent DOUBLE PRECISION,
  mem_percent DOUBLE PRECISION,
  disk_io     BIGINT
);

-- Convert to hypertable (auto-partitions by time)
SELECT create_hypertable('metrics', 'time');

-- Insert data (same as regular INSERT)
INSERT INTO metrics VALUES
  (NOW(), 'web-01', 73.5, 62.1, 1024),
  (NOW(), 'web-02', 45.2, 55.8, 512);

-- Query: Average CPU per host over last hour (5-min buckets)
SELECT
  time_bucket('5 minutes', time) AS bucket,
  host,
  AVG(cpu_percent) AS avg_cpu,
  MAX(cpu_percent) AS max_cpu,
  percentile_cont(0.95) WITHIN GROUP (ORDER BY cpu_percent) AS p95_cpu
FROM metrics
WHERE time > NOW() - INTERVAL '1 hour'
GROUP BY bucket, host
ORDER BY bucket DESC;

-- Query: Detect anomalies (CPU > 90% for 5+ minutes)
SELECT host, MIN(time) AS started, MAX(time) AS ended,
       AVG(cpu_percent) AS avg_cpu
FROM metrics
WHERE time > NOW() - INTERVAL '1 hour'
  AND cpu_percent > 90
GROUP BY host
HAVING MAX(time) - MIN(time) > INTERVAL '5 minutes';

-- Auto-retention: Delete data older than 30 days
SELECT add_retention_policy('metrics', INTERVAL '30 days');

-- Continuous aggregate (materialized rollups, auto-refreshed)
CREATE MATERIALIZED VIEW metrics_hourly
WITH (timescaledb.continuous) AS
SELECT
  time_bucket('1 hour', time) AS hour,
  host,
  AVG(cpu_percent) AS avg_cpu,
  MAX(cpu_percent) AS max_cpu
FROM metrics
GROUP BY hour, host;
```

```text
3. SEARCH ENGINES
═══════════════════════════════════════════════════════

WHAT: Inverted index + relevance scoring for
      full-text search, autocomplete, and facets.

WHY NOT POSTGRES FULL-TEXT SEARCH?
  - Postgres: Good for simple search on <10M rows
  - Elasticsearch: Typo tolerance, stemming, synonyms,
    multi-language, facets, highlights, autocomplete
    at scale with distributed architecture

POPULAR OPTIONS:
  ┌──────────────┬──────────────────────────────────────┐
  │ Elasticsearch│ Most popular, Lucene-based, mature   │
  │ OpenSearch   │ AWS fork of Elasticsearch             │
  │ Meilisearch  │ Lightweight, fast, great DX          │
  │ Typesense    │ Easy to set up, typo-tolerant        │
  │ Algolia      │ Managed SaaS, excellent for frontend │
  └──────────────┴──────────────────────────────────────┘

HOW INVERTED INDEX WORKS:

  Documents:
    doc1: "Redis is a fast cache"
    doc2: "PostgreSQL is a fast database"
    doc3: "Redis cache improves speed"

  Inverted Index:
    "redis"      → [doc1, doc3]
    "fast"       → [doc1, doc2]
    "cache"      → [doc1, doc3]
    "postgresql" → [doc2]
    "database"   → [doc2]
    "speed"      → [doc3]

  Search "fast cache" → intersect [doc1,doc2] ∩ [doc1,doc3] = [doc1]
  With scoring: doc1 matches both terms → highest relevance

4. COLUMNAR / OLAP DATABASES
═══════════════════════════════════════════════════════

WHAT: Store data by column instead of row.
      Blazing fast for analytical queries.

WHY: Row-based (Postgres) reads entire rows.
     For "AVG(price) across 1B rows" it reads every
     column even though you only need one.
     Columnar DB reads just the price column → 10-100x faster.

USE CASES:
  - Business intelligence dashboards
  - Ad-hoc analytics on billions of rows
  - Log analysis and event analytics
  - Real-time reporting

POPULAR OPTIONS:
  ┌──────────────┬──────────────────────────────────────┐
  │ ClickHouse   │ Open-source, extremely fast           │
  │ Apache Druid │ Real-time analytics, high concurrency│
  │ DuckDB       │ Embedded OLAP (SQLite for analytics) │
  │ BigQuery     │ Google managed, serverless            │
  │ Redshift     │ AWS managed columnar warehouse        │
  │ Snowflake    │ Cloud-native data warehouse           │
  └──────────────┴──────────────────────────────────────┘

Row vs Column storage:

  ROW STORE (PostgreSQL):
  Row 1: [id=1, name="Alice", age=30, city="NYC"]
  Row 2: [id=2, name="Bob",   age=25, city="SF"]
  → Good for: SELECT * FROM users WHERE id = 1

  COLUMN STORE (ClickHouse):
  id:   [1, 2, 3, 4, ...]
  name: ["Alice", "Bob", "Charlie", ...]
  age:  [30, 25, 35, ...]
  city: ["NYC", "SF", "NYC", ...]
  → Good for: SELECT AVG(age) FROM users WHERE city = 'NYC'
  → Only reads age and city columns, skips name entirely
  → Compression: same-type data compresses 10x better
```

```sql
-- ClickHouse example: Analytics on 1B events
CREATE TABLE events (
  timestamp DateTime,
  user_id UInt64,
  event_type String,
  page String,
  duration_ms UInt32,
  country String
) ENGINE = MergeTree()
ORDER BY (timestamp, user_id);

-- Query: Page views per country, last 24 hours
-- Scans 1 billion rows in < 1 second
SELECT
  country,
  count() AS page_views,
  avg(duration_ms) AS avg_duration,
  uniq(user_id) AS unique_users
FROM events
WHERE timestamp > now() - INTERVAL 1 DAY
  AND event_type = 'page_view'
GROUP BY country
ORDER BY page_views DESC
LIMIT 20;
```

```text
═══════════════════════════════════════════════════════

"Start With" = lowest operational cost to get started.
pgvector and TimescaleDB are PostgreSQL extensions —
no new infrastructure needed.

WHEN TO ADD A SPECIALIZED DB:
  1. PostgreSQL handles the workload → don't add anything
  2. Workload grows beyond Postgres capabilities → add extension
     (pgvector, TimescaleDB, pg_trgm for search)
  3. Extension hits limits → add dedicated specialized DB
  4. Always keep PostgreSQL as source of truth
```

## Explanation

**The key principle: start with PostgreSQL, specialize when needed.**

PostgreSQL with extensions (pgvector, TimescaleDB, pg_trgm) can handle most workloads up to moderate scale. Add a specialized database only when:
- **Ingestion rate** exceeds what Postgres can handle (time-series at 1M+ points/sec)
- **Query patterns** need specialized indexes (vector similarity, full-text with typo tolerance)
- **Data volume** makes general-purpose storage too expensive (columnar compression for analytics)
- **Latency requirements** demand purpose-built engines (sub-ms cache lookups)

Each specialized database adds **operational complexity** — another system to deploy, monitor, back up, and keep in sync. The cost of adding a database is real, so only add one when the benefit clearly outweighs the cost.

## ELI5

Regular databases are like a filing cabinet — great for organizing papers by name or date. But imagine trying to find all photos that *look like* your cat (vector DB), track your plant's temperature every second for a year (time-series DB), or find every document mentioning "birthday" even if it's spelled "brithday" (search engine). Each of these needs a special tool. You still keep the filing cabinet for important documents, but you add a photo album, a thermometer chart, and a search engine when you need them.
