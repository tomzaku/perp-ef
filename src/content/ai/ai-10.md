**How does function calling work in LLMs, and how do you implement structured output?**

Function calling (tool use) allows LLMs to interact with external systems by generating structured calls to predefined functions. This bridges the gap between natural language and programmatic actions.

## Solution

```typescript
// ════════════════════════════════════
// FUNCTION CALLING FLOW
// ════════════════════════════════════
// 1. Developer defines available tools (name, description, JSON schema)
// 2. User sends a message
// 3. LLM decides whether to call a tool (and which one)
// 4. LLM generates structured arguments matching the schema
// 5. Your code executes the function with those arguments
// 6. Result is sent back to the LLM
// 7. LLM generates natural language response with the result

// ════════════════════════════════════
// EXAMPLE: Tool Definition + Execution
// ════════════════════════════════════

// Define tools
const tools = [
  {
    name: "get_weather",
    description: "Get current weather for a location",
    input_schema: {
      type: "object",
      properties: {
        location: { type: "string", description: "City name" },
        unit: { type: "string", enum: ["celsius", "fahrenheit"] }
      },
      required: ["location"]
    }
  }
];

// Send to LLM (Anthropic example)
const response = await anthropic.messages.create({
  model: "claude-sonnet-4-6",
  tools,
  messages: [{ role: "user", content: "What's the weather in Tokyo?" }]
});

// Handle tool use
if (response.stop_reason === "tool_use") {
  const toolCall = response.content.find(b => b.type === "tool_use");
  // toolCall = { name: "get_weather", input: { location: "Tokyo" } }

  const result = await getWeather(toolCall.input);

  // Send result back to LLM for final response
  const finalResponse = await anthropic.messages.create({
    model: "claude-sonnet-4-6",
    tools,
    messages: [
      { role: "user", content: "What's the weather in Tokyo?" },
      { role: "assistant", content: response.content },
      { role: "user", content: [{
        type: "tool_result",
        tool_use_id: toolCall.id,
        content: JSON.stringify(result)
      }]}
    ]
  });
}

// ════════════════════════════════════
// STRUCTURED OUTPUT (OpenAI JSON Mode)
// ════════════════════════════════════

const structured = await openai.chat.completions.create({
  model: "gpt-4o",
  response_format: {
    type: "json_schema",
    json_schema: {
      name: "product_review",
      schema: {
        type: "object",
        properties: {
          sentiment: { type: "string", enum: ["positive", "negative", "neutral"] },
          score: { type: "number", minimum: 0, maximum: 10 },
          key_points: { type: "array", items: { type: "string" } }
        },
        required: ["sentiment", "score", "key_points"]
      }
    }
  },
  messages: [{ role: "user", content: "Review: This phone is amazing!" }]
});
```

## Explanation

Function calling transforms LLMs from text-in-text-out systems into **orchestration engines**. The model acts as a natural language interface that translates user intent into structured API calls.

**Key design principle:** The model decides *what* to do (choosing tools and parameters) while your code controls *how* it happens (executing with proper auth, validation, and error handling).

**Best practices:**
1. **Clear tool descriptions** — the model selects tools based on descriptions, make them precise
2. **Constrained schemas** — use enums, required fields, and validation
3. **Parallel tool calls** — some models support calling multiple tools at once
4. **Error handling** — return clear error messages so the model can retry
5. **Idempotency** — design tools to be safe to retry (especially with agents)

## ELI5

Imagine you're talking to a very smart receptionist. You say "book me a flight to Paris next Friday." The receptionist doesn't actually book the flight — they fill out a booking form with the right details (destination: Paris, date: Friday) and hand it to the booking system. Function calling is the same: the AI understands what you want and fills out the right "form" for your code to process.
