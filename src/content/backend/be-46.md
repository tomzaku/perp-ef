**What is gRPC, how does it work under the hood, and when should you choose it over REST?**

gRPC is Google's open-source RPC framework. Understanding its architecture and streaming models is essential for microservices and high-throughput backend systems.

## Solution

```protobuf
// ═══════════════════════════════════
// 1. DEFINE THE SERVICE IN .proto
// ═══════════════════════════════════

syntax = "proto3";

package order;

import "google/protobuf/timestamp.proto";

message OrderRequest {
  string user_id  = 1;
  repeated string product_ids = 2;
}

message Order {
  string id         = 1;
  string user_id    = 2;
  double total      = 3;
  string status     = 4;
  google.protobuf.Timestamp created_at = 5;
}

message OrderId {
  string id = 1;
}

message TrackingEvent {
  string order_id  = 1;
  string location  = 2;
  string status    = 3;
  google.protobuf.Timestamp timestamp = 4;
}

service OrderService {
  // Pattern 1: Unary (one request, one response — like REST)
  rpc CreateOrder(OrderRequest) returns (Order);

  // Pattern 2: Server Streaming (one request, stream of responses)
  rpc TrackOrder(OrderId) returns (stream TrackingEvent);

  // Pattern 3: Client Streaming (stream of requests, one response)
  rpc BatchCreateOrders(stream OrderRequest) returns (Order);

  // Pattern 4: Bidirectional Streaming (stream both ways)
  rpc LiveOrderFeed(stream OrderId) returns (stream Order);
}
```

type orderServer struct {
    order.UnimplementedOrderServiceServer
    db *sql.DB
}

// Unary RPC — simple request/response
func (s *orderServer) CreateOrder(ctx context.Context, req *order.OrderRequest) (*order.Order, error) {
    o := &order.Order{
        Id:     uuid.New().String(),
        UserId: req.UserId,
        Status: "pending",
    }
    if err := s.db.SaveOrder(ctx, o); err != nil {
        return nil, status.Errorf(codes.Internal, "failed to save order: %v", err)
    }
    return o, nil
}

// Server streaming — push updates to client
func (s *orderServer) TrackOrder(req *order.OrderId, stream order.OrderService_TrackOrderServer) error {
    ticker := time.NewTicker(5 * time.Second)
    defer ticker.Stop()
    for {
        select {
        case <-stream.Context().Done():
            return nil  // client disconnected
        case <-ticker.C:
            event := s.db.GetLatestTrackingEvent(req.Id)
            if err := stream.Send(event); err != nil {
                return err  // client gone
            }
        }
    }
}

## Explanation

**The four gRPC communication patterns:**

| Pattern | Request | Response | Use case |
|---|---|---|---|
| Unary | Single | Single | Standard CRUD (like REST) |
| Server streaming | Single | Stream | Live feeds, file download, notifications |
| Client streaming | Stream | Single | File upload, batch ingestion |
| Bidirectional | Stream | Stream | Chat, real-time collaboration, gaming |

**gRPC vs REST:**

| | REST | gRPC |
|---|---|---|
| Protocol | HTTP/1.1 (usually) | HTTP/2 |
| Encoding | JSON (text) | Protobuf (binary) |
| Schema | Optional (OpenAPI) | Required (.proto) |
| Code gen | Tools available | First-class |
| Browser support | Native | Needs grpc-web proxy |
| Streaming | SSE / WebSocket | Native (4 patterns) |
| Performance | Baseline | 5–10x faster |
| Human-readable | Yes | No |

**When to choose gRPC:**
- Internal microservice-to-microservice communication
- High-throughput APIs where payload size matters
- Need real-time bidirectional streaming
- Polyglot services (generate clients for Go, Java, Python, etc. from one .proto)

**When to keep REST:**
- Public-facing APIs consumed by browsers or third parties
- Simple CRUD with no streaming needs
- Team unfamiliar with Protobuf toolchain

**Interceptors** are gRPC's middleware — use them for auth, logging, tracing, and rate limiting:
```go
// Unary interceptor (like Express middleware)
func loggingInterceptor(ctx context.Context, req any, info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {
    start := time.Now()
    resp, err := handler(ctx, req)
    log.Printf("%s took %v, err=%v", info.FullMethod, time.Since(start), err)
    return resp, err
}
```

## ELI5

REST is like texting someone — you send a message (HTTP request), wait for a reply (HTTP response), and that's it. gRPC is like a phone call — you establish a connection once and can both talk and listen at the same time (streaming), and the language you're speaking is a compact shorthand (Protobuf) instead of plain English (JSON). Phone calls are faster for long conversations; texting is fine for quick one-off questions.
