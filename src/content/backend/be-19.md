**When should you scale vertically vs horizontally, and what architectural decisions enable each?**

Scaling is how you handle more users, more data, and more traffic as your application grows.

## Solution

```text
VERTICAL SCALING (Scale Up)
═══════════════════════════

Add more resources to ONE server.

  Before: 2 CPU, 4GB RAM, 50GB SSD
  After:  16 CPU, 64GB RAM, 1TB NVMe

  ✓ Simple — no code changes needed
  ✓ No distributed system complexity
  ✓ Strong consistency (one database)
  ✓ Lower operational cost
  ✗ Hardware limits (can't buy a 10,000 CPU server)
  ✗ Single point of failure
  ✗ Downtime during upgrades


HORIZONTAL SCALING (Scale Out)
══════════════════════════════

Add MORE servers.

  Before: [Server 1]
  After:  [Server 1] [Server 2] [Server 3] [Server 4]

  ✓ Theoretically unlimited scaling
  ✓ Fault tolerance (one server down, others serve)
  ✓ Zero-downtime upgrades (rolling deploys)
  ✗ Requires stateless design
  ✗ Distributed system complexity
  ✗ Data consistency challenges


WHAT NEEDS TO BE STATELESS?
═══════════════════════════

For horizontal scaling, application servers must be stateless.
Any server should handle any request.

Sessions:
  ✗ BAD:  In-memory sessions on the server
  ✓ GOOD: Redis session store (shared across servers)
  ✓ GOOD: JWT tokens (stateless by design)

File uploads:
  ✗ BAD:  Save to local disk (only that server has the file)
  ✓ GOOD: S3 / cloud storage (all servers can access)

Caching:
  ✗ BAD:  In-process cache (each server has different cache)
  ✓ GOOD: Redis/Memcached (shared cache)

WebSocket connections:
  ✗ BAD:  Direct WebSocket to app server (sticky to one server)
  ✓ GOOD: Redis Pub/Sub to broadcast across servers


SCALING THE DATABASE (hardest part)
═══════════════════════════════════

Application servers are easy to scale (stateless + load balancer).
Databases are the bottleneck.

1. READ REPLICAS
   ┌──────────────┐     ┌──────────────┐
   │ Primary (RW) │────→│ Replica (R)  │
   │  Writes here │     │  Reads here  │
   └──────────────┘     ├──────────────┤
                        │ Replica (R)  │
                        │  Reads here  │
                        └──────────────┘

   Write to primary, read from replicas.
   Works when reads >> writes (most apps).
   Replication lag: replicas may be slightly behind.

2. CONNECTION POOLING
   Without: Each request opens a new DB connection (expensive!)
   With: Pool of reusable connections shared across requests

   // PgBouncer or application-level pooling
   const pool = new Pool({
     max: 20,              // Max concurrent connections
     idleTimeoutMillis: 30000,
     connectionTimeoutMillis: 5000,
   });

3. CACHING LAYER
   [App] → check Redis cache → if miss → query DB → store in Redis
   Reduces database load by 80-90% for read-heavy workloads.

4. SHARDING (last resort)
   Split data across multiple databases by a shard key.

   Shard by user_id:
   Users 1-1M     → Database A
   Users 1M-2M    → Database B
   Users 2M-3M    → Database C

   ✓ Scales writes (the hard problem)
   ✗ Cross-shard queries are very complex
   ✗ Rebalancing shards is painful
   ✗ Application must know which shard to query


SCALING STRATEGY BY STAGE
═════════════════════════

Stage 1: Single Server (0 → 10K users)
  [App + DB on one server]
  - Optimize queries, add indexes
  - Vertical scaling (bigger server)

Stage 2: Separate DB (10K → 100K users)
  [App Server] → [Database Server]
  - Dedicated DB server
  - Add Redis for sessions + caching

Stage 3: Load Balanced (100K → 1M users)
  [LB] → [App 1] [App 2] [App 3]
                    ↓
          [Redis] [Primary DB] → [Read Replica]
  - Stateless app servers
  - Read replicas for queries

Stage 4: Advanced (1M+ users)
  [CDN] → [LB] → [App cluster]
                       ↓
  [Redis Cluster] [DB Primary + Replicas]
  [Message Queue] [Background Workers]
  [Search (Elasticsearch)]
  - Microservices where needed
  - Sharding if write throughput demands it


COMMON BOTTLENECKS & SOLUTIONS
══════════════════════════════

Slow DB queries → Add indexes, optimize queries, read replicas
Too many DB connections → Connection pooling (PgBouncer)
High read traffic → Caching layer (Redis)
High write traffic → Write batching, sharding
Large files → Object storage (S3) + CDN
CPU-intensive work → Background workers, dedicated compute
Global latency → Multiple regions, CDN, edge computing
```

## Explanation

**The practical approach:**

1. **Start simple** — Single server with PostgreSQL handles more than you think (millions of rows, thousands of requests/second)
2. **Optimize first** — Indexes, query optimization, and caching solve 90% of scaling problems
3. **Scale application horizontally** — Make app servers stateless, put them behind a load balancer
4. **Scale database carefully** — Read replicas first, then caching, sharding as last resort

**The key insight:** Most scaling problems are database problems. A stateless Node.js app behind a load balancer scales trivially. The database is where the real engineering challenge lies.

**Don't prematurely scale.** Many companies waste engineering time building for "1M users" when they have 1,000. Build for your current scale + 10x, not 1000x.

## ELI5

Vertical scaling is like making your car engine bigger — at some point, you can't fit a bigger engine. Horizontal scaling is like adding more cars — you can keep adding cars as long as you have roads (load balancers) to manage traffic. The tricky part isn't adding more cars (servers) — it's making sure they can all use the same GPS map (database) without crashing into each other.
