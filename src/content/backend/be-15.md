**What is observability, and how do you implement logging, monitoring, and tracing?**

Observability is the ability to understand your system's internal state from its external outputs. It's how you debug production issues and ensure reliability.

## Solution

```text
THREE PILLARS OF OBSERVABILITY
══════════════════════════════

1. LOGS    — "What happened?" (discrete events)
2. METRICS — "How is the system performing?" (aggregated numbers)
3. TRACES  — "How did this request flow through the system?" (request path)

                  ┌──────────┐
  Request ──────→ │ Service A│──→ log: "User login attempt"
                  │          │──→ metric: login_count++
                  └────┬─────┘──→ trace: span A (5ms)
                       │
                  ┌────▼─────┐
                  │ Service B│──→ log: "Auth check passed"
                  │          │──→ metric: auth_latency_ms
                  └────┬─────┘──→ trace: span B (12ms)
                       │
                  ┌────▼─────┐
                  │ Database │──→ log: "Query executed"
                  │          │──→ metric: query_duration_ms
                  └──────────┘──→ trace: span C (3ms)

  Total trace: A(5ms) → B(12ms) → C(3ms) = 20ms
```

```typescript
// ═══════════════════════════════════
// STRUCTURED LOGGING
// ═══════════════════════════════════

import pino from 'pino';

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  // Structured JSON output (not plain text!)
  formatters: {
    level: (label) => ({ level: label })
  }
});

// BAD — unstructured, hard to search/filter:
console.log('User 123 placed order 456 for $99.99');

// GOOD — structured, searchable, filterable:
logger.info({
  event: 'order_created',
  userId: 'user-123',
  orderId: 'order-456',
  total: 99.99,
  items: 3,
  requestId: req.id        // Correlation ID
}, 'Order created successfully');

// Output (JSON):
// {"level":"info","event":"order_created","userId":"user-123",
//  "orderId":"order-456","total":99.99,"requestId":"abc-123",
//  "msg":"Order created successfully","time":1706000000}


// ─── Log Levels ───
logger.trace('Entering function');     // Most verbose
logger.debug('Cache miss for key X');  // Development debugging
logger.info('Order created');          // Normal operations
logger.warn('Rate limit approaching'); // Potential issues
logger.error('Payment failed');        // Errors that need attention
logger.fatal('Database connection lost'); // System is broken


// ─── Request Logging Middleware ───
function requestLogger(req, res, next) {
  req.id = req.headers['x-request-id'] || crypto.randomUUID();
  req.log = logger.child({ requestId: req.id }); // Attach to all logs

  const start = Date.now();
  res.on('finish', () => {
    req.log.info({
      method: req.method,
      path: req.path,
      statusCode: res.statusCode,
      duration: Date.now() - start,
      userAgent: req.headers['user-agent']
    }, 'Request completed');
  });
  next();
}


// ═══════════════════════════════════
// METRICS
// ═══════════════════════════════════

// Common metric types:
// Counter — only goes up (total_requests, error_count)
// Gauge — goes up and down (active_connections, queue_depth)
// Histogram — distribution of values (request_latency, response_size)

// Using prom-client for Prometheus metrics:
import { Counter, Histogram, Registry } from 'prom-client';

const register = new Registry();

const httpRequests = new Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'path', 'status'],
  registers: [register]
});

const httpDuration = new Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration',
  labelNames: ['method', 'path'],
  buckets: [0.01, 0.05, 0.1, 0.5, 1, 5],
  registers: [register]
});

// Middleware to record metrics
function metricsMiddleware(req, res, next) {
  const end = httpDuration.startTimer({ method: req.method, path: req.route?.path });
  res.on('finish', () => {
    httpRequests.inc({ method: req.method, path: req.route?.path, status: res.statusCode });
    end();
  });
  next();
}

// Expose metrics endpoint for Prometheus scraping
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});


// ═══════════════════════════════════
// KEY METRICS TO TRACK (RED / USE)
// ═══════════════════════════════════
//
// RED Method (for services):
//   Rate    — Requests per second
//   Errors  — Error rate (% of requests that fail)
//   Duration — Latency percentiles (p50, p95, p99)
//
// USE Method (for infrastructure):
//   Utilization — CPU %, memory %, disk %
//   Saturation  — Queue depth, thread pool usage
//   Errors      — Hardware errors, OOM kills
//
// Four Golden Signals (Google SRE):
//   Latency, Traffic, Errors, Saturation
```

```text
ALERTING RULES
═════════════

Good alerts are:
  - Actionable (someone needs to do something)
  - Urgent (needs attention now, not tomorrow)
  - Not noisy (too many alerts = ignored alerts)

Example alert rules:
  - Error rate > 5% for 5 minutes → Page on-call
  - p99 latency > 2s for 10 minutes → Page on-call
  - Database connection pool > 80% → Warn
  - Disk usage > 90% → Warn
  - Service down for > 30 seconds → Page on-call


DISTRIBUTED TRACING (OpenTelemetry)
═══════════════════════════════════

A trace follows a single request across multiple services:

Trace ID: abc-123
├── Span: API Gateway (2ms)
│   └── Span: Auth Service (8ms)
│       └── Span: Redis Cache Lookup (1ms)
├── Span: Order Service (45ms)
│   ├── Span: Postgres Query (12ms)
│   └── Span: Payment Service (30ms)
│       └── Span: Stripe API Call (25ms)
└── Span: Response (1ms)

Total: 56ms | Bottleneck: Stripe API Call (25ms)

When something is slow, traces show exactly WHERE the time is spent.
Tools: Jaeger, Zipkin, Datadog APM, Honeycomb
```

## Explanation

Observability answers the question: **"Why is the system behaving this way?"**

- **Logs** tell you what happened (debugging after the fact)
- **Metrics** tell you what's happening now (dashboards, alerts)
- **Traces** tell you how a request moved through the system (finding bottlenecks)

**Key practices:**
1. **Structured logging** — JSON logs are searchable and filterable. Plain text is not
2. **Correlation IDs** — One request ID that appears in every log across all services
3. **Alert on symptoms, not causes** — Alert on "error rate > 5%" (symptom), not "CPU > 80%" (cause). High CPU is fine if users aren't affected
4. **Dashboard hierarchy** — Overview dashboard → service dashboard → individual metrics

## ELI5

Observability is like having security cameras, a dashboard, and GPS tracking for your application. Logs are the security cameras — you can review footage to see what happened. Metrics are the dashboard — you see speed, fuel level, and engine temperature at a glance. Traces are GPS — you can follow one car's entire journey from start to finish. Together, they help you understand and fix problems quickly.
