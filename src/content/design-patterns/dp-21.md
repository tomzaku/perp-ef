The Template Method pattern defines the **skeleton of an algorithm** in a base class, deferring some steps to subclasses. The overall structure (the "template") is fixed; only the variable steps are overridden.

**Template Method vs Strategy:**
- **Template Method** uses *inheritance* — the skeleton lives in the base class, subclasses override steps
- **Strategy** uses *composition* — the whole algorithm is swapped out via an injected object
- Rule of thumb: if the algorithm structure is fixed and only a few steps vary → Template Method. If the whole algorithm needs to swap → Strategy.

**Frontend use cases:**
- Data fetching lifecycle: `fetchStart → transform → cache → render` (steps vary per resource type)
- React class component lifecycle (`componentDidMount`, `render`, `componentWillUnmount` are template hooks)
- Test frameworks: `beforeEach → test → afterEach`

## Solution

```js
// Abstract base class — defines the template
class DataExporter {
  // Template method — fixed skeleton, not overridable
  export(data) {
    const validated = this.validate(data);      // step 1 — fixed
    const transformed = this.transform(validated); // step 2 — varies
    const formatted = this.format(transformed);    // step 3 — varies
    this.write(formatted);                         // step 4 — fixed
    return formatted;
  }

  validate(data) {
    if (!data) throw new Error('No data');
    return data;
  }

  write(output) {
    console.log('Writing output:', output);
  }

  // Subclasses must implement these
  transform(data) { throw new Error('transform() must be implemented'); }
  format(data)    { throw new Error('format() must be implemented'); }
}

// Concrete subclass 1 — CSV export
class CsvExporter extends DataExporter {
  transform(data) {
    return data.map(row => Object.values(row));
  }
  format(rows) {
    return rows.map(row => row.join(',')).join('\n');
  }
}

// Concrete subclass 2 — JSON export
class JsonExporter extends DataExporter {
  transform(data) {
    return data.map(row => ({ ...row, exported: true }));
  }
  format(data) {
    return JSON.stringify(data, null, 2);
  }
}

const data = [{ name: 'Alice', age: 30 }, { name: 'Bob', age: 25 }];
new CsvExporter().export(data);
// Alice,30
// Bob,25

new JsonExporter().export(data);
// [{ "name": "Alice", "age": 30, "exported": true }, ...]
```

## Explanation

The key insight is the **inversion of control** inside the inheritance hierarchy. The *base class calls the subclass*, not the other way around. The base class controls *when* each step runs; subclasses control *what* each step does.

```
DataExporter.export() — skeleton is fixed:
  1. validate()   ← base class handles this
  2. transform()  ← subclass overrides this
  3. format()     ← subclass overrides this
  4. write()      ← base class handles this

CsvExporter only needs to implement steps 2 and 3.
JsonExporter only needs to implement steps 2 and 3.
Neither can accidentally break the validate → write structure.
```

**Hook methods** are an optional variant — the base class provides a default implementation (often a no-op) that subclasses *may* override. React class lifecycle methods are hooks in this sense: `componentDidMount` has a default no-op but you can override it.

## ELI5

A recipe card has fixed steps: prep → cook → plate → serve. The *how* you cook changes per dish (boil vs fry vs bake), but the order of steps never changes. The recipe is the Template Method — it enforces the structure, and each dish fills in the cooking step differently.
