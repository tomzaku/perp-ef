**What are the key server-side caching strategies, and how do you implement them with Redis?**

Caching is often the difference between a 500ms response and a 5ms response. It's the single most effective way to improve backend performance.

## Solution

```text
CACHING LAYERS
══════════════

Client → [CDN] → [Load Balancer] → [App Cache (Redis)] → [Database]
          ↑           ↑                    ↑                   ↑
      Edge cache   Response cache    Application cache    Query cache
      (static)     (full pages)      (data objects)       (DB-level)

Each layer caches closer to the user = faster response.


CACHING STRATEGIES
══════════════════

1. CACHE-ASIDE (Lazy Loading) — Most common
   App checks cache first. On miss, reads from DB and populates cache.

   Read:
   [App] → Cache hit? → Yes → return cached data
                      → No  → query DB → store in cache → return

   Write:
   [App] → Write to DB → invalidate/delete cache entry

2. WRITE-THROUGH
   Write to cache AND DB at the same time.

   Write:
   [App] → Write to cache + Write to DB (together)

   Read:
   [App] → Always read from cache (always fresh)

   ✓ Cache is always consistent with DB
   ✗ Write latency increases (two writes)

3. WRITE-BEHIND (Write-Back)
   Write to cache, async sync to DB later.

   Write:
   [App] → Write to cache → (async) → Write to DB

   ✓ Very fast writes
   ✗ Risk of data loss if cache crashes before sync
```

```typescript
// ═══════════════════════════════════
// REDIS CACHE-ASIDE IMPLEMENTATION
// ═══════════════════════════════════

import Redis from 'ioredis';
const redis = new Redis();

// Generic cache-aside helper
async function cached<T>(
  key: string,
  ttlSeconds: number,
  fetcher: () => Promise<T>
): Promise<T> {
  // 1. Check cache
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  // 2. Cache miss — fetch from source
  const data = await fetcher();

  // 3. Store in cache with TTL
  await redis.set(key, JSON.stringify(data), 'EX', ttlSeconds);

  return data;
}

// Usage
app.get('/api/users/:id', async (req, res) => {
  const user = await cached(
    `user:${req.params.id}`,     // Cache key
    300,                          // TTL: 5 minutes
    () => db.user.findUnique({ where: { id: req.params.id } })
  );
  res.json(user);
});

// Invalidate on write
app.put('/api/users/:id', async (req, res) => {
  const user = await db.user.update({
    where: { id: req.params.id },
    data: req.validatedBody
  });
  await redis.del(`user:${req.params.id}`);  // Invalidate cache
  res.json(user);
});


// ═══════════════════════════════════
// COMMON CACHING PATTERNS
// ═══════════════════════════════════

// 1. Cache expensive queries
const dashboardData = await cached('dashboard:stats', 60, async () => {
  const [users, orders, revenue] = await Promise.all([
    db.user.count(),
    db.order.count({ where: { status: 'completed' } }),
    db.order.aggregate({ _sum: { total: true } }),
  ]);
  return { users, orders, revenue: revenue._sum.total };
});

// 2. Cache with tags (invalidate related keys)
await redis.set(`product:${id}`, data, 'EX', 3600);
await redis.sadd('cache:products', `product:${id}`);

// Invalidate all product caches:
const keys = await redis.smembers('cache:products');
if (keys.length) await redis.del(...keys);
await redis.del('cache:products');

// 3. Cache stampede prevention (lock pattern)
async function cachedWithLock<T>(key: string, ttl: number, fetcher: () => Promise<T>): Promise<T> {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  // Try to acquire lock
  const lockKey = `lock:${key}`;
  const acquired = await redis.set(lockKey, '1', 'EX', 10, 'NX');

  if (acquired) {
    // We got the lock — fetch and cache
    const data = await fetcher();
    await redis.set(key, JSON.stringify(data), 'EX', ttl);
    await redis.del(lockKey);
    return data;
  }

  // Another process is fetching — wait and retry
  await new Promise(r => setTimeout(r, 100));
  return cachedWithLock(key, ttl, fetcher);
}

// 4. Rate limiting with Redis
async function rateLimit(userId: string, limit: number, windowSec: number): Promise<boolean> {
  const key = `ratelimit:${userId}`;
  const current = await redis.incr(key);
  if (current === 1) await redis.expire(key, windowSec);
  return current <= limit;
}
```

```text
CDN CACHING
═══════════

CDN (Content Delivery Network) caches at edge locations worldwide.

User (Tokyo) → CDN Edge (Tokyo) → Origin (US)
                ↓ cached
User (Tokyo) → CDN Edge (Tokyo) → returns cached response (fast!)

What to cache on CDN:
  ✓ Static assets (JS, CSS, images, fonts) — long TTL
  ✓ API responses that rarely change — short TTL
  ✓ HTML pages (for SSR/SSG apps)

Cache-Control headers:
  Cache-Control: public, max-age=31536000, immutable
  // Static assets with content hash: cache forever

  Cache-Control: public, max-age=60, stale-while-revalidate=300
  // API data: fresh for 60s, serve stale for 5min while revalidating

  Cache-Control: private, no-store
  // Sensitive data: never cache


CACHE INVALIDATION STRATEGIES
═════════════════════════════

1. TTL (Time-To-Live) — Simplest
   Cache expires after N seconds. May serve stale data until expiry.

2. Event-based invalidation — Most accurate
   When data changes → delete/update cache.
   Requires knowing all cache keys affected by a change.

3. Version-based — For static assets
   /app.v2.js instead of /app.js
   New version = new URL = new cache entry

4. Stale-while-revalidate — Best of both
   Serve stale cache immediately, refresh in background.
   User always gets fast response, data is eventually fresh.


WHAT TO CACHE (and what NOT to)
═══════════════════════════════

✓ CACHE:
  - Database query results (hot data)
  - API responses from external services
  - Computed/aggregated data (dashboards, reports)
  - Session data
  - Static assets
  - Configuration that rarely changes

✗ DON'T CACHE:
  - Rapidly changing data (real-time prices, live scores)
  - User-specific sensitive data (without proper isolation)
  - Data that must be perfectly consistent (financial balances)
  - Write-heavy data (cache gets invalidated too often)
```

## Explanation

Caching follows the **80/20 rule**: 80% of requests hit 20% of your data. Caching that 20% gives massive performance gains.

**The cache hierarchy:**
1. **In-process cache** (fastest, per-server, small) — LRU map for config/lookups
2. **Redis** (fast, shared, medium) — Main application cache
3. **CDN** (fast, global, large) — Static assets and cacheable API responses
4. **Database query cache** (limited) — Don't rely on this

**The hardest problem:** Cache invalidation. When data changes, you must know which cache keys to invalidate. Strategies:
- Simple apps: TTL-based (accept staleness)
- Complex apps: Event-based (publish data change events → invalidate affected keys)
- Critical apps: Write-through (never stale, but slower writes)

## ELI5

Caching is like keeping a Post-it note on your desk with answers to questions people ask you all the time. Instead of looking up the answer in a big filing cabinet every time, you just glance at the Post-it. If someone asks a rare question, you go to the filing cabinet and maybe add it to a new Post-it. The tricky part is remembering to update the Post-it when the answer changes in the filing cabinet.
