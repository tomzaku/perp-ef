**What is the difference between a forward proxy, reverse proxy, and VPN? When would you use each?**

Proxies and VPNs are network intermediaries that add security, performance, and control layers. Understanding them is crucial for designing production infrastructure.

## Solution

```text
FORWARD PROXY (Client-side)
═══════════════════════════

  Client → [Forward Proxy] → Internet → Server

  The server sees the proxy's IP, not the client's

Use cases:
  ✓ Anonymity (hide client IP)
  ✓ Access control (corporate firewall/content filtering)
  ✓ Caching (shared cache for all clients)
  ✓ Bypass geo-restrictions

Examples: Squid, corporate HTTP proxies


REVERSE PROXY (Server-side)
═══════════════════════════

  Client → Internet → [Reverse Proxy] → Backend Servers

  The client sees the proxy's IP, not the backend's

Use cases:
  ✓ Load balancing (distribute traffic across servers)
  ✓ SSL termination (decrypt HTTPS at proxy, HTTP to backends)
  ✓ Caching (serve static assets without hitting backends)
  ✓ Compression (gzip/brotli responses)
  ✓ Security (hide backend topology, WAF, rate limiting)
  ✓ A/B testing, canary deployments (route % of traffic)

Examples: Nginx, HAProxy, Caddy, Traefik, AWS ALB


FORWARD vs REVERSE PROXY
═════════════════════════

┌───────────────┬──────────────────┬──────────────────┐
│               │ Forward Proxy    │ Reverse Proxy    │
├───────────────┼──────────────────┼──────────────────┤
│ Acts for      │ Client           │ Server           │
│ Hides         │ Client identity  │ Server identity  │
│ Configured by │ Client/network   │ Server/ops team  │
│ Client knows? │ Yes (explicit)   │ No (transparent) │
│ Main purpose  │ Access control   │ Load balancing   │
│               │ Anonymity        │ SSL termination  │
└───────────────┴──────────────────┴──────────────────┘


VPN (Virtual Private Network)
═════════════════════════════

  Client ===[Encrypted Tunnel]=== VPN Server → Internet

  ALL traffic goes through the encrypted tunnel

How it works:
  1. Client establishes encrypted tunnel to VPN server
  2. All network traffic is routed through tunnel
  3. VPN server decrypts and forwards to destination
  4. Response comes back through tunnel

Use cases:
  ✓ Remote access to corporate network
  ✓ Encrypt traffic on untrusted networks (public WiFi)
  ✓ Bypass censorship/geo-restrictions
  ✓ Site-to-site VPN (connect office networks)

Protocols: WireGuard, OpenVPN, IPSec, IKEv2

Proxy vs VPN:
  Proxy: application level (HTTP only usually)
  VPN: network level (ALL traffic encrypted)


REVERSE PROXY vs LOAD BALANCER
══════════════════════════════

Reverse proxy IS often the load balancer:
  Nginx, HAProxy, AWS ALB all do both

  Load balancer only: distributes traffic
  Reverse proxy:      distributes traffic + SSL + caching + compression

In practice, they overlap heavily.


NGINX REVERSE PROXY EXAMPLE
════════════════════════════

  upstream backend {
      server 10.0.0.1:3000 weight=3;
      server 10.0.0.2:3000 weight=2;
      server 10.0.0.3:3000 backup;
  }

  server {
      listen 443 ssl;
      server_name api.example.com;

      ssl_certificate     /etc/ssl/cert.pem;
      ssl_certificate_key /etc/ssl/key.pem;

      location /static/ {
          proxy_cache my_cache;
          proxy_cache_valid 200 1h;
          proxy_pass http://backend;
      }

      location /api/ {
          proxy_pass http://backend;
          proxy_set_header X-Real-IP $remote_addr;
          proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
      }
  }


HOME ROUTER & NAT
══════════════════

Your home network:
  Phone:    192.168.1.10 (private IP)
  Laptop:   192.168.1.11 (private IP)
  Router:   203.0.113.1  (public IP)

NAT (Network Address Translation):
  Phone sends packet → Router rewrites source to 203.0.113.1
  Server replies to 203.0.113.1 → Router maps back to 192.168.1.10

Inside LAN:
  - Devices find each other via MAC addresses (ARP protocol)
  - Switch: learns MACs → sends frames to correct port only
  - Hub: broadcasts everything (dumb, obsolete)
```

## Explanation

These three technologies are **intermediaries** that sit between clients and servers, each solving different problems. A **forward proxy** is like a secretary who makes calls on your behalf — the person you're calling doesn't know who you really are. A **reverse proxy** is like a receptionist at a company — callers reach the receptionist, who routes them to the right department internally.

In production systems, **reverse proxies are nearly universal**. They're the front door to your infrastructure, handling SSL/TLS termination (so backend services don't need to manage certificates), caching (reducing load on application servers), and load balancing (distributing requests across multiple instances). Nginx and HAProxy are the most common choices.

**VPNs** operate at a lower level than proxies, encrypting all network traffic between the client and the VPN server. This is why VPNs are used for security (protecting traffic on public WiFi) and remote access (connecting to a corporate network as if you're physically there).

## ELI5

A **forward proxy** is like asking your friend to buy something for you — the store sees your friend, not you. A **reverse proxy** is like a restaurant host who seats you — you talk to the host, not directly to the kitchen. A **VPN** is like a secret underground tunnel from your house to another building — nobody can see what you're carrying through it, and when you come out the other end, it looks like you started there.
