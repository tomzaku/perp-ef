**What are the key differences between TCP and UDP, and when would you choose one over the other?**

TCP and UDP are the two main transport layer protocols. Understanding their tradeoffs is fundamental to networking and system design.

TCP GUARANTEES IN DETAIL
════════════════════════

1. RELIABLE DELIVERY
   Sender: [Packet 1] → → → Receiver
   Receiver: [ACK 1] ← ← ←
   If no ACK received → retransmit after timeout

2. ORDERED DELIVERY
   Packets arrive: 3, 1, 2 → TCP reorders → delivers 1, 2, 3
   Uses sequence numbers to reconstruct original order

3. FLOW CONTROL (Sliding Window)
   Receiver tells sender: "I can handle 64KB at a time"
   Sender adjusts rate to avoid overwhelming receiver

4. CONGESTION CONTROL
   Slow Start → Exponential growth until threshold
   Congestion Avoidance → Linear growth
   Fast Recovery → On packet loss, halve window


WHEN TO USE WHAT
════════════════

TCP (need reliability):
  ✓ HTTP/HTTPS (web browsing, APIs)
  ✓ File transfer (FTP, SFTP)
  ✓ Email (SMTP, IMAP)
  ✓ Database connections
  ✓ SSH

UDP (need speed):
  ✓ DNS lookups (small, single request-response)
  ✓ Video/audio streaming (dropped frame < delay)
  ✓ Online gaming (position updates, latest state matters)
  ✓ IoT sensor data (high volume, some loss OK)

## Explanation

TCP and UDP sit at the transport layer (Layer 4) of the network stack. **TCP** establishes a connection before sending data, ensures every byte arrives correctly and in order, and adapts its sending rate to network conditions. This reliability comes at the cost of latency — the handshake, acknowledgments, and retransmissions all add overhead.

**UDP** skips all of that. It just sends datagrams with no connection setup, no acknowledgments, and no ordering. If a packet is lost, UDP doesn't care. This makes it perfect for real-time applications where a late packet is worse than a lost one — in a video call, you'd rather skip a frame than pause to wait for a retransmission.

Modern protocols like **QUIC** (used in HTTP/3) build TCP-like reliability on top of UDP, getting the best of both worlds: reliable delivery without TCP's head-of-line blocking problem.

## ELI5

TCP is like sending a letter by registered mail — you get confirmation that it arrived, and if it gets lost, they resend it. It's reliable but slow. UDP is like shouting across a room — it's fast, but some words might get lost in the noise. For important documents, use registered mail (TCP). For a live conversation where speed matters more than catching every word, just shout (UDP).
