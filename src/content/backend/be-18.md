**How do you implement proper input validation and data sanitization in backend applications?**

Input validation is your first line of defense against bugs, data corruption, and security vulnerabilities.

## Solution

```typescript
// ═══════════════════════════════════
// ZOD — TYPE-SAFE SCHEMA VALIDATION
// ═══════════════════════════════════

import { z } from 'zod';

// Define schema
const createUserSchema = z.object({
  email: z.string().email().max(255),
  name: z.string().min(1).max(100).trim(),
  password: z.string()
    .min(8, 'Password must be at least 8 characters')
    .max(128)
    .regex(/[A-Z]/, 'Must contain uppercase letter')
    .regex(/[0-9]/, 'Must contain a number'),
  age: z.number().int().min(13).max(120).optional(),
  role: z.enum(['user', 'admin']).default('user'),
});

// Infer TypeScript type from schema
type CreateUserInput = z.infer<typeof createUserSchema>;
// { email: string; name: string; password: string; age?: number; role: 'user' | 'admin' }

// Validate
const result = createUserSchema.safeParse(req.body);
if (!result.success) {
  return res.status(400).json({
    error: 'Validation failed',
    details: result.error.issues
    // [{ path: ['password'], message: 'Must contain uppercase letter' }]
  });
}
const validData = result.data; // Fully typed and validated


// ═══════════════════════════════════
// VALIDATION MIDDLEWARE (reusable)
// ═══════════════════════════════════

function validate(schema: z.ZodSchema) {
  return (req, res, next) => {
    const result = schema.safeParse(req.body);
    if (!result.success) {
      return res.status(400).json({
        error: 'Validation failed',
        details: result.error.issues.map(i => ({
          field: i.path.join('.'),
          message: i.message
        }))
      });
    }
    req.validatedBody = result.data;
    next();
  };
}

// Query parameter validation
const paginationSchema = z.object({
  page: z.coerce.number().int().min(1).default(1),
  limit: z.coerce.number().int().min(1).max(100).default(20),
  sort: z.enum(['created_at', 'updated_at', 'name']).default('created_at'),
  order: z.enum(['asc', 'desc']).default('desc'),
});

// URL parameter validation
const idParamSchema = z.object({
  id: z.string().uuid('Invalid ID format'),
});

app.get('/api/users/:id',
  validate(idParamSchema),     // Validate params
  async (req, res) => { /* ... */ }
);


// ═══════════════════════════════════
// COMMON VALIDATION PATTERNS
// ═══════════════════════════════════

// Complex object with nested validation
const createOrderSchema = z.object({
  items: z.array(z.object({
    productId: z.string().uuid(),
    quantity: z.number().int().min(1).max(99),
  })).min(1).max(50),  // 1-50 items per order

  shippingAddress: z.object({
    street: z.string().min(1).max(200),
    city: z.string().min(1).max(100),
    state: z.string().length(2),
    zip: z.string().regex(/^\d{5}(-\d{4})?$/),
    country: z.string().length(2),
  }),

  couponCode: z.string().max(50).optional(),
});

// Custom refinements
const dateRangeSchema = z.object({
  startDate: z.coerce.date(),
  endDate: z.coerce.date(),
}).refine(
  (data) => data.endDate > data.startDate,
  { message: 'End date must be after start date' }
);

// Transform during validation
const searchSchema = z.object({
  query: z.string()
    .trim()
    .min(1)
    .max(200)
    .transform(q => q.toLowerCase()),  // Normalize
});


// ═══════════════════════════════════
// OUTPUT SANITIZATION
// ═══════════════════════════════════

// Prevent XSS — sanitize data before rendering in HTML
import DOMPurify from 'isomorphic-dompurify';

// If user submitted HTML content:
const safeHtml = DOMPurify.sanitize(userInput);

// For API responses — strip sensitive fields:
const userResponse = z.object({
  id: z.string(),
  email: z.string(),
  name: z.string(),
  createdAt: z.date(),
  // password_hash is NOT in this schema → excluded
});

app.get('/api/users/:id', async (req, res) => {
  const user = await db.user.findUnique({ where: { id: req.params.id } });
  res.json(userResponse.parse(user)); // Only allowed fields returned
});


// ═══════════════════════════════════
// VALIDATION RULES BY DATA TYPE
// ═══════════════════════════════════
//
// Strings:
//   ✓ Max length (prevent memory/storage abuse)
//   ✓ Trim whitespace
//   ✓ Format validation (email, URL, UUID)
//   ✓ No null bytes (\0)
//
// Numbers:
//   ✓ Integer vs float
//   ✓ Min/max range
//   ✓ No NaN/Infinity
//
// Arrays:
//   ✓ Max length (prevent DOS)
//   ✓ Validate each element
//   ✓ No duplicates (if required)
//
// Files:
//   ✓ Max size
//   ✓ Allowed MIME types (check magic bytes, not just extension)
//   ✓ Rename files (don't use user-provided names)
//
// URLs:
//   ✓ Protocol whitelist (https only)
//   ✓ Block internal IPs (SSRF prevention)
//   ✓ Max length
```

## Explanation

**The golden rule:** Validate input, sanitize output, never trust the client.

- **Validate input** — Check type, format, length, range at the API boundary. Reject bad data early
- **Sanitize output** — Escape HTML when rendering user content. Strip sensitive fields from API responses
- **Never trust the client** — Frontend validation is for UX only. A curl command bypasses all of it

**Why Zod?** It combines runtime validation with TypeScript type inference. Define the schema once, get both validation and types. This eliminates the common bug where TypeScript types say one thing but the actual data looks different.

**Where to validate:**
- API request body, query params, URL params → always
- Database output before returning → strip sensitive fields
- Third-party API responses → they can change without warning
- Environment variables at startup → fail fast if misconfigured

## ELI5

Input validation is like airport security. You don't just trust passengers when they say "I don't have anything dangerous." You scan their bags (validate), check their ID (authenticate), and make sure they're going to the right gate (authorize). You do this for EVERY passenger, even frequent flyers. Sanitization is like wrapping fragile items before putting them in the overhead bin — you make data safe before it gets to its destination.
