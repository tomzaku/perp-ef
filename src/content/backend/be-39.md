**What types of DDoS attacks exist and how do you defend against them?**

DDoS attacks are one of the most common threats to internet services. Understanding attack vectors and defense layers is critical for designing resilient infrastructure.

## Solution

```text
DDoS ATTACK LAYERS
═══════════════════

Layer 3/4 (Network/Transport):
  Target: bandwidth, connection state tables
  Volume: massive (100+ Gbps)
  Examples: UDP flood, SYN flood, DNS amplification

Layer 7 (Application):
  Target: web server, application logic
  Volume: lower but harder to detect
  Examples: HTTP flood, API abuse, Slowloris


ATTACK TYPES
════════════

1. VOLUMETRIC ATTACKS (flood bandwidth)
───────────────────────────────────────
UDP Flood:
  Send massive UDP packets to random ports
  Server wastes resources responding "port unreachable"
  → Saturates network bandwidth

DNS Amplification:
  Attacker sends small DNS queries with spoofed source IP (victim)
  DNS servers send large responses to victim
  Amplification: 50x (28 byte query → 3000+ byte response)

  Attacker → [small query, src=victim] → DNS Server
  DNS Server → [LARGE response] → Victim

NTP Amplification:
  Similar using NTP monlist command
  Amplification: up to 556x


2. PROTOCOL ATTACKS (exhaust resources)
───────────────────────────────────────
SYN Flood:
  Send thousands of SYN packets, never complete handshake
  Server allocates memory for each half-open connection
  Connection table fills up → legitimate clients rejected

  Attacker: SYN → SYN → SYN → SYN → (never sends ACK)
  Server:   [half-open] [half-open] [half-open] ... FULL!

Ping of Death:
  Oversized ICMP packets crash vulnerable systems
  (mostly historical, modern OS immune)

Slowloris:
  Open many HTTP connections, send headers very slowly
  Keep connections open as long as possible
  Server exhausts max connection limit

  Attacker sends:
    GET / HTTP/1.1\r\n
    Host: target.com\r\n
    X-a: b\r\n            ← sends one header every 10 seconds
    X-c: d\r\n            ← never sends final \r\n\r\n
    ...                   ← connection stays open!


3. APPLICATION LAYER ATTACKS (mimic legitimate traffic)
──────────────────────────────────────────────────────
HTTP Flood:
  Send thousands of legitimate-looking HTTP requests
  Hard to distinguish from real users
  Target expensive endpoints (search, login, checkout)

API Abuse:
  Exploit expensive operations (complex queries, file uploads)
  Low request volume but high server cost per request

Cache-busting:
  Add random query params to bypass CDN cache
  /page?cb=rand123 → forces origin server to handle every request


DEFENSE STRATEGIES
══════════════════

LAYER 1: NETWORK LEVEL
───────────────────────
Anycast routing:
  Distribute traffic across global network of servers
  Attack traffic absorbed by nearest node
  No single point gets full attack volume

Blackhole routing:
  Redirect attack traffic to null route
  Last resort — also drops legitimate traffic to that IP

ISP-level filtering:
  Upstream provider filters known attack patterns
  Effective for volumetric attacks

BGP Flowspec:
  Distribute filtering rules across ISP routers
  Block specific traffic patterns at network edge


LAYER 2: PROTOCOL LEVEL
────────────────────────
SYN Cookies:
  Don't allocate memory for SYN packets
  Encode connection info in the SYN-ACK sequence number
  Only allocate on valid ACK (completed handshake)
  → SYN floods waste no server resources

Connection rate limiting:
  Limit new connections per IP per second
  Drop excess SYN packets from same source

TCP proxy:
  CDN/proxy completes TCP handshake on behalf of server
  Only forwards legitimate completed connections


LAYER 3: APPLICATION LEVEL
──────────────────────────
Rate limiting:
  - Per-IP rate limits (100 req/min)
  - Per-user rate limits (for authenticated)
  - Per-endpoint limits (expensive endpoints get lower limits)
  - Sliding window or token bucket algorithms

WAF (Web Application Firewall):
  - Block known attack patterns
  - SQL injection, XSS detection
  - Custom rules for suspicious behavior
  - Managed rulesets (AWS WAF, Cloudflare WAF)

CAPTCHA/Challenge:
  - JavaScript challenge (bots can't execute JS)
  - CAPTCHA for suspicious traffic
  - Proof-of-work challenge (Cloudflare Turnstile)

Bot detection:
  - Browser fingerprinting
  - Behavioral analysis (mouse movement, timing)
  - TLS fingerprinting (JA3 hash)
  - Reputation scoring per IP


LAYER 4: INFRASTRUCTURE
───────────────────────
CDN (Content Delivery Network):
  - Absorb attack traffic at edge (200+ PoPs)
  - Cache static content → origin never sees request
  - Cloudflare, AWS CloudFront, Fastly

Auto-scaling:
  - Scale up servers to absorb increased load
  - Works for moderate attacks
  - Not cost-effective for massive volumetric attacks

Geographic distribution:
  - Multiple regions/datacenters
  - No single point of failure
  - DNS-based routing to nearest healthy region


MONITORING & RESPONSE
═════════════════════

Detection signals:
  - Sudden traffic spike (10x normal)
  - Geographic anomalies (traffic from unusual countries)
  - Request pattern anomalies (identical requests, no cookies)
  - Error rate spike (connection timeouts)
  - Bandwidth utilization spike

Response playbook:
  1. Detect (automated monitoring)
  2. Classify (volumetric vs protocol vs application)
  3. Mitigate (activate appropriate defenses)
  4. Communicate (status page, incident channel)
  5. Post-mortem (improve defenses)


MANAGED DDOS PROTECTION
════════════════════════

  Cloudflare:   Free tier protects against most attacks
  AWS Shield:   Standard (free L3/4) + Advanced ($3K/mo L7)
  Google Cloud Armor: WAF + DDoS protection
  Akamai:       Enterprise DDoS protection

For most companies: put Cloudflare in front of your servers
  → Handles 90%+ of DDoS attacks automatically
```

## Explanation

DDoS defense is fundamentally about **asymmetry** — an attacker with modest resources can generate traffic that overwhelms a much larger target, especially with amplification attacks. A 1 Gbps botnet can generate 50+ Gbps through DNS amplification. This is why defense must be distributed — no single server can absorb a large volumetric attack.

**Application-layer attacks are the hardest to defend** because they look like legitimate traffic. An HTTP flood of valid-looking requests is indistinguishable from a flash crowd (genuine traffic spike). Defense requires behavioral analysis — real users browse pages sequentially, have varied timing, and execute JavaScript; bots hit the same endpoint repeatedly with identical patterns.

The practical approach for most companies is to **use a CDN/DDoS protection service** (Cloudflare, AWS Shield) as the first line of defense, combined with application-level rate limiting and monitoring. Building your own DDoS mitigation infrastructure only makes sense at very large scale (companies like Google, Netflix, Meta).

## ELI5

A DDoS attack is like thousands of people calling a pizza shop at the same time — the phone lines are jammed and real customers can't get through. Defense strategies: get more phone lines (scaling), hire a call screener who checks if callers are real (WAF/bot detection), limit each person to one call per minute (rate limiting), and have multiple shop locations so the crowd is spread out (CDN/anycast). No single trick stops all attacks — you need all of them working together.
