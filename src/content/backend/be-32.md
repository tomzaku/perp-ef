**What are system calls? Explain user space vs kernel space and how they interact.**

System calls are the interface between your application code and the operating system kernel. Understanding this boundary explains why I/O is expensive, how security is enforced, and how the OS manages hardware.

## Solution

```text
USER SPACE vs KERNEL SPACE
══════════════════════════

  ┌─────────────────────────────────┐
  │         USER SPACE (Ring 3)     │
  │  ┌─────────┐  ┌─────────┐      │
  │  │  Your   │  │  Other  │      │
  │  │  App    │  │  Apps   │      │
  │  └────┬────┘  └────┬────┘      │
  │       │            │            │
  │  C Library (glibc/musl)        │
  │  printf → write()              │
  ├─────────────┬───────────────────┤
  │    SYSCALL INTERFACE            │
  │    (trap / software interrupt)  │
  ├─────────────┴───────────────────┤
  │       KERNEL SPACE (Ring 0)     │
  │  ┌──────────────────────────┐   │
  │  │  Process Management      │   │
  │  │  Memory Management       │   │
  │  │  File Systems            │   │
  │  │  Network Stack           │   │
  │  │  Device Drivers          │   │
  │  └──────────────────────────┘   │
  │                                 │
  │  Hardware Access                │
  └─────────────────────────────────┘

Ring 3 (user): limited instructions, can't access hardware
Ring 0 (kernel): full access to CPU, memory, devices

Why separate?
  - Security: buggy app can't crash the OS
  - Stability: one app can't corrupt another's memory
  - Control: OS mediates all hardware access


HOW A SYSCALL WORKS
═══════════════════

  1. App calls C library function: write(fd, buf, len)

  2. Library puts syscall number in register:
     (write = 1 on Linux x86_64)
     Put arguments in registers (fd, buf, len)

  3. Execute SYSCALL instruction (x86_64) or INT 0x80 (x86)
     → CPU trap: switches from Ring 3 to Ring 0

  4. Kernel syscall handler runs:
     - Validates arguments (is fd valid? is buf accessible?)
     - Performs the operation (write to file/socket)
     - Copies result to user space

  5. Return to user space:
     - Switch back to Ring 3
     - Result in register (bytes written, or -1 on error)

  Cost: ~100-1000ns per syscall (mode switch overhead)
  Regular function call: ~1-5ns
  → Syscalls are 100x+ more expensive than regular calls


COMMON SYSTEM CALLS
════════════════════

File I/O:
  open(path, flags)      → returns file descriptor (int)
  read(fd, buf, count)   → read bytes from fd
  write(fd, buf, count)  → write bytes to fd
  close(fd)              → release file descriptor
  lseek(fd, offset)      → move read/write position

Process:
  fork()                 → create child process (copy-on-write)
  exec(path, args)       → replace current process with new program
  wait(status)           → wait for child to finish
  exit(code)             → terminate process
  getpid()               → get process ID

Memory:
  mmap(addr, len, ...)   → map file/memory into address space
  brk(addr)              → expand heap (used by malloc internally)
  munmap(addr, len)      → unmap memory

Network:
  socket(domain, type)   → create network endpoint
  bind(fd, addr)         → assign address to socket
  listen(fd, backlog)    → mark socket as passive
  accept(fd, addr)       → accept connection → new fd
  connect(fd, addr)      → initiate connection

Other:
  ioctl(fd, request)     → device-specific operations
  poll/epoll             → I/O multiplexing


EVERYTHING IS A FILE (Linux Philosophy)
═══════════════════════════════════════

In Linux, diverse resources share the file interface:

  Resource          │ File Descriptor  │ Operations
  ──────────────────┼──────────────────┼──────────────
  Regular file      │ open("/etc/foo") │ read, write
  Directory         │ opendir("/tmp")  │ readdir
  Device            │ open("/dev/sda") │ read, write, ioctl
  Pipe              │ pipe(fd)         │ read, write
  Socket            │ socket(...)      │ read, write
  Terminal          │ fd 0, 1, 2       │ read, write
  /proc entries     │ open("/proc/..") │ read

  Standard file descriptors:
    0 = stdin  (keyboard input)
    1 = stdout (terminal output)
    2 = stderr (error output)


/proc FILESYSTEM
════════════════

Virtual filesystem exposing kernel/process data:

  /proc/PID/status    → process state, memory usage
  /proc/PID/fd/       → open file descriptors
  /proc/PID/maps      → memory mappings
  /proc/cpuinfo       → CPU information
  /proc/meminfo       → memory statistics
  /proc/net/tcp       → open TCP connections

  cat /proc/self/status  → info about current process


STRACE: DEBUGGING SYSCALLS
══════════════════════════

  strace -p <pid>           → attach to running process
  strace ls                 → trace ls command
  strace -e trace=network   → only network syscalls
  strace -c ls              → summary of syscall counts/time

  Output example:
  open("/etc/passwd", O_RDONLY) = 3
  read(3, "root:x:0:0:...", 4096) = 1542
  close(3) = 0

  Essential for debugging:
  - "Why is my app slow?" → strace shows blocking syscalls
  - "Why can't it open the file?" → shows permission errors
  - "What network calls is it making?" → shows connect/send
```

## Explanation

The user/kernel boundary is a **hardware-enforced security mechanism**. The CPU's privilege rings (Ring 0 for kernel, Ring 3 for user) ensure that application code can't directly access hardware, other processes' memory, or critical OS data structures. System calls are the **only** way for applications to request kernel services.

This architecture explains several practical observations: **I/O is expensive** because every file read or network write requires a mode switch (user → kernel → user). This is why buffering exists — `printf` batches writes internally to minimize syscalls. It's also why `sendfile()` exists — it copies data between file descriptors entirely within kernel space, avoiding two unnecessary copies through user space.

The **"everything is a file" abstraction** is Linux's most powerful design decision. By giving diverse resources (devices, network sockets, pipes, proc entries) the same interface (open/read/write/close), programs can be composed generically. This is why shell redirection (`>`, `|`) works seamlessly — it just redirects file descriptors.

## ELI5

Imagine the kernel is a bank vault and your program is a customer. You can't go into the vault yourself — that would be dangerous. Instead, you fill out a form (syscall) and hand it to the teller (trap to kernel mode). The teller goes into the vault, does what you asked (read a file, send data), and brings back the result. This keeps the vault safe even if a customer tries something weird. The form process is slower than just grabbing cash yourself, which is why programs try to minimize trips to the teller.
