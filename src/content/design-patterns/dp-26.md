The Module pattern uses **closures** to create private scope and expose only a controlled public API. Variables and functions inside the closure are inaccessible from outside — true private state without classes.

**Why it still matters:**
- Pre-class JavaScript relied entirely on this pattern for encapsulation
- ES Modules (import/export) are the modern, built-in version of this pattern
- Still useful for stateful utility objects where a full class is overkill

**Variants:**
- **IIFE Module** — immediately invoked, classic form
- **Revealing Module** — define everything privately, then explicitly return the public API
- **ES Module** — `export` is the modern equivalent; unexported names are private by default

## Solution

```js
// ─── Variant 1: IIFE Module (classic) ───
const CounterModule = (() => {
  // Private — not accessible from outside
  let count = 0;
  const step = 1;

  function validate(n) {
    return typeof n === 'number';
  }

  // Public API — only what we expose
  return {
    increment()        { count += step; },
    decrement()        { count -= step; },
    incrementBy(n)     { if (validate(n)) count += n; },
    getCount()         { return count; },
    reset()            { count = 0; },
  };
})();

CounterModule.increment();
CounterModule.increment();
CounterModule.incrementBy(5);
console.log(CounterModule.getCount()); // 7
console.log(CounterModule.count);      // undefined — private!

// ─── Variant 2: Revealing Module ───
const CartModule = (() => {
  const items = [];

  function addItem(item) {
    items.push(item);
  }

  function removeItem(id) {
    const idx = items.findIndex(i => i.id === id);
    if (idx !== -1) items.splice(idx, 1);
  }

  function getTotal() {
    return items.reduce((sum, item) => sum + item.price * item.qty, 0);
  }

  function getItems() {
    return [...items]; // return a copy — prevent direct mutation
  }

  // Reveal only the public surface
  return { addItem, removeItem, getTotal, getItems };
})();

// ─── Variant 3: ES Module (modern equivalent) ───
// cart.ts
let items = []; // private — not exported

export function addItem(item) { items.push(item); }
export function getTotal() { return items.reduce((s, i) => s + i.price, 0); }
// items is never accessible from outside the module
```

## Explanation

**How closure creates privacy:**

```
const CounterModule = (() => {
  let count = 0;   ← lives in closure scope, inaccessible externally

  return {
    getCount: () => count   ← the returned function closes over count
  };
})();

CounterModule.count     // undefined — not on the returned object
CounterModule.getCount() // 7       — closure still has access
```

The returned object only contains what you explicitly put on it. The closure variables (`count`, `validate`) are trapped inside — only the returned functions can see them.

**ES Modules are the same idea, built in:**
```
// counter.ts
let count = 0;             // private — module scope only
export const get = () => count;  // public — exported
export const inc = () => count++;
```

Anything not exported is private. Consumers `import { get, inc }` and never see `count`.

## ELI5

Imagine a vending machine. You can press buttons (public API) and get snacks. But you can't reach inside to change the price list or steal snacks — that's the private state. The module is the vending machine: controlled access to internal state through a defined interface only.
