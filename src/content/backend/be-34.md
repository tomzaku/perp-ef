**What is database sharding? How do you choose a shard key and handle cross-shard queries?**

Sharding is the nuclear option for database scaling. It solves problems that replication and vertical scaling can't, but introduces significant complexity.

## Solution

```text
WHY SHARD?
══════════

When a single database server can't handle:
  - Data size: 10TB+ doesn't fit on one machine
  - Write throughput: one master is bottleneck
  - Query load: even with read replicas, master is overloaded

Replication scales READS. Sharding scales WRITES and DATA SIZE.


HORIZONTAL vs VERTICAL PARTITIONING
════════════════════════════════════

Vertical Partitioning:
  Split columns across databases
  Users DB: (id, name, email)
  Profiles DB: (user_id, bio, avatar, settings)
  → Simple, limited scalability

Horizontal Partitioning (Sharding):
  Split rows across databases
  Shard 1: users with id 1-1M
  Shard 2: users with id 1M-2M
  Shard 3: users with id 2M-3M
  → Scales to any size


SHARDING STRATEGIES
═══════════════════

1. RANGE-BASED SHARDING
   Shard by ranges of the shard key

   Shard 1: user_id 1 - 1,000,000
   Shard 2: user_id 1,000,001 - 2,000,000
   Shard 3: user_id 2,000,001 - 3,000,000

   ✓ Range queries are efficient (one shard)
   ✓ Easy to understand
   ✗ Hotspots (new users all go to last shard)
   ✗ Uneven distribution over time

2. HASH-BASED SHARDING
   shard = hash(shard_key) % num_shards

   shard = hash(user_id) % 4
   user_123 → hash(123) % 4 = 2 → Shard 2
   user_456 → hash(456) % 4 = 0 → Shard 0

   ✓ Even distribution
   ✓ No hotspots
   ✗ Range queries hit ALL shards
   ✗ Resharding requires rehashing everything

3. DIRECTORY-BASED SHARDING
   Lookup table maps keys to shards

   ┌──────────┬────────┐
   │ Key      │ Shard  │
   ├──────────┼────────┤
   │ US users │ Shard 1│
   │ EU users │ Shard 2│
   │ Asia     │ Shard 3│
   └──────────┴────────┘

   ✓ Flexible (geographic, tenant-based)
   ✓ Easy to rebalance (update directory)
   ✗ Directory is single point of failure
   ✗ Extra lookup per query


SHARD KEY SELECTION
═══════════════════

The most critical decision in sharding:

Good shard key:
  ✓ High cardinality (many unique values)
  ✓ Even distribution (no hotspots)
  ✓ Used in most queries (avoid cross-shard)
  ✓ Doesn't change (moves between shards are expensive)

  E-commerce: shard by user_id
    → User's orders, cart, profile all on same shard
    → Most queries are per-user → single shard

  Bad: shard by created_date
    → All today's writes hit one shard (hotspot!)

  Bad: shard by country
    → US shard has 60% of data (uneven)


CROSS-SHARD QUERIES
═══════════════════

  SELECT * FROM orders WHERE user_id = 123  → single shard ✓
  SELECT * FROM orders WHERE total > 100     → ALL shards ✗

Cross-shard operations are expensive:
  - Scatter-gather: query all shards, merge results
  - Cross-shard joins: nearly impossible efficiently
  - Cross-shard transactions: require 2-phase commit

Minimize cross-shard queries by:
  - Co-locating related data on same shard
  - Denormalizing to avoid joins
  - Using application-level joins
  - Maintaining materialized views


GLOBAL UNIQUE IDs
═════════════════

Auto-increment doesn't work across shards:
  Shard 1: id=1, Shard 2: id=1 → collision!

Solutions:
  UUID: 128-bit random → unique but large, bad for indexing
  Snowflake ID: timestamp + machine ID + sequence
    → Sortable, unique, compact (64-bit)
    → Used by Twitter, Discord

  ┌─────────────────────────────────────────────────┐
  │ Snowflake ID (64 bits)                          │
  │ [1 bit unused][41 bits timestamp][10 bits machine][12 bits sequence]│
  └─────────────────────────────────────────────────┘
    → 41 bits = ~69 years of milliseconds
    → 10 bits = 1024 machines
    → 12 bits = 4096 IDs per millisecond per machine


CONSISTENT HASHING (RESHARDING)
═══════════════════════════════

Problem: hash(key) % N → adding shard changes N → ALL data moves!

Consistent hashing: only K/N keys move when adding a shard

  Ring of hash space (0 to 2^32):

       Node A
      /       \
  Node D       Node B
      \       /
       Node C

  - Each key hashed to a point on the ring
  - Walk clockwise to find responsible node
  - Adding Node E: only keys between D and E move to E
  - Removing Node B: only B's keys move to C

  Used by: DynamoDB, Cassandra, memcached


WHEN TO SHARD (AND WHEN NOT TO)
═══════════════════════════════

Before sharding, try:
  1. Vertical scaling (bigger machine)
  2. Read replicas (scale reads)
  3. Caching (Redis, CDN)
  4. Query optimization (indexes, query rewrite)
  5. Data archival (move old data to cold storage)

Shard when:
  ✓ Data exceeds single server capacity
  ✓ Write throughput exceeds single master
  ✓ Requirements demand it (data residency laws)

Sharding costs:
  ✗ Operational complexity (backup, schema changes, monitoring)
  ✗ Application complexity (routing, cross-shard queries)
  ✗ Resharding difficulty (adding/removing shards)
  ✗ Limited cross-shard operations
```

## Explanation

Sharding is fundamentally about **distributing data across machines** so no single machine is a bottleneck. While replication creates copies of the same data for read scaling, sharding splits the data itself so each server handles only a fraction of the total workload.

The **shard key choice** determines everything. A good shard key ensures even data distribution and that most queries hit a single shard. A bad shard key creates hotspots (one shard gets all the load) or forces cross-shard queries (expensive scatter-gather operations). Most applications shard by user ID or tenant ID because queries are typically scoped to a single user/tenant.

**Sharding should be a last resort**, not a first approach. It adds enormous operational complexity: schema migrations must be coordinated across shards, backups are more complex, and many database features (joins, transactions, foreign keys) don't work well across shards. Companies like Instagram and Pinterest scaled to millions of users before needing to shard.

## ELI5

Imagine a library so big that one building can't hold all the books. So you build three libraries — Library A has books by authors A-H, Library B has I-Q, and Library C has R-Z. That's sharding. The tricky part is knowing which library to go to (the shard key). If someone asks "find all books published in 2024," you'd have to visit all three libraries (cross-shard query) — that's slow. But "find all books by Author X" is fast because you know exactly which library to visit.
