**What is virtual memory? How does paging work and why is it necessary?**

Virtual memory is one of the most important OS abstractions. It enables process isolation, allows programs larger than physical RAM, and makes sharing safe and efficient.

## Solution

```text
WHY VIRTUAL MEMORY?
═══════════════════

Without virtual memory:
  - Processes access physical RAM directly
  - Process A could read/write Process B's memory (no isolation!)
  - Program must fit entirely in RAM
  - Memory fragmentation wastes space

With virtual memory:
  ✓ Each process gets its own address space (isolation)
  ✓ Programs can use more memory than physical RAM (swap)
  ✓ Shared libraries loaded once, mapped into many processes
  ✓ Memory protection (read-only pages, no-execute pages)


HOW PAGING WORKS
════════════════

Virtual address → Page Table → Physical address

  Virtual Memory         Page Table         Physical Memory
  (per process)                             (shared RAM)
  ┌──────────┐         ┌─────┬──────┐      ┌──────────┐
  │ Page 0   │────────→│  0  │Frame5│─────→│ Frame 5  │
  │ Page 1   │────────→│  1  │Frame2│─────→│ Frame 2  │
  │ Page 2   │────────→│  2  │ DISK │      │ Frame 0  │
  │ Page 3   │────────→│  3  │Frame0│─────→│ Frame 1  │
  │ ...      │         │ ... │ ...  │      │ ...      │
  └──────────┘         └─────┴──────┘      └──────────┘

  Page: fixed-size block of virtual memory (typically 4KB)
  Frame: fixed-size block of physical memory (same size)
  Page Table: maps virtual pages → physical frames


PAGE FAULT
══════════

When a process accesses a page not in physical memory:

  1. CPU generates page fault exception
  2. OS page fault handler runs:
     a. Find the page on disk (swap space)
     b. If no free frame → evict a page (page replacement)
     c. Load page from disk into free frame
     d. Update page table entry
     e. Restart the instruction
  3. Process continues (unaware anything happened)

  Page fault cost: ~1-10ms (disk I/O)
  Normal memory access: ~100ns
  → 10,000-100,000x slower! Minimize page faults.


TLB (Translation Lookaside Buffer)
══════════════════════════════════

Problem: every memory access needs page table lookup
  → instruction fetch + data access = 2 lookups per instruction!

Solution: TLB caches recent virtual→physical translations

  CPU → [TLB] → hit? → Physical address (fast: ~1ns)
            ↓
          miss? → Page Table lookup → update TLB (~10-100ns)

  TLB is small (64-1024 entries) but has ~99% hit rate
  TLB flush on context switch (expensive for processes!)


PAGE REPLACEMENT ALGORITHMS
═══════════════════════════

When RAM is full and a new page is needed:

FIFO (First In, First Out):
  Evict the oldest page
  Simple but poor performance (Belady's anomaly)

LRU (Least Recently Used):
  Evict page not used for longest time
  Good performance, expensive to implement exactly
  Approximated with "clock algorithm" in practice

Clock (Second Chance):
  Pages in circular list with "referenced" bit
  Scan: if referenced=1 → set to 0, skip
        if referenced=0 → evict this page
  Good approximation of LRU, efficient to implement


MULTI-PROCESS MEMORY SHARING
═════════════════════════════

Process A              Physical RAM         Process B
┌──────────┐          ┌──────────┐         ┌──────────┐
│ Page 0→F5│─────────→│ Frame 5  │         │          │
│ Page 1→F2│─────→    │ Frame 2  │    ←────│Page 0→F2 │
│ Page 2→F8│─────→    │ Frame 8  │←────────│Page 1→F8 │
│          │          │ (libc)   │         │          │
└──────────┘          └──────────┘         └──────────┘

Shared library (libc): loaded into ONE physical frame
  → mapped into BOTH process's virtual address spaces
  → saves huge amounts of RAM


THRASHING
═════════

When too many processes compete for too little RAM:
  → Constant page faults
  → OS spends more time swapping than executing
  → System becomes unresponsive

  Detection: high page fault rate + low CPU utilization
  Solution: reduce number of processes, add RAM, or
            reduce working set size


VIRTUAL MEMORY SIZE
═══════════════════

32-bit system: 2^32 = 4 GB virtual address space per process
64-bit system: 2^48 = 256 TB (current practical limit)
  → Far more than any physical RAM
  → Most of it is unmapped (accessing triggers segfault)


SWAP SPACE
══════════

Swap = disk space used as overflow for RAM

  RAM full → evict page to swap file/partition
  Page needed → load from swap back to RAM

  Linux: /proc/swaps, swapon/swapoff
  Recommendation: SSD swap is 100x faster than HDD swap
  Some systems (containers) disable swap entirely
```

## Explanation

Virtual memory is an **indirection layer** between the addresses a program uses and the physical RAM. Every pointer in your program is a virtual address — the CPU translates it to a physical address through the page table on every memory access. The TLB makes this translation fast by caching recent lookups.

This abstraction provides three critical benefits: **isolation** (processes can't access each other's memory), **flexibility** (programs don't need to know their physical location in RAM), and **overcommitment** (you can allocate more virtual memory than physical RAM, relying on the fact that not all pages are needed simultaneously).

Understanding virtual memory explains many practical behaviors: why `fork()` is fast (Copy-on-Write shares physical pages), why memory-mapped files are efficient (OS manages caching), and why your 8GB RAM machine can run programs using 20GB total (swap). It also explains performance cliffs — when your working set exceeds physical RAM, thrashing makes the system crawl.

## ELI5

Virtual memory is like a library card catalog where every reader thinks they have the entire library to themselves. Your card catalog (page table) says "your book #5 is on shelf #12." Another reader's catalog says "their book #3 is also on shelf #12" — same physical shelf, different virtual books. If the library runs out of shelf space, they store some books in the basement (swap/disk). When you ask for a basement book, it takes longer to fetch (page fault), but you still get it.
