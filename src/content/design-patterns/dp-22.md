The Chain of Responsibility pattern passes a request along a **chain of handlers**. Each handler decides whether to:
1. **Handle** the request and stop the chain
2. **Pass** the request to the next handler
3. **Stop** the chain without handling (e.g. auth failure)

No single handler knows the full chain — each only knows its next neighbour.

**Key difference from Decorator:**
- **Decorator** always passes to the next wrapper and adds behaviour
- **Chain of Responsibility** may *stop* the chain — not every handler passes the request forward

**Frontend use cases:**
- Express / Koa middleware pipelines
- DOM event bubbling (each parent can handle or let it bubble up)
- Validation chains (each rule runs and can abort early)
- Permission checks (admin? → manager? → employee?)

## Solution

```js
// Handler base class
class Handler {
  setNext(handler) {
    this.next = handler;
    return handler; // enables chaining: a.setNext(b).setNext(c)
  }

  handle(request) {
    if (this.next) return this.next.handle(request);
    return null; // end of chain
  }
}

// Concrete handlers
class AuthHandler extends Handler {
  handle(request) {
    if (!request.token) {
      console.log('AuthHandler: no token — request blocked');
      return { error: 401 };
    }
    console.log('AuthHandler: token valid — passing on');
    return super.handle(request);
  }
}

class RateLimitHandler extends Handler {
  handle(request) {
    if (request.requestsPerMinute > 100) {
      console.log('RateLimitHandler: rate limit exceeded — blocked');
      return { error: 429 };
    }
    console.log('RateLimitHandler: within limits — passing on');
    return super.handle(request);
  }
}

class LogHandler extends Handler {
  handle(request) {
    console.log(`LogHandler: logging request to ${request.url}`);
    return super.handle(request);
  }
}

class RouteHandler extends Handler {
  handle(request) {
    console.log(`RouteHandler: handling ${request.url}`);
    return { status: 200, body: 'OK' };
  }
}

// Build the chain
const auth = new AuthHandler();
const rateLimit = new RateLimitHandler();
const log = new LogHandler();
const route = new RouteHandler();

auth.setNext(rateLimit).setNext(log).setNext(route);

// Test 1 — no token
auth.handle({ url: '/api/data', requestsPerMinute: 5 });
// AuthHandler: no token — request blocked → { error: 401 }

// Test 2 — valid token, within limits
auth.handle({ url: '/api/data', token: 'abc', requestsPerMinute: 5 });
// AuthHandler: token valid → RateLimitHandler: within limits → LogHandler: logging → RouteHandler: handling
```

## Explanation

```
Request → [AuthHandler] → [RateLimitHandler] → [LogHandler] → [RouteHandler]
               ↓                  ↓                  ↓               ↓
           block 401?         block 429?         log it         return 200
```

Each handler is independent — you can add, remove, or reorder handlers without touching the others. This is exactly how Express middleware works:

```js
app.use(authMiddleware);      // handler 1
app.use(rateLimitMiddleware); // handler 2
app.use(logMiddleware);       // handler 3
app.get('/api', routeHandler); // final handler
```

`next()` in Express is the `super.handle(request)` call — it advances the chain. Not calling `next()` stops the chain.

## ELI5

Imagine airport security. You go through passport control → security scan → customs check. Each checkpoint either lets you through or stops you. If passport control blocks you, you never reach the security scanner. The guards don't know or care about each other — each just does their job and says "next" or "stop."
