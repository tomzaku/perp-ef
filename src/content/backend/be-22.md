**Explain the TCP 3-way handshake. Why can't it be done in 2 steps? How does connection teardown work?**

The TCP handshake is the foundation of every reliable connection on the internet. Understanding it reveals why TCP is reliable and what happens when things go wrong.

## Solution

```text
3-WAY HANDSHAKE
════════════════

  Client                          Server
    │                               │
    │──── SYN (seq=100) ──────────→│  Step 1: Client sends SYN
    │                               │          with random sequence number
    │                               │
    │←── SYN-ACK (seq=300,ack=101)─│  Step 2: Server sends SYN-ACK
    │                               │          own seq + acknowledges client's
    │                               │
    │──── ACK (seq=101,ack=301) ──→│  Step 3: Client acknowledges server's seq
    │                               │
    │     CONNECTION ESTABLISHED    │
    │←─────── data flow ──────────→│


WHY NOT 2-WAY?
══════════════

2-way attempt:
  Client → SYN(seq=100) → Server
  Client ← SYN-ACK(seq=300, ack=101) ← Server

Problem: Server's sequence number (300) is never acknowledged!
  - Client knows server received its SYN (got ACK=101)
  - Server does NOT know client received its SYN (seq=300)
  - If that SYN-ACK was lost, server thinks connection is open
    but client will retry → duplicate "ghost" connections

The 3rd ACK confirms both sides are synchronized.


WHY RANDOM SEQUENCE NUMBERS?
═════════════════════════════

1. Prevents confusion with old packets from previous connections
   (stale packets won't match current sequence space)
2. Security: predictable seq numbers allow TCP hijacking attacks
   (attacker could inject packets by guessing the next seq)


CONNECTION TEARDOWN (4-WAY)
═══════════════════════════

  Client                          Server
    │                               │
    │──── FIN (seq=500) ──────────→│  Step 1: Client done sending
    │                               │
    │←── ACK (ack=501) ───────────│  Step 2: Server acknowledges
    │                               │          (may still send data!)
    │                               │
    │←── FIN (seq=700) ───────────│  Step 3: Server done sending
    │                               │
    │──── ACK (ack=701) ──────────→│  Step 4: Client acknowledges
    │                               │
    │     CONNECTION CLOSED         │

Why 4 steps (not 3)?
  Each side must independently signal "I'm done sending"
  Server may still have data to send after receiving client's FIN


TCP STATE DIAGRAM (SIMPLIFIED)
══════════════════════════════

  CLOSED → SYN_SENT → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
                                                                     ↑
  CLOSED → LISTEN → SYN_RCVD → ESTABLISHED → CLOSE_WAIT → LAST_ACK ─┘


TIME_WAIT STATE
═══════════════

After sending final ACK, client enters TIME_WAIT for 2×MSL (2 minutes)

Why?
  1. If final ACK is lost, server will retransmit FIN
     → client must be around to re-ACK
  2. Ensures all old packets from this connection expire
     → prevents them from being misinterpreted in a new connection
     on the same port

Problem in practice:
  High-traffic servers accumulate thousands of TIME_WAIT sockets
  Solution: SO_REUSEADDR socket option, or connection pooling


WHAT HAPPENS WHEN THINGS GO WRONG
══════════════════════════════════

SYN lost:        Client retransmits (exponential backoff: 1s, 2s, 4s...)
SYN-ACK lost:    Server retransmits SYN-ACK; client retransmits SYN
3rd ACK lost:    Server retransmits SYN-ACK (stays in SYN_RCVD)
                  → usually resolves when first data packet arrives

Peer crashes mid-connection:
  - If peer OS sends RST → immediate cleanup
  - If peer dies silently → TCP keepalive detects (default: 2 hours!)
  - Application-level heartbeats are faster (seconds)

Internet drops:
  - Data packets get no ACK → retransmit with exponential backoff
  - After ~13-30 retransmissions (OS dependent) → connection timeout
  - Total timeout: typically 2-15 minutes
```

## Explanation

The 3-way handshake solves a fundamental problem: **both sides need to agree on starting sequence numbers** before they can reliably track data. The SYN flag means "here's my starting sequence number," and ACK means "I received yours." Three steps is the minimum needed for both sides to send and confirm a sequence number.

The 4-way teardown exists because TCP is **full-duplex** — each direction closes independently. When the client sends FIN, it's saying "I'm done sending" but can still receive. The server acknowledges and continues sending until it's also done, then sends its own FIN. This allows graceful shutdown where no data is lost.

**TIME_WAIT** is often misunderstood. It's not a bug — it's essential for reliability. Without it, a new connection on the same port could receive stale packets from the old connection and interpret them as valid data.

## ELI5

The 3-way handshake is like a phone call. You call and say "Hey, can you hear me?" (SYN). They reply "Yes I hear you, can you hear me?" (SYN-ACK). You say "Yep, I hear you too!" (ACK). Now you can both talk. If you only did 2 steps, one person would never be sure the other can hear them. Hanging up takes 4 steps because both people need to say "I'm done talking" and hear "OK, goodbye" from the other.
