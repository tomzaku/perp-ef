**You have a slow API endpoint backed by a database query. Walk through how you'd diagnose and fix it.**

Query performance issues are the most common backend bottleneck. This question tests your ability to systematically diagnose and optimize real-world database queries.

## Solution

```text
STEP 1: IDENTIFY THE SLOW QUERY
════════════════════════════════

Enable slow query logging:

  -- PostgreSQL: log queries taking > 200ms
  ALTER SYSTEM SET log_min_duration_statement = 200;

  -- MySQL: enable slow query log
  SET GLOBAL slow_query_log = 'ON';
  SET GLOBAL long_query_time = 0.2;

APM tools (Datadog, New Relic) also surface slow queries
with call counts and p95 latencies.


STEP 2: DIAGNOSE WITH EXPLAIN ANALYZE
══════════════════════════════════════

  EXPLAIN ANALYZE
  SELECT u.name, COUNT(o.id) AS order_count
  FROM users u
  LEFT JOIN orders o ON u.id = o.user_id
  WHERE u.created_at > '2024-01-01'
  GROUP BY u.id;

  Read the output bottom-up:

  ┌─ Hash Aggregate (cost=1500..1600 rows=5000)
  │   Group Key: u.id
  │
  ├─ Hash Left Join (cost=800..1400 rows=50000)
  │   Hash Cond: (u.id = o.user_id)
  │   │
  │   ├─ Seq Scan on users u (cost=0..500 rows=5000)  ← PROBLEM!
  │   │   Filter: (created_at > '2024-01-01')
  │   │   Rows Removed by Filter: 45000
  │   │
  │   └─ Hash (cost=300..300 rows=100000)
  │       → Seq Scan on orders o
  └─

  Red flags:
  ✗ Seq Scan on large table → needs index
  ✗ Rows Removed by Filter is high → index would help
  ✗ Actual rows >> Estimated rows → stale statistics
  ✗ Hash Join on huge table → check join column indexes


STEP 3: APPLY TARGETED FIXES
═════════════════════════════


FIX 1: Add missing indexes
───────────────────────────

  -- Index on filter column
  CREATE INDEX idx_users_created_at ON users(created_at);

  -- Index on join column (foreign key)
  CREATE INDEX idx_orders_user_id ON orders(user_id);

  -- Composite index for common query patterns
  CREATE INDEX idx_orders_user_status
    ON orders(user_id, status, created_at);


FIX 2: Rewrite anti-pattern queries
────────────────────────────────────

  -- ANTI-PATTERN: Function on indexed column
  -- Prevents index usage (scans all rows)
  SELECT * FROM orders
  WHERE YEAR(created_at) = 2024;

  -- FIX: Use range condition
  SELECT * FROM orders
  WHERE created_at >= '2024-01-01'
    AND created_at < '2025-01-01';

  -- ANTI-PATTERN: Implicit type conversion
  -- Column is INTEGER but comparing with STRING
  SELECT * FROM users WHERE phone = '5551234';

  -- FIX: Match the column type
  SELECT * FROM users WHERE phone = 5551234;

  -- ANTI-PATTERN: OR across different columns
  SELECT * FROM products
  WHERE name LIKE '%phone%' OR category = 'electronics';

  -- FIX: UNION (each branch can use its own index)
  SELECT * FROM products WHERE name LIKE '%phone%'
  UNION
  SELECT * FROM products WHERE category = 'electronics';


FIX 3: Eliminate N+1 queries
────────────────────────────

  -- N+1 PROBLEM (1 + N queries):
  -- Query 1: SELECT * FROM posts LIMIT 20
  -- For each post:
  --   Query N: SELECT * FROM users WHERE id = post.author_id

  -- FIX A: Eager load with JOIN
  SELECT p.*, u.name AS author_name
  FROM posts p
  JOIN users u ON p.author_id = u.id
  LIMIT 20;

  -- FIX B: Batch load with IN
  SELECT * FROM users
  WHERE id IN (1, 2, 3, 5, 8);  -- collected author_ids

  -- FIX C: ORM eager loading
  -- Prisma
  const posts = await prisma.post.findMany({
    include: { author: true },
    take: 20
  });

  -- Rails
  Post.includes(:author).limit(20)


FIX 4: Optimize pagination
──────────────────────────

  -- SLOW: OFFSET scans and discards rows
  SELECT * FROM products
  ORDER BY created_at DESC
  LIMIT 20 OFFSET 100000;
  -- ↑ Scans 100,020 rows, returns 20

  -- FAST: Keyset pagination (cursor-based)
  SELECT * FROM products
  WHERE created_at < '2024-06-15T10:30:00Z'
  ORDER BY created_at DESC
  LIMIT 20;
  -- ↑ Index seek, scans only 20 rows

  -- Client sends: ?cursor=2024-06-15T10:30:00Z
  -- Server returns: { data: [...], nextCursor: "2024-06-14..." }


FIX 5: Reduce data fetched
───────────────────────────

  -- SLOW: SELECT * returns all columns
  SELECT * FROM users;  -- includes bio, avatar_blob, metadata

  -- FAST: Select only needed columns
  SELECT id, name, email FROM users;

  -- FASTER: Covering index (index-only scan, no table access)
  CREATE INDEX idx_users_email_name
    ON users(email) INCLUDE (name);
  SELECT name FROM users WHERE email = 'x@y.com';


FIX 6: Denormalize for read-heavy queries
──────────────────────────────────────────

  -- SLOW: COUNT on every page load
  SELECT COUNT(*) FROM comments WHERE post_id = 42;

  -- FAST: Maintain a counter cache
  ALTER TABLE posts ADD COLUMN comment_count INT DEFAULT 0;

  -- Update counter in a trigger or application code
  UPDATE posts
  SET comment_count = comment_count + 1
  WHERE id = 42;

  -- SLOW: Complex aggregation on every request
  SELECT AVG(rating) FROM reviews WHERE product_id = 99;

  -- FAST: Materialized view (refreshed periodically)
  CREATE MATERIALIZED VIEW product_stats AS
  SELECT product_id, AVG(rating) AS avg_rating, COUNT(*) AS review_count
  FROM reviews
  GROUP BY product_id;

  REFRESH MATERIALIZED VIEW CONCURRENTLY product_stats;


FIX 7: Partition large tables
─────────────────────────────

  -- Table with 500M rows, mostly queried by date range
  CREATE TABLE events (
    id BIGINT,
    event_type TEXT,
    created_at TIMESTAMPTZ,
    payload JSONB
  ) PARTITION BY RANGE (created_at);

  CREATE TABLE events_2024_q1 PARTITION OF events
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');
  CREATE TABLE events_2024_q2 PARTITION OF events
    FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');

  -- Query only scans relevant partition
  SELECT * FROM events
  WHERE created_at >= '2024-03-01'
    AND created_at < '2024-04-01';
```

## Explanation

**The optimization decision tree:**

| Symptom | Diagnosis | Fix |
|---|---|---|
| Seq Scan on large table | Missing index | Add index on WHERE/JOIN columns |
| High "Rows Removed by Filter" | Unselective filter or wrong index | Add composite index or rethink query |
| Nested Loop with inner Seq Scan | Missing index on join column | Add index on foreign key |
| Many round trips | N+1 query problem | Use JOIN, IN, or ORM eager loading |
| Slow at high page numbers | OFFSET pagination | Switch to keyset/cursor pagination |
| Repeated expensive aggregation | Recomputing on every request | Counter cache, materialized view, or app-level cache |
| Table too large for single-column index | Table has hundreds of millions of rows | Partition by date range or tenant |

**Performance impact rankings (most to least impactful):**
1. Add missing index on WHERE/JOIN column
2. Eliminate N+1 queries
3. Switch from OFFSET to keyset pagination
4. Remove SELECT * (use specific columns)
5. Denormalize with counter caches
6. Partition oversized tables

## ELI5

Imagine you're looking for a book in a huge library. A slow query is like walking through every shelf. The fix depends on the problem: if there's no catalog (add an index), if you're asking for every book just to count them (keep a running tally), if you're skipping 1000 shelves to get to shelf 1001 (remember where you left off instead). The first step is always figuring out *why* it's slow — then the fix is usually obvious.
