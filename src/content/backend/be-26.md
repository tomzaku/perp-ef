**What happens when you type google.com into your browser and press Enter?**

This is the classic interview question that tests your breadth across networking, DNS, security, HTTP, and browser rendering. A strong answer covers each layer with appropriate depth.

## Solution

```text
COMPLETE FLOW: Typing "https://google.com" + Enter
═══════════════════════════════════════════════════

1. URL PARSING
───────────────
Browser parses: https://google.com
  → Scheme: https (port 443)
  → Host: google.com
  → Path: / (default)

Checks HSTS list: google.com requires HTTPS?
  → Yes → ensures HTTPS even if user typed HTTP


2. DNS RESOLUTION
──────────────────
Need IP address for "google.com"

Cache hierarchy (checked in order):
  1. Browser DNS cache      → miss
  2. OS DNS cache           → miss
  3. Router DNS cache       → miss
  4. ISP recursive resolver → miss
  5. Full DNS lookup:
     Root (.) → .com TLD → google.com authoritative NS
     → Returns: 142.250.80.46, TTL=300

Result cached at every level for future lookups.


3. TCP CONNECTION (3-Way Handshake)
────────────────────────────────────
Client                     Server (142.250.80.46:443)
  |--- SYN -------------->|
  |<-- SYN-ACK -----------|
  |--- ACK -------------->|
  Connection established (~1 RTT)


4. TLS HANDSHAKE (HTTPS)
──────────────────────────
Client                     Server
  |--- ClientHello ------>|  (supported ciphers, TLS version)
  |<-- ServerHello -------|  (chosen cipher, certificate)
  |                        |
  |  Verify certificate:   |
  |  - Check CA signature  |
  |  - Check expiry        |
  |  - Check domain match  |
  |                        |
  |--- Key Exchange ----->|  (pre-master secret)
  |<-- Finished ----------|
  |--- Finished --------->|

  Both sides now have symmetric session key
  All further data is encrypted (~1-2 RTT)

  TLS 1.3: reduced to 1 RTT (or 0-RTT with resumption)


5. HTTP REQUEST
────────────────
GET / HTTP/2
Host: google.com
User-Agent: Chrome/120
Accept: text/html
Accept-Encoding: gzip, br
Accept-Language: en-US
Cookie: [session cookies]


6. SERVER PROCESSING
─────────────────────
Load Balancer → Web Server → Application Logic
  → Check auth/session
  → Generate HTML (or serve from cache)
  → Compress with brotli/gzip
  → Return response


7. HTTP RESPONSE
─────────────────
HTTP/2 200 OK
Content-Type: text/html; charset=UTF-8
Content-Encoding: br
Cache-Control: private, max-age=0
Set-Cookie: [updated session]

<!DOCTYPE html><html>...</html>


8. HTML PARSING & ADDITIONAL REQUESTS
───────────────────────────────────────
Browser parses HTML top to bottom:

  <link rel="stylesheet" href="/style.css">  → Request CSS
  <script src="/app.js">                     → Request JS
  <img src="/logo.png">                      → Request image

HTTP/2 multiplexing: all requests over SINGLE TCP connection
  → No new handshakes per asset
  → Server can push assets before browser requests them


9. RENDERING
─────────────
DOM (HTML)  -->+
               +--> Render Tree → Layout → Paint → Composite
CSSOM (CSS) -->+

JavaScript:
  → Can modify DOM/CSSOM
  → Blocks rendering if synchronous
  → async/defer attributes prevent blocking


CACHING AT EVERY LAYER
═══════════════════════

Layer              | What is Cached           | TTL
-------------------+--------------------------+-----------
Browser DNS        | DNS records              | ~60s
Browser HTTP       | Responses (Cache-Ctrl)   | Varies
CDN (Cloudflare)   | Static assets            | Hours-days
Reverse Proxy      | Full responses           | Seconds-min
Application        | DB queries, computed     | App logic
Database           | Query plans, buffers     | Session


OPTIMIZATION SUMMARY
════════════════════

DNS:       Pre-resolve: <link rel="dns-prefetch" href="//cdn.com">
TCP:       Connection reuse (keep-alive), connection pooling
TLS:       Session resumption (0-RTT), TLS 1.3
HTTP:      HTTP/2 multiplexing, compression, proper caching
Assets:    CDN, lazy loading, code splitting
Rendering: Critical CSS inline, defer non-critical JS
```

## Explanation

This question is beloved in interviews because it spans **every layer of the web stack**. A junior developer might mention "DNS lookup, then HTTP request." A senior developer will discuss cache hierarchies, TLS certificate verification, HTTP/2 multiplexing, and rendering optimization.

The key insight is that **every step has been optimized through caching and connection reuse**. DNS results are cached at 4+ levels. TCP connections are kept alive and reused. TLS sessions can be resumed without a full handshake. HTTP/2 multiplexes many requests over one connection. Browsers cache responses based on Cache-Control headers. CDNs cache assets at edge locations worldwide.

When answering in an interview, **go broad first, then dive deep** into the areas you know best. Start with the high-level flow (DNS → TCP → TLS → HTTP → Render), then offer to go deeper on specific topics.

## ELI5

Imagine you want to visit a friend's house. First, you look up their address in your phone (DNS). Then you drive there and knock on the door (TCP handshake). They check your ID through the peephole (TLS). Once they open the door, you ask "what's for dinner?" (HTTP request). They go to the kitchen and bring you food (server response). Then you eat it — the last step is the browser showing you the page. If you visit often, you remember the address (DNS cache) and they leave the door unlocked (keep-alive connection).
