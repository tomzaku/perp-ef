**What is the difference between a process and a thread? When would you use multi-process vs multi-thread?**

Understanding processes and threads is fundamental to building concurrent systems, from web servers to background job workers.

## Solution

```text

═════════════════

┌──────────────────┬────────────────────┬────────────────────┐
│ Aspect           │ Process            │ Thread             │
├──────────────────┼────────────────────┼────────────────────┤
│ Memory           │ Own address space  │ Shared with process│

  Process

  │                                     │

  │  └─────────┘ └─────────┘ └───────┘ │
  └─────────────────────────────────────┘

Each thread has its own: stack, registers, program counter
Threads share: heap, global data, file descriptors, code


PROCESS STATES
══════════════

  New → Ready ←→ Running → Terminated
                    ↓  ↑
                  Waiting
                (I/O, sleep,
                 lock wait)

  Ready:    In queue, waiting for CPU
  Running:  Currently executing on CPU
  Waiting:  Blocked on I/O, sleep, or resource
  A sleeping thread is in Waiting state
  OS moves it to Ready when sleep timer expires


CONTEXT SWITCHING
═════════════════

When OS switches from Process A to Process B:
  1. Save A's registers, program counter, stack pointer
  2. Save A's memory mappings (page table pointer)
  3. Load B's page table → TLB flush (expensive!)
  4. Load B's registers, program counter, stack pointer
  5. Resume B

When switching between threads (same process):
  1. Save thread registers, PC, stack pointer
  2. Load other thread's registers, PC, stack pointer
  → No TLB flush needed (same address space) → faster!


THREAD POOL
═══════════

Problem: creating/destroying threads per task is expensive
Solution: pre-create a pool of threads, reuse them

  Task Queue: [Task1, Task2, Task3, Task4, ...]
                 ↓       ↓       ↓
              Thread1  Thread2  Thread3  (pool of N workers)

  - Limits resource usage (bounded number of threads)
  - Amortizes creation cost
  - Used by: web servers, database connection handlers,
    Java ExecutorService, Python ThreadPoolExecutor


WHEN TO USE WHAT
════════════════

Multi-PROCESS:
  ✓ Need crash isolation (one worker crashes, others survive)
  ✓ CPU-bound in languages with GIL (Python multiprocessing)
  ✓ Security isolation (sandboxing: Chrome tabs)
  ✓ Simple programming model (no shared state bugs)
  Examples: Nginx workers, Chrome tabs, PostgreSQL backends

Multi-THREAD:
  ✓ Need shared memory (fast data sharing)
  ✓ I/O-bound work (threads waiting on network/disk)
  ✓ Lower overhead per concurrent unit
  ✓ Need fine-grained synchronization
  Examples: Java web servers, game engines, database engines

Event Loop (single-thread):
  ✓ High-concurrency I/O (thousands of connections)
  ✓ Simple concurrency model (no locks)
  Examples: Node.js, Redis, Nginx event handling
```

## Explanation

The core difference is **isolation vs efficiency**. Processes are isolated — each has its own memory space, so one process can't corrupt another's data, and a crash in one process doesn't affect others. But this isolation comes at a cost: creating processes is expensive, context switching requires flushing the TLB, and sharing data requires explicit IPC mechanisms.

Threads share memory within a process, making them cheaper to create and faster to switch between. But shared memory is a double-edged sword — it enables efficient communication but also introduces race conditions, requiring careful synchronization with locks, semaphores, or atomic operations.

In modern systems, the choice often depends on your language and workload. **Python** uses multi-process for CPU-bound work (due to the GIL). **Java/Go** use threads/goroutines for concurrent I/O. **Node.js** uses a single-threaded event loop for I/O with worker threads for CPU-bound tasks. Production systems often combine all three approaches.

## ELI5

A process is like a separate house — it has its own kitchen, bathroom, and living room. Nobody else can mess with your stuff, but if you want to share something with your neighbor, you have to walk over and hand it to them. A thread is like a roommate in the same house — you share the kitchen and living room (fast and easy), but you have to take turns using the bathroom (synchronization), and if one roommate sets the house on fire, everyone's affected.
