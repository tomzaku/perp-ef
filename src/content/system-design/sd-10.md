Rate limiting protects APIs from abuse and ensures fair usage. Common algorithms: **Token Bucket**, **Sliding Window**, **Fixed Window**.

**Client-side concerns:**
- Handle 429 (Too Many Requests) responses gracefully
- Implement exponential backoff for retries
- Read Retry-After header
- Queue requests to stay under limits

Implement a rate limiter and a client that respects rate limits.

## Solution

```js
// ════════════════════════════════════════════
// TOKEN BUCKET RATE LIMITER
// ════════════════════════════════════════════

class TokenBucket {
  constructor(capacity, refillRate) {
    this.capacity = capacity;       // Max tokens
    this.tokens = capacity;         // Current tokens
    this.refillRate = refillRate;   // Tokens per second
    this.lastRefill = Date.now();
  }

  #refill() {
    const now = Date.now();
    const elapsed = (now - this.lastRefill) / 1000;
    this.tokens = Math.min(this.capacity, this.tokens + elapsed * this.refillRate);
    this.lastRefill = now;
  }

  tryConsume(tokens = 1) {
    this.#refill();
    if (this.tokens >= tokens) {
      this.tokens -= tokens;
      return { allowed: true, remaining: Math.floor(this.tokens) };
    }
    const waitTime = (tokens - this.tokens) / this.refillRate;
    return { allowed: false, retryAfter: Math.ceil(waitTime) };
  }
}

// 100 requests per minute
const limiter = new TokenBucket(100, 100 / 60);

function handleRequest(req) {
  const result = limiter.tryConsume();
  if (!result.allowed) {
    return {
      status: 429,
      headers: { 'Retry-After': result.retryAfter },
      body: 'Too Many Requests',
    };
  }
  return {
    status: 200,
    headers: { 'X-RateLimit-Remaining': result.remaining },
    body: 'OK',
  };
}

// ════════════════════════════════════════════
// SLIDING WINDOW RATE LIMITER
// ════════════════════════════════════════════

class SlidingWindowLimiter {
  constructor(windowMs, maxRequests) {
    this.windowMs = windowMs;
    this.maxRequests = maxRequests;
    this.requests = []; // Timestamps
  }

  tryConsume() {
    const now = Date.now();
    // Remove expired timestamps
    this.requests = this.requests.filter(t => now - t < this.windowMs);

    if (this.requests.length < this.maxRequests) {
      this.requests.push(now);
      return { allowed: true, remaining: this.maxRequests - this.requests.length };
    }

    const oldestInWindow = this.requests[0];
    const retryAfter = Math.ceil((oldestInWindow + this.windowMs - now) / 1000);
    return { allowed: false, retryAfter };
  }
}

// ════════════════════════════════════════════
// CLIENT: Rate-limit-aware HTTP client
// ════════════════════════════════════════════

class RateLimitAwareClient {
  #queue = [];
  #processing = false;
  #retryAfter = 0;

  async fetch(url, options = {}) {
    return new Promise((resolve, reject) => {
      this.#queue.push({ url, options, resolve, reject });
      this.#processQueue();
    });
  }

  async #processQueue() {
    if (this.#processing || this.#queue.length === 0) return;
    this.#processing = true;

    while (this.#queue.length > 0) {
      // Wait if rate limited
      if (this.#retryAfter > 0) {
        const waitMs = this.#retryAfter * 1000;
        console.log('Rate limited. Waiting ' + this.#retryAfter + 's...');
        await new Promise(r => setTimeout(r, waitMs));
        this.#retryAfter = 0;
      }

      const { url, options, resolve, reject } = this.#queue.shift();

      try {
        const res = await fetch(url, options);

        if (res.status === 429) {
          // Put request back in queue and wait
          this.#queue.unshift({ url, options, resolve, reject });
          this.#retryAfter = parseInt(res.headers.get('Retry-After') || '5');
          continue;
        }

        resolve(res);
      } catch (err) {
        reject(err);
      }
    }

    this.#processing = false;
  }
}

const client = new RateLimitAwareClient();
// All requests go through the client — auto-handles 429
client.fetch('/api/data').then(res => console.log('Got data'));
```

## ELI5

Imagine a popular coffee shop. If they let everyone order as fast as they want, one customer could place 1000 orders and block everyone else. So they put up a sign: **"Max 3 orders per minute per person."**

That's rate limiting.

**Token bucket** is the most natural way to think about it:

```
Token bucket (capacity=10, refills 2 tokens/second):

  Start:  [🪙🪙🪙🪙🪙🪙🪙🪙🪙🪙]  10 tokens

  Request 1: consume 1 → [🪙🪙🪙🪙🪙🪙🪙🪙🪙]  9 tokens  ✓
  Request 2: consume 1 → [🪙🪙🪙🪙🪙🪙🪙🪙]    8 tokens  ✓
  ...burst of 8 more...
  Bucket empty:          []                              0 tokens
  Request 11: no tokens → 429 Too Many Requests!

  Wait 0.5s:  bucket refills to 1 token
  Request 12: consume 1 → ✓
```

**Why token bucket?** It allows short bursts (you can use all 10 tokens at once), but over time keeps the rate steady (2 per second). Real usage is bursty, not perfectly spread out.

**Client-side handling** is just as important:

```
Server says 429 (Too Many Requests):
  ❌ Don't: keep hammering the server immediately
  ✅ Do:    read Retry-After header, wait that long, then retry

  Retry-After: 30
  → Wait 30 seconds before trying again

Good client behavior:
  Request #1 → 200 OK
  Request #2 → 429 (rate limited) → wait 30s
  Request #2 (retry) → 200 OK ✓
```

Think of the client as a polite person who says "Sorry to bother you — I'll come back later" instead of knocking on the door 100 times.
