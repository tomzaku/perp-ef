**What is Protocol Buffers (Protobuf), how does binary encoding work, and how do you evolve schemas safely?**

Protobuf is Google's binary serialization format, widely used for internal service communication and gRPC. Understanding it is critical for high-performance backend systems.

## Solution

```protobuf
// ═══════════════════════════════════
// 1. DEFINE A .proto SCHEMA
// ═══════════════════════════════════

syntax = "proto3";

package ecommerce;

// Field numbers (1, 2, 3...) are the real identifiers — not field names.
// Numbers 1–15 use 1 byte; 16–2047 use 2 bytes.
// Reserve small numbers for frequently-used fields.

message User {
  uint64 id        = 1;  // field number 1
  string name      = 2;  // field number 2
  string email     = 3;  // field number 3
  repeated string roles = 4;  // repeated = array
  UserStatus status = 5;
  Address address  = 6;  // nested message
}

message Address {
  string street  = 1;
  string city    = 2;
  string country = 3;
}

enum UserStatus {
  USER_STATUS_UNSPECIFIED = 0;  // proto3 requires default 0 value
  USER_STATUS_ACTIVE      = 1;
  USER_STATUS_SUSPENDED   = 2;
}

// ═══════════════════════════════════
// 2. WIRE FORMAT — HOW BINARY ENCODING WORKS
// ═══════════════════════════════════

// Each field encoded as: [field_number << 3 | wire_type] [value]
//
// Wire types:
//   0 = Varint (int32, int64, bool, enum)
//   1 = 64-bit (fixed64, double)
//   2 = Length-delimited (string, bytes, embedded messages, repeated)
//   5 = 32-bit (fixed32, float)
//
// Example: User { id: 1, name: "Alice" }
//
// Field 1 (id=1, varint): tag=0x08 (field 1, type 0), value=0x01
// Field 2 (name="Alice", string): tag=0x12 (field 2, type 2), len=0x05, "Alice"
//
// Total: 9 bytes  vs  {"id":1,"name":"Alice"} = 22 bytes
//
// Varint encoding (saves space for small numbers):
//   Values < 128: 1 byte
//   Values < 16384: 2 bytes
//   etc.

// ═══════════════════════════════════
// 3. SCHEMA EVOLUTION RULES
// ═══════════════════════════════════

// SAFE changes (backward + forward compatible):
//   ✓ Add a new field with a NEW field number
//   ✓ Remove a field (mark as reserved)
//   ✓ Rename a field (numbers are the real IDs)

// BREAKING changes (avoid):
//   ✗ Reuse a field number for a different type
//   ✗ Change a field's type (e.g., int32 → string)
//   ✗ Change a field number

// RESERVED keyword prevents accidental reuse:
message UserV2 {
  uint64 id    = 1;
  string name  = 2;
  string email = 3;
  // "nickname" was field 4 — now removed but reserved
  reserved 4;
  reserved "nickname";
  UserStatus status = 5;
  Address address   = 6;
  string phone      = 7;  // NEW field — safe to add
}

// ═══════════════════════════════════
// 4. GENERATED CODE USAGE (Go example)
// ═══════════════════════════════════

// protoc --go_out=. user.proto
// import "ecommerce/user.pb.go"

// Serialize:
// user := &ecommerce.User{Id: 1, Name: "Alice", Email: "alice@example.com"}
// data, err := proto.Marshal(user)

// Deserialize:
// user2 := &ecommerce.User{}
// err = proto.Unmarshal(data, user2)
```

## Explanation

**Why Protobuf over JSON?**

| Feature | JSON | Protobuf |
|---|---|---|
| Format | Human-readable text | Binary |
| Size | Baseline | 3–10x smaller |
| Parse speed | Slower | 5–10x faster |
| Schema | Optional (JSON Schema) | Required (.proto) |
| Type safety | Loose | Strict |
| Versioning | Manual | Built-in (field numbers) |
| Readability | Yes | No (need schema to decode) |

**The key insight**: Protobuf encodes field *numbers*, not field *names*. `name` in JSON is the string `"name"` (4 bytes per field). In Protobuf, field 2 is just `0x12` (1 byte). This is why renaming a field is safe — the wire format doesn't change. This is also why reusing a field number is catastrophic — old readers will interpret the bytes as the wrong type.

**Field number strategy**: Use numbers 1–15 for your most common fields (they encode in 1 byte). Numbers 16+ take 2 bytes. Plan this before your first release — changing field numbers is a breaking change.

**Proto3 vs Proto2**: Proto3 (current default) removes required fields (everything is optional), simplifies the API, and uses `0`/`""` as defaults. Proto2 still exists in legacy codebases and supports required fields and custom defaults.

## ELI5

Imagine you and a friend agree on a secret code before a trip: "1 means name, 2 means age, 3 means city." Now instead of writing "name=Alice, age=30, city=Paris" you just write "1:Alice, 2:30, 3:Paris" — much shorter. If you later want to add "4 means phone", your friend with the old codebook just ignores field 4. That's Protobuf: a shared schema that lets both sides communicate efficiently, and adding new fields doesn't break old readers.
